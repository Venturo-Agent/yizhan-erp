# 健檢擴展：ERP 移動端 App 架構討論框架

> 建立日期：2026-05-20
> 觸發背景：William 規劃 ERP 移動端 App，需要系統化討論整體架構

---

## Why

現有 VENTURO ERP 是網頁版（B/S 架構），但實際業務場景需要：

1. **移動端即時性**：業務在外需要快速查訂單、司機需要即時收到派遣
2. **LINE 風格體驗**：團隊已習慣 LINE 溝通模式，移動端應延續這個 UX
3. **供應鏈整合**：租戶之間的報價協作需要雙向同步的 SSOT
4. **看板觸控**：建立團選項多、看板模式適合觸控操作

## 架構方向（待討論）

### 方向 A：多角色分流

每個角色有自己的 App 入口，介面完全客製化：

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   業務 App      │     │   司機 App      │     │   供應商 App    │
│  - 我的訂單     │     │  - 派遣訂單     │     │  - 報價回覆     │
│  - 顧客管理     │     │  - 完成回報     │     │  - 行程確認     │
│  - 建立團       │     │  - 緊急聯繫     │     │  - 對帳         │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**優點**：每個角色介面單純、學習成本低
**缺點**：需要維護多個 App、跨角色協作複雜

### 方向 B：單一 App + 角色切換

一個 App、根據登入角色顯示不同功能：

```
┌─────────────────────────┐
│       Venturo App       │
│  ┌───────────────────┐ │
│  │  角色：業務 ▼      │ │
│  └───────────────────┘ │
│  ┌───────────────────┐ │
│  │  Tab 1 │ Tab 2 │..│ │
│  │  動態內容（根據角色│ │
│  │  顯示不同功能）    │ │
│  └───────────────────┘ │
└─────────────────────────┘
```

**優點**：一個 App 好維護、跨角色協作自然
**缺點**：介面複雜度提升、需要好的資訊架構

### 方向 C：LINE 風格 + 功能模組

以 LINE 為 UX 模板，整合所有功能：

```
┌─────────────────────────┐
│  Venturo 應用           │
│  ┌───────────────────┐  │
│  │ 📱 首頁（LINE style）│ │
│  │ ┌─────────┬─────┐ │ │
│  │ │ 我的訂單│ 訊息 │ │ │
│  │ ├─────────┼─────┤ │ │
│  │ │ 顧客    │ 頻道 │ │ │
│  │ ├─────────┼─────┤ │ │
│  │ │ AI助手  │ 設定 │ │ │
│  │ └─────────┴─────┘ │ │
│  └───────────────────┘ │
│  ┌───────────────────┐ │
│  │  底部 Tab：主功能  │ │
│  └───────────────────┘ │
└─────────────────────────┘
```

**優點**：最符合現有 ERP 的 LINE 風格、學習曲線低
**缺點**：功能越多、首頁越複雜

## 討論問題（需要 William 確認）

### Q1：App 架構

你傾向哪個方向？（A 多角色 / B 單一 + 切換 / C LINE 風格）

### Q2：LINE@ 整合程度

現有 LINE@ 功能（訊息、標籤、回覆）是否要整合進 App？

- 完全整合：App 就是 LINE@ 的另一個入口
- 部分整合：訊息走 LINE@、其他功能在 App 內
- 完全分離：LINE@ 繼續當外部入口、App 獨立

### Q3：供應鏈角色

哪些角色需要納入供應鏈 SSOT？

- [ ] 車行（遊覽車）
- [ ] 小車（派遣）
- [ ] 地接社（當地旅行社）
- [ ] 飯店（未來）
- [ ] 餐廳（未來）
- [ ] 郵輪（未來）

### Q4：看板模式範圍

「建立團」看板要涵蓋哪些選項？

- 行程規劃（路線、時間）
- 團員管理（報到、名單）
- 車輛派遣（車行、司機）
- 餐食景點（供應商指派）

### Q5：SSOT 同步機制

廠商報價如何在雙方系統同步？

- 即時同步（每次改動都同步）
- 離線優先（網路不穩時本地先存、再同步）
- 定時批次（固定時間同步）

## Impact

- **受影響的系統**：
  - 現有 Web ERP（需要提供 API 給 App）
  - LINE@ 串接（需要確認整合程度）
  - 供應鏈資料庫（需要新增租戶類型）
  - 報價行程表（核心 SSOT）

- **關鍵技術決策**：
  - React Native vs Flutter vs Web App（PWATWA）
  - 離線能力（需要支援司機在網路不稳地區）
  - 即時推播（LINE 通知 / 自建推播）
  - 認證機制（延續既有的 Supabase Auth）
