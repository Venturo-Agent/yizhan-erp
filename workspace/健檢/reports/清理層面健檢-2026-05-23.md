### A3. 為什麼會這樣 / 業務白話

過去這套 code 是「**先建梯子、後決定要不要爬**」：
- 建 module 時把 read / write / manage / 各種子 tab 的 capability 一次宣告完
- 實際寫 API 時、只有少數 capability 真的被 `requireCapability(CAPABILITIES.XXX)` 守門
- 結果：HR 頁面（/hr/roles）列給 admin 勾的選項裡、**有一半勾了也沒效果**（因為沒人查那個 capability）

對 William 的影響：
1. **客戶 admin 在 /hr/roles 看到「會計-檢查單寫入」這個權限**、勾或不勾結果都一樣（沒人查）→ 客訴
2. **新人 onboarding 看 capabilities.ts 不知道哪個活的哪個死的**、誤刪活的或誤加新死碼
3. **DB role_capabilities 表會累積死 row**（admin 勾過的死 capability 也存）

### A4. 修法（建議 William 拍板）

**Phase A（這週、低風險、純 code 層）**：
1. 寫小腳本 `scripts/audit-dead-capabilities.ts`、列 87 個死碼
2. 對每個死碼判斷：
   - 「未來會用」→ 加 `// TODO(M2): wire up` 註記、保留
   - 「打算砍」→ 從 capabilities.ts 刪 + 從 features.ts / module-tabs.ts 同步刪
3. 跑 `npm run codegen:permissions` 重 generate

**Phase B（DB 層、要寫 migration）**：
- 清完 code 後、寫 migration 刪 role_capabilities 對應 row（防殭屍 row 累積）

**估時**：Phase A 半天、Phase B 1 hr
