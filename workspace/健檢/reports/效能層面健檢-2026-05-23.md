## A. 新發現：全鏈路延遲分解（5/20 沒做）

### A1. 單 request 累計延遲（從 client click 到 response）

```
[user click]
  ↓ 5-20 ms 本地 → CDN
[CDN edge]
  ↓ 50-150 ms 跨 Pacific（user 在台灣 ↔ Vultr Tokyo）
[Vultr Tokyo: Coolify handler 啟動]
  ↓ 10-30 ms cold start / warm hit
[Next.js route handler]
  ↓ 30-80 ms supabase.auth.getUser() 打 GoTrue（每 request 1 次、無 cache）
  ↓ 5 ms 查 employees.role_id
  ↓ 5 ms 查 role_capabilities
  ↓ ★ Logan 警告：proxy.ts + getServerAuth 各打 1 次 auth.getUser、totalauth ms 雙倍 ★
[業務 DB calls]
  ↓ N × 70-90 ms（Tokyo Vultr ↔ Singapore Supabase, 5-10 次 sequential、實測 RTT 待 SSH 驗）
  = 350-900 ms 純等網路
[DB triggers]
  ↓ N × 0.5-10 ms（trg_messages_set_read_state、audit log、updated_at 等、實際成本待 Task 5 查）
[response 回程]
  ↓ 同來程
```

**單按鈕總延遲推估**：**500-1500 ms**（70% 是「等網路 / 等 auth」、不是業務邏輯本身）

### A2. 最肥 top 5 route（Task 1 從 159 條 route 抓出）

| 路徑 | DB call 數 | 備註 |
|---|---|---|
| `POST /api/line/webhook` | ~18 | LINE 訊息 webhook、無守門但內部巨 |
| `POST /api/accounting/period-closing` | ~14 | 月結 |
| `GET /api/accounting/opening-balances` | ~11 | 期初餘額 |
| `POST /api/accounting/receipts/[id]/refund` | ~11 | 退款 |
| `POST /api/hr/bonus-settlements/settle` | ~10 | 業績結算 |

⚠️ 數字「ranking 對、絕對值偏高」— spot check 顯示 Task 1 把 helper 內 DB call 算進去、是估值不是精確值。

### A3. 全站共通 pattern（最大 leverage）

- **104 個 route**（65%）走 `requireCapability` chain、每個跑「1 GoTrue + 2-3 employees/role_capabilities 查」
- **99 個 route**（62%）用 admin client 繞 RLS、手動驗 workspace_id
- **71 個寫入 route** 每個 +1 RPC（`set_audit_context` ~2.5ms）
- **33 個肥 route**（21%）有 ≥5 sequential DB call、只 9 個用 `Promise.all`（6%）— 改 parallel 是低風險 leverage

## C. Logan 補的 5 個沒查面向（要 MCP 才能查）

### C1. Storage egress 成本

- yizhan-erp 有沒有用 Supabase Storage？grep `.env*` 找不到 STORAGE 設定、可能沒用
- 如果有用、每次 user 下載檔案計 egress 費、隨 user 增加線性成長

### C2. Postgres autovacuum / table bloat

- 長期營運 dead tuple 累積 → query 越來越慢
- 要查：`SELECT relname, n_dead_tup, last_autovacuum FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 20`
- **卡 MCP**

### C3. Connection pooler 沒實際確認

- 升等概念 H 條提到、但 grep `.env*` 找不到 `DATABASE_URL` / `pooler` 字樣
- 可能：app server 直接連 Postgres、不走 pooler、高並發會 connection exhaustion
- 要查 Supabase 連線字串實際長相

### C4. RLS function recursion

- RLS policy call `get_current_user_workspace()`、function 內又觸 RLS check → 每 row check 跑 2x policy
- 要查：`pg_get_expr(pol.polqual, pol.polrelid)` 看 pattern
- **卡 MCP**

### C5. Postgres statement_timeout

- 缺 index 的 query 會佔住 connection、卡住其他 request 全部
- 建議 set 5-10 秒上限
- **卡 MCP** 才能查現在有沒有設
