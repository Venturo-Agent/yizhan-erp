## 0. 一句話總結

抽象層（entity hook + apiMutate）方向是對的、做進去的天生達標；問題出在**讀取層沒有完整 SSOT**：

1. **最常用的幾個列表（收款 / 請款 / 出納 / 訂單）為了效能改成「自訂分頁查詢」，集體失去了「同事改了即時同步」的能力**——它們繞過 entity hook、沒接廣播線。自己改只能靠手動補刷（散落、易漏）。
2. **字典型資料（員工 / 供應商 / 銀行 / 國家 / 會計科目）讀法各搞各的**——有的永久快取+廣播（對）、有的每次開都重打 DB（浪費）、有的沒廣播（同事改看不到）。
3. **訂單成員編輯在搬 SSOT 時被改成半套**（直接寫 DB + 畫面暫存硬追、沒撕小抄），所以時好時壞。
4. **發現 code↔production 漂移**：有 migration 號稱補了廣播，但 production 實際沒套上（見 §3 校正一）。

## 1. William 的北極星（驗收標準）

| 訴求 | 技術機制 | 白話 |
|---|---|---|
| 自己 增 / 刪 / 改 → 畫面立刻反映 | 寫入後撕小抄（apiMutate / entity invalidate）重撈 | 改完自己馬上看到、不用 F5 |
| **同事** 增 / 刪 / 改 → 我畫面跟著動 | DB realtime 廣播 → entity hook 內建 useRealtimeSync 收到自動刷 | 同事核准一張單、我列表上那張立刻變狀態 |
| 很少變的資料（下拉）→ 不要每次重讀、但變了要同步 | 永久快取（staleTime Infinity、零讀取成本）+ 廣播推送（變了才刷一次） | 電話簿貼牆上、總機有人異動才換那頁 |

「自己改」與「同事改」是**兩套不同機制**：前者靠撕小抄、後者靠廣播。自訂查詢列表把兩套都丟了。

## 3. 兩個關鍵校正（主對話親驗、推翻 subagent）

### 校正一：廣播缺口——code↔production 漂移（Max 用 live DB 查證）

subagent 讀 migration 檔說「ai_products/brands/checks 已於 20260524 補進 publication、無缺口」。**錯**。Max 直接查 production 的 `pg_publication_tables`（ground truth）：

| 表 | live publication 實況 |
|---|---|
| ai_products | ❌ MISSING |
| brands | ❌ MISSING |
| checks | ❌ MISSING |
| journal_vouchers | ❌ MISSING |
| roles | ❌ MISSING |
| branches | ❌ MISSING |
| orders / order_members / receipts / payment_requests / notes / workspaces | ✓ IN_PUBLICATION |

**意義**：有 migration 寫了「補廣播」但 production 實際沒套上 → 違反 Migration SOP（寫了要 apply + 驗證）。entity hook 訂了 ai_products/brands/checks 的廣播、但喇叭沒開 → 同事改這幾張表，別人靜默收不到。
**待 William 拍板**：這幾張是否該補廣播（一個 migration apply）；roles/branches/workspaces 走 API 非 entity、是否本就不需 realtime（多人同改頻率低，可標 N/A）。

### 校正二：自訂 ListView 完全沒接廣播（Max 親讀原始碼）

subagent 對「收款列表同事改同不同步」判斷不一。Max 親讀 `useRequestsListView.ts` 全文（請款列表）：內部就是純 `useSWR(自訂key, () => supabase.from('payment_requests').select())`，**零 useRealtimeSync、零 supabase.channel**，dedupingInterval 5 分鐘。

**意義**（適用所有 *ListView：收款/請款/出納/訂單）：這些列表為了 server 端複合篩選+分頁，刻意繞過 entity hook，**代價是丟掉 entity hook 內建的 realtime**。
- 自己改：靠 page 的 `refreshAll`（invalidateXxx + refreshListView）手動補。補對的即時、漏接的不即時（請款「修改」就漏了）。
- 同事改：**一律收不到**，最多等 5 分鐘 dedupe 過期或換頁/F5。

這是整份盤點最核心的系統性結論——漫途最常用的財務+訂單列表，正是 William 說「同事核准/刪除我看不到」的重災區。

## 4. 訂單成員編輯專項診斷（William：「搬 SSOT 把它改得亂七八糟」——體感屬實）

> **為何單獨做（William 2026-05-27 補充）**：成員牽涉「辨識」（證件/護照辨識、跟客戶資料比對）+「驗證」（號碼格式、身份核對），比一般表多兩層業務邏輯，走自訂流程是正當的、不是被亂搞。
> 根治原則：**保留整套辨識/驗證邏輯不動，只在存檔收尾補撕小抄**——不是把它硬塞回陽春 entity hook。下次重構不可再把它當普通表硬搬（已壞過一次）。這也是最該補「守門測試」的對象（見 §7 第 4 點）。

成員 CRUD 在 `entities/order-members.ts` 有完整 entity hook（createOrderMember/updateOrderMember/deleteOrderMember + invalidateOrderMembers），但業務層繞過它直接寫 DB：

| 動作 | 位置 | 現況 | 病灶 |
|---|---|---|---|
| 新增成員 | useOrderMemberActions.ts:117 | `supabase.insert` + `setMembers(local)` | 沒撕小抄→換頁/別人看不到 |
| 編輯成員 | useMemberEditDialog.ts:148 | `supabase.update` + `setMembers(local)` | 直接寫、沒走 entity hook、沒 invalidate |
| 刪除成員 | useOrderMemberActions.ts:175 | 走 entity delete（會撕小抄）… | …但被 `recalculate({skipInvalidate:true})` 軟阻廣播 |

**為什麼時好時壞**：靠 `useOrderMembersRealtime` 救火（只救「該訂單展開表」、不救訂單列表）。正軌應改回 entity hook 或補齊 invalidate；並檢討 `skipInvalidate:true` 這個「怕整頁重刷」的設計陷阱（結果讓團/訂單金額跨頁不同步）。

| 破口 | 位置 | 現況 | 成本 |

對照「做對了」的標準答案：員工/供應商/請款類別走 entity hook = staleTime Infinity（永不過期、零 polling、零多餘 egress）+ dedupe 2s + 廣播推送。**字典資料全部該收斂到這個讀法**。

## 6. 根因：讀取層沒有完整 SSOT

| 散落形態 | 代表 | 後果 |
|---|---|---|
| 自訂分頁 useSWR 繞過 entity hook | 收款/請款/出納/訂單 ListView | 失廣播（同事改不同步）+ 自己改靠手動補（易漏） |
| 直接 supabase→useState | 會計科目/傳票/支票操作/筆記 | 改完不刷、要手動重撈 |
| 走 REST API 非 entity | roles/branches/workspaces | 無廣播 |
| migration 漂移 | ai_products/brands/checks/journal_vouchers 沒進 publication | 喇叭沒開 |
| entity hook 規範破損 | 訂單成員直接寫 DB | 半套、時好時壞 |

## 7. 根治方向（待 William 拍板）

1. **訂「讀取 SSOT 分類表」當鐵律**：每種資料按「多久變一次 × 要不要多人即時」歸檔，每格綁死讀法（走哪個 hook + 快取多久 + 要不要廣播）。新資料照表填、不准自己發明。
2. **全遷**：列表(含自訂 ListView 補廣播或補手動刷雙保險) + 下拉字典(收斂到 entity dictionary) + 訂單成員(改回 entity hook)。
3. **裝哨兵防回退**：
   - 把 `venturo/no-direct-useswr-in-pages` 偵測範圍從 page 擴到 `_hooks/`（堵請款那種「包進 hook 就漏網」）。
   - 加偵測「自訂 SWR list + 寫後沒雙刷 / 沒接廣播」。
   - 既有 `audit:realtime` 加 `--db` 比對自動化、抓 migration 漂移（這次手動抓到的）。
4. **補守門測試（防「修這壞那」的關鍵一層）**：給「絕不能被改壞」的關鍵體驗各補一個守門測試——即時刷新（增/刪/改後列表跟著動）、計算式輸入、訂單成員辨識/驗證。任何重構（資安 / SSOT / 效能）動到它，CI 當場紅、不靜默退化。
   - 對比實證：計算式有 18 個測試守著 → 5/25 被資安改動波及時當場抓到、同日修回；讀取層多數無測試 → 請款修改不刷這種靜默流血、等用戶發現。**有沒有守門員，就是「修這壞那」的分水嶺。**
5. **補 production 漂移**：確認 ai_products/brands/checks/journal_vouchers 是否補廣播、apply migration、驗證。
