## B. CI / CD 健康度

### B1. 5 個 workflow 現況

| Workflow | 觸發 | 用途 | 狀態 |
|---|---|---|---|
| `ci.yml` | push / PR | 3 jobs: quality (lint + type) / build / e2e-smoke | ✅ 跑 |
| `audit-rls.yml` | push / PR | 6 層 blueprint audit、需 `SUPABASE_DB_URL` secret | ⚠️ secret 未設、降級為 code-grep |
| `bundle-size.yml` | PR | 檢查 bundle 大小變化 | ✅ 跑 |
| `standards-check.yml` | PR | 跑 `scripts/check-standards.sh --strict` 守 10 條憲法 | ✅ 跑 |
| `line-flush-cron.yml` | cron */5min | LINE Bot 訊息 flush（非品管、是業務功能） | ✅ |

### B2. CI 缺口（待補）

1. **`audit:writes`** 沒在 CI 跑 → trigger × API 雙寫不會擋
2. **`audit:realtime`** 沒在 CI 跑 → publication 缺口不會擋
3. **`audit:orphans`** 沒在 CI 跑 → auth.users 孤兒累積
4. **`SUPABASE_DB_URL` secret** 未在 CI 設 → `audit:rls` 退化 code-grep

**建議補一個 audit-aggregate workflow**：把 audit:writes / audit:realtime 加進 PR 強制 check。`audit:rls` 看是否願意加 secret。

### B3. e2e/full 在 CI 走哪一塊

- ci.yml 的 e2e-smoke job 只跑 `smoke.spec.ts` 一個檔
- tests/e2e/full/* 20 個檔的全鏈路測試**只在本地 slow.config 手動跑**
- 4/20 事故那種「全員登不進來」靠 login-api.spec.ts 守
- 但 critical-path-prod.config / critical-path-prod.spec 是針對 production 後驗、不是 PR 守門

## 結論

**靜態品管 90% 綠燈**（pre-commit + type-check + ratchet + check-standards 都到位）。
**動態 / runtime 品管缺口**：CI 的 DB 連線不通、audit:rls 退化、Realtime e2e 沒測。
**最快 ROI**：把 `audit:writes / audit:realtime` 加 CI（不需 DB、純 grep）、防同類雙寫事故再發生。
