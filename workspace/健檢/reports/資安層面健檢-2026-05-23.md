## 紅旗（必修、優先級排序）

### 🔴 P0 — 即刻修（資安風險）

**1. `getSupabaseAdminClient` 在 module top-level 呼叫**
- 位置：`src/app/public/contract/sign/[code]/page.tsx:5`
- 內容：`const supabase = getSupabaseAdminClient()`
- 影響：違反紅線 C「admin client per-request」。雖然 `getSupabaseAdminClient` 本身每次都 `createClient` 新建（不真的是 singleton）、但 module top-level 只執行一次、整個 server lifecycle 共用同一個 client instance
- 風險評估：**中等**。`createClient` 不真的建連線、只是 HTTP fetcher、`autoRefreshToken: false` + `persistSession: false` + per-call `.from(...)` 不帶狀態。但**違反 SOP 字面**、若未來改 admin client 加 cache / context、會 leak
- 修法：把 `const supabase = ...` 移進 `export default async function Page(...)` 內、每 request 內建
- 業務語意：公開合約簽署頁、不是 sales path、低流量、修起來 1 行

### 🟡 P1 — 兩週內

**2. Auth orphan 1 筆累積**
- `qaz1300233@gmail.com`、`id=5d8bb5e2-6dff-415d-9913-d5074c7b0a00`、5/21 註冊、`auth.users` 有但 `employees` 沒
- 影響：低（用戶看不到、但累積會撞「同 email 再註冊」`23505 unique violation`）
- 修法：`npm run audit:orphans -- --clean`（互動式確認後清）
- 根因：多步註冊（建 auth.users → 建 employees → 建配套）某步失敗、rollback 漏清 auth.users（這是業界常踩的坑、Supabase 沒提供 transactional 多 step）

## 紅線 D 補強重點變動（5/22 之後）

**5/22 新加 `assert_tour_active_for_financial_doc` DB trigger**（`20260522220310_block_draft_tour_financial_writes.sql`）：

- 行為：BEFORE INSERT on `receipts` / `payment_requests`、查 `tours.status`、若是 `template` / `proposal` → reject
- 業務語意：客人還在報價階段、不能開正式收款 / 請款單。客戶端守門可繞過、DB trigger 是最後一道保險
- 為什麼是紅線 D 補強：過去客戶端守門（`usePaymentForm` / `useReceiptMutations`）能被繞過、加 DB trigger 後就算用 SQL 直連也擋
- 不擋 UPDATE — 老單據若 tour 變回 proposal（理論不應該）不會炸、只擋新建

**仍缺**：
- `journal_vouchers` 跨期間鎖定：沒有 closed period 概念
- `disbursement_orders`：沒有 lock guard
- 評估：業務嚴格度由 William 拍板。目前漫途規模還沒到需要「月結後 100% 不能改」的稅務要求

## 5/21 紅線 H 第一個案例修補（已 apply）

**Migration**：`20260521170000_fix_expense_categories_rls.sql`

**過去漏洞**：`expense_categories` 表早期建的、RLS policy 只檢 `auth.role() = 'authenticated'` → **任何登入用戶可讀寫任意 workspace 的請款類別**

**修法**：
1. backfill `workspace_id` from `user_id`（legacy 欄位）
2. DROP 弱 policy
3. 加 SELECT policy：`workspace_id IS NULL OR workspace_id = get_current_user_workspace()`（系統預設 + 自家）
4. 加 ALL policy：`workspace_id = get_current_user_workspace()`（只能動自家）

**為什麼這條重要**：紅線 0/A/F/G 各守不同層、沒一條明說「業務表 RLS 必過 workspace」基本守則 → 紅線 H 5/21 加進來 CLAUDE.md。`expense_categories` 是第一個 grandfather catch、後續會加 audit 自動偵測剩餘類似表
