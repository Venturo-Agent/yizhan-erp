## 關鍵：UI 實測要「清單/規格驅動」、不能「隨便點」

**William 的精準顧慮**：「如果有程式碼沒做 UI、UI 測試會發現嗎？還是會越過？」

**答案：純『在頁面上隨便點』會靜默越過缺的 UI** —— 沒按鈕可點 = 它根本不知道要測 = 不會 fail、只會「沒測到」。這就漏了「後端做了、前端沒做」這類洞（合約功能本來就是這樣：API/表都有、UI 是 stub）。

**解法：清單驅動** —— 先從「這功能『該有』哪些 UI」推導出清單、再逐筆驗「在不在 + 動不動」。缺 UI = 該筆 **FAIL**（不是靜默跳過）。

清單從哪來（接既有 5 SSOT）：
- 路由（`app/`）→ 每條路由該有可進的頁面（缺=404 fail）
- capability（capabilities.ts）→ 有 capability 就該有對應 UI 能用（有權限卻沒 UI = fail）
- feature（features.ts）+ module-tab → 開了 feature 該看到對應入口
- 換句話說：**5 SSOT 已經列了「每個功能該有什麼」、UI 測試清單就從這推導** → 自動抓「路由有但頁面缺」「capability 有但沒 UI」「後端有但前端沒做」

## 例外（別誤報）
- 純後端（API-only / cron / webhook）本來就沒 UI → 清單標 N/A、不算缺
- 凍住/半成品 module（travel_invoice / office）→ 已知、不當 bug
