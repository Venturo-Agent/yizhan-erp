## 1. 讀取引擎（紅線 F 其實沒做到）

| 引擎 | 數量 | 代表 | 快取行為 | 風險 |
|---|---|---|---|---|
| EntityHook `useList`/`Slim`/`Detail` | 48 entity | `orders.ts:20`、`todos`、`channel_messages` | idb 讀一次墊檔、**無 keepPreviousData** | 換 filter/key 時清單會閃成空或舊快照 |
| EntityHook `usePaginated` | 同 48（方法）| `createEntityHook.ts:443` | **keepPreviousData**、**無 idb** | 不閃但冷啟轉圈 + 每次 count 查詢（egress）|
| `createReportHook` | 8 個財務 hook | `useReceivables.ts:38`、`usePayables.ts:37` | 無 idb、無 keepPrev、fetcher 內多表 join | 第二個讀取工廠（合理但要承認）|
| 裸 `useSWR`（紅線 F 違規）| **~33 個 call site**（baseline 只算 18）| 見下 | 多數無 idb | 真實違規數被低估 2–3 倍 |
| `src/data/hooks/` 逃生艙 | 6+ | `useRoles`、`useBranches`、`useMessagingConversations` | REST 包裝 | **故意放這逃 lint 規則**（註解自承）；未編目的第三套 |
| 直連 `supabase.from().select()`（client，無快取）| ~10 | `tour-edit-dialog.tsx:69`、`library/customers/[id]/page.tsx:229` | 完全無快取、effect/handler 內直打 | 最高：繞過快取+realtime，每次 mount/click 打 DB |

**關鍵發現**
- **同一個 entity，兩種相反的快取語意**：走 `useList` 會 idb 墊檔不保留前值（換 filter 閃）；走 `usePaginated` 保留前值但冷啟轉圈。挑哪個方法決定行為。
- **`CACHE_PRESETS.high/medium/low` 幾乎是裝飾**：三者 ttl/staleTime/dedupe 設一樣，工廠又忽略這些值（dedupingInterval 寫死 2000ms），唯一差別是 `low` 關掉 `revalidateOnReconnect`。挑 high 還 low 幾乎沒差。
- **最大一群裸 useSWR 的根因 = entity hook 的 filter 只支援 `.eq()`**。`useOrdersListView` / `useReceiptsListView` / `useRequestsListView` / `useLinkedPaymentRequests` / `useToursPaginated` 都因為需要 OR / IS NULL / 複合條件,只好自己刻。**補強 `usePaginated` 的 filter API,這群就能折回來、然後刪掉。**

## 3. 寫入 / 失效 / 樂觀更新

**兩條正確 SSOT 路徑（SAFE）**：`apiMutate`（body/formData/blob/successFlag/optimistic + 單參數 invalidate）、`entityHookCrud`（function-as-data 樂觀 + `invalidateEntity` 單參數）。問題是**採用率只有部分**。

- **沒配中央失效的（脆弱）**：finance payment services（`disbursement-order.service.ts` 11 寫、payment-request 系列共 32 寫,**零失效**,全靠呼叫端記得刷)、`ReceiptTransferDialog`、`OrderStatusChangeDialog` 等靠 `onSuccess` prop 往上冒。呼叫端忘了就 stale。

## 5. 訊息（AI Hub + Channels）為什麼「常常這樣」

**共同根因**：兩邊送出**都不是樂觀更新**——訊息要等 POST 回來 + 重抓才出現;而且收訊息**完全依賴 realtime**(AI Hub 用 broadcast、Channels 用 postgres_changes),code 自己註解就承認 realtime 在 hard reload 前不穩(INITIAL_SESSION JWT race),**又沒有 polling 後援**。任何一次 realtime 打嗝 = 晚出現 / 要重整。

**AI Hub 額外問題**
- 過濾後的 key(`?channel=line`)和 sidebar 的未過濾 key 不一致 → 一邊 stale（`AiConversationsTab` vs `AiSidebar`/`useMessagingConversations`）。
- `AiConversationsTab` 和 `AiSidebar` **各開一條同名 broadcast channel + 各註冊 useRealtimeMutate** → 一個事件觸發多達 4 次 mutate,放大閃爍、雙倍 Supabase 負載。
- `refreshInterval` 已被移除(`:201-205`),沒有 polling 後援。

**Channels 額外問題**
- 草稿瞬間清空(`ChannelView.tsx:159`),泡泡要等 `await invalidateChannelMessages()` 才落地 → 中間有空檔。
- 自己送的訊息:本地重抓 vs postgres_changes 回音 race 同一個 cache key → 間歇閃。
- idb 讀一次墊檔 → 切頻道先空再填。

## 7. 效能 / egress 帳（William 排序：資安#1 → 效能#2 → SSOT#3）

> Supabase 按讀取量收費，所以 SSOT 沒做到＝同時在扣「效能#2」。以下是結構性判斷、**未實測 egress 數字**，按影響大小排。

| # | 成本來源 | 機制 | 影響 | 修在哪層 |
|---|---|---|---|---|
| 1 | **`useList` 撈整張表** | `PAGE=1000` 迴圈撈到完（非分頁）| 每次掛載 + 每次 realtime 事件都把整表重拉。違反自己定的「列表載少」| B/C（最大宗）|
| 2 | **realtime 全表重拉 + 放大** | 訂 `postgres_changes *` 無 filter；一列變動 → 所有訂閱者各重拉整表 | tours/calendar 雙訂閱、AI Hub 一事件 4× fan-out、publish-but-unused 表空轉廣播 | B/C |
| 3 | **`usePaginated` 每次 `count:'exact'`** | 每次載入/換 filter 跑全表精確 count | 大表很貴，財務/訂單列表每次付 | B |
| 4 | **無快取直連讀** | `tour-edit-dialog` 每開就 `workspaces.select('*')`、`library/customers` 存檔後手動重撈 | 繞過 5min dedupe，同 query 反覆打 | B |
| 5 | **多引擎重複快取同資料** | tours 被 entity hook + `useTours-advanced` + `useToursPaginated` 各撈各存 | 同資料抓三遍、不共用 | C |

**分層結論**：A 層（footgun/lint）主要省「閃爍/正確性」，效能其次。**真正的 egress 大頭在 B/C**——「useList 撈整表」＋「realtime 全表重拉」。要省 Supabase 錢，重點是這兩條，不是 A 層。

## 8. 核心不變式（這次定錨的新概念）

> 過去反覆復發，是因為只修症狀、沒有一條不變式。這次把它寫死：

**不變式：畫面上的資料，永遠不會在 session 中途變成 `undefined`。**

- **失效只走「單參數 revalidate」**：`globalMutate(key)` / `mutate(key)`——保留現值、邊背景刷新。
- **永不把 cache 寫成 `undefined`**：那是「清空」不是「刷新」。唯一能清空的時機是登入/登出（`clearAllSwrCacheKeys`，紅線 G）。
- **要寫 cache 一定傳真實 data 或 updater function**（樂觀更新、登入預寫 layout-context 都合法）。

**三層守護**（缺一就會復發）：
1. **寫入/realtime 都走單參數** — entityHookCrud / apiMutate / 8 個 footgun / realtime handler 全部對齊。
2. **lint `venturo/no-mutate-clear`（error）** — 擋住 `mutate(key, undefined, …)` 再被種回去（連我自己以後都擋）。
3. **`keepPreviousData` 當保底**（待資安驗證後再加，見 §9）— 萬一未來仍有人清成 undefined，畫面停在上一筆好資料、不暴露 stale fallback。

**概念定位**：這是把紅線 F 從「**入口統一**（讀取走 entity hook）」推進到「**行為統一**（有一條不變式 + 機器強制）」。入口統一只防散刻，行為不變式才防復發。
