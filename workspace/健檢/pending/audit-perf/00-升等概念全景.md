## 概念：效能升等不是只動 code、是動「全鏈路」

把網站想成「客人從進門到拿到菜」的餐廳：

| 環節           | 餐廳譬喻                 | 我們現況                         |
| -------------- | ------------------------ | -------------------------------- |
| 客人到櫃台     | 顧客走進門               | user 點按鈕 → 經 CDN（沒設）     |
| 點餐單送廚房   | 服務生跑單               | request → Next.js（Vultr Tokyo） |
| 廚房問食材庫存 | 廚房對倉庫 walkie-talkie | API → Supabase（可能在 US！）    |
| 倉庫拿菜       | 倉庫員撈食材             | DB 查 + RLS + trigger            |
| 倉庫回廚房     | 走回廚房                 | DB response 跨太平洋回           |
| 廚房煮         | 烹調                     | server processing                |
| 菜送客人       | 出餐                     | response → user                  |

**現在每個環節都有油可榨、3 把刀只動了 1 條鏈路（auth + capability check）。**

## 升等的 10 個面向（按 ROI 排序）

### 🔴 P0 — 最大可能 leverage（先驗證再動）

#### A. **跨太平洋延遲：Vultr Tokyo ↔ Supabase US?**

- **現況懷疑**：app server 在 Tokyo（Vultr）、Supabase project URL 是 `aawrgygqgemgqssflfrx.supabase.co`、預設 region 沒確認
- **如果真跨洋**：每次 DB call 100-150ms 純網路、每個 API 5-10 次 DB call = **500-1500ms 純等網路**
- **省的 ms**：跨洋變同區（Tokyo / Singapore）→ 每次 DB call 從 100-150ms 降到 5-20ms = **每按鈕快 500-1300ms**
- **動法**：Supabase dashboard 看 project region、如果不在 Tokyo / Singapore：
  - 選項 1：Supabase 不支援搬 region、要新建 project + 資料遷移（大工程）
  - 選項 2：app server 從 Vultr 搬去 region 跟 Supabase 同（破 Tokyo 配置）
  - 選項 3：兩邊都搬到 Singapore（亞洲匯流點、對台灣 / 大陸都 OK）
- **動之前**：先確認 region（10 秒、Supabase dashboard 看一眼）
- **這條若成立、是全站最大 leverage**

#### B. **HTTP 快取沒設：reference data 159 條 API 只 1 條有快取**

- **現況**：grep 159 條 API、只 `fetch-image/route.ts` 有 `Cache-Control`
- **問題**：銀行清單、國家清單、機場、客戶類型等「半天才變一次」的資料、每次 user 操作都重新 query
- **動法**：reference data 類 API（list-banks / countries / airports 等）加 `Cache-Control: public, s-maxage=3600, stale-while-revalidate=86400`
- **效益**：被快取的 API 從 200ms 降到 10ms（CDN edge 命中）、每 user 每天少打 200+ 次同樣 query

#### D. **DB index 對齊真實 hot query**

- **現況**：migration 已建 268 條 index、但 production 真實最慢 SQL 是哪幾條沒人確認過
- **問題**：index 有但不一定對齊 query（譬如 `WHERE workspace_id = ? AND status = ?` 沒 composite index、得 seq scan）
- **動法**：跑 `pg_stat_statements` 抓 top 50 慢 SQL → 對每條 `EXPLAIN ANALYZE` → 補對應 composite index
- **卡關**：要 Task 3 報告（卡 MCP）才能精準動
- **效益**：top 慢 SQL 從 100-500ms 降到 5-30ms

#### E. **N+1 寫入：批次操作的 client 端 sequential await**

- **Task 2 抓到**：18 個 N+1 hotspot（收款建立 / 確認、請款單建立、月結等）
- **最痛**：會計按「批次確認 10 筆收款」、client 跑 `for (...) { await POST }` = 10 × (200-500ms) = **2-5 秒卡死**
- **動法**：改 `Promise.all` 並發 + server 端 transactional batch endpoint
- **效益**：批次操作 10x 加速、user 體感差別最大的場景

### 🟢 P2 — 中等 leverage（可順手）

#### F. **圖片散刻 `<img>`、64% 沒用 next/image**

- **現況**：25 處 `<img>` vs 14 處 `next/image`
- **問題**：`<img>` 不會 auto webp、不會 responsive size、不會 lazy load、客戶端載一張原圖 2-3MB
- **動法**：grep 出 25 處、改 next/image
- **效益**：頁面初載時間降 20-50%、流量省 70%（webp 壓縮）

#### G. **Realtime 計費風險**

- **現況**：12 處 subscription
- **問題**：Supabase Realtime 按 concurrent connection 計費、每個 user 開幾個 channel 就計幾個
- **動法**：審視 12 處、看有沒有「user 一進 page 就 sub 一堆 table」、改成「按需 sub」
- **效益**：concurrent connection 數降、Supabase 月費降

#### H. **連線池（PgBouncer）確認**

- **動法**：確認 `DB_URL` 是用 pooler endpoint（`*.pooler.supabase.com:6543` transaction mode）而不是 direct（`*.supabase.co:5432`）
- **效益**：concurrent request 高時 connection exhaustion 不會發生

---

### ⚪ P3 — 低 leverage（最後動）

#### I. **CDN 前置**

- **現況**：Coolify on Vultr Tokyo、沒前置 CDN
- **動法**：Cloudflare 前置（DNS 已在 Cloudflare、加 proxy 即可）
- **效益**：static asset（next/image 產出的、\_next/static）邊緣快取、Asia/Europe/US 用戶都快
- **風險**：要驗 Coolify auto-deploy 跟 Cloudflare cache 互動正常

#### J. **Edge runtime 機會**

- **動法**：純讀 reference data API 改 `export const runtime = 'edge'`、跑在 Vercel/Cloudflare edge
- **效益**：cold start 從 200ms 降到 5ms
- **限制**：edge runtime 不支援所有 Node API、要逐條審視
