## 0. 這份規格在解決什麼（一句話）

讓「自己增刪改即時反映、同事增刪改自動同步、全程不用 F5」成為**全站每個列表的硬標準**，並用「分類鐵律 + 守門機制」保證它**不再被重構波及而靜默退化**。

---

## 1. 北極星 / 驗收標準（William 親口）

> 「刪除或修改時立刻消失、新增時立刻出現，**不可以重新整理畫面**。而且同事同時新增 / 核准 / 刪除時，我這邊也要一起跟著動。」

三條驗收線（缺一不算達標）：

| # | 驗收項 | 機制 | 白話 |
|---|---|---|---|
| V1 | 自己 增/刪/改 → 當前畫面立刻反映 | 寫入後撕小抄（invalidate / refresh）重撈 | 改完馬上看到、不 F5 |
| V2 | **同事** 增/刪/改 → 我畫面跟著動 | DB realtime 廣播 → hook 自動刷 | 同事核准一張單、我列表那筆立刻變 |
| V3 | 很少變的資料 → 不每次重讀、變了才同步 | 永久快取（零讀取成本）+ 廣播推送 | 電話簿貼牆、總機異動才換頁 |

## 3. 【核心】讀取 SSOT 分類表（鐵律）

> 這是本規格的靈魂。每種資料按「**多久變一次 × 要不要多人即時**」歸到一類，**每類綁死讀法**。新資料照表填、**不准自己發明讀法**。違反 = commit 哨兵擋（§6）。

| 類別 | 代表資料 | 變動頻率 | 多人即時 | **指定讀法（SSOT）** | 快取策略 | 廣播 |
|---|---|---|---|---|---|---|
| **C1 業務交易列表** | 收款/請款/出納/訂單/旅遊團/報價 | 高 | 要 | entity hook `usePaginated`；**若需複合篩選只能用自訂查詢時，必須補 (a) 寫後雙刷 (b) useRealtimeSync** | staleTime Infinity + dedupe 2s | **必接** |
| **C2 字典 / 下拉（電話簿型）** | 員工/供應商/付款方式/科目 | 低 | 要（少變但要同步） | entity `useDictionary` / `useListSlim`，**server 端 filter、不 `all:true` 前端過濾** | staleTime Infinity（零成本） | **必接**（表入 publication） |
| **C3 全域 master（平台級）** | 國家/機場/銀行代碼 `ref_*` | 極低（年改數次） | 不需 | 集中 hook + 自訂 useSWR 長快取（可接受、標 N/A 廣播） | dedupe 1hr+ | 可不接 |
| **C4 子表 / 明細（隨父刷）** | 請款明細/行程項目/訂單成員 | 隨操作 | 要 | entity inherited hook；**父操作收尾一起 invalidate 子表 key**；成員額外保留辨識/驗證邏輯 | 跟父 | 必接 |
| **C5 聚合 / 報表** | 應收應付/業績/銀行餘額 | 衍生 | 通常即時 | `createReportHook`；**來源表寫入後 invalidate 報表 key** | 短~中 | 來源表廣播觸發 |
| **C6 外部聚合（特殊）** | inbox 對話/訊息（多通路） | 高 | 要 | 自訂 + **custom broadcast**（webhook/API 推）；非 postgres_changes | — | broadcast event |

**鐵律補充**：
- C1 的「自訂分頁查詢」是效能例外（複合篩選 entity hook .eq chain 不支援），**但例外必須付出代價：補雙刷 + 補 realtime**，否則違反 V1/V2。這正是收款/請款/出納/訂單現在的破口。
- C2 字典「永久快取 + 廣播」= 即時與省錢的同一招（員工/供應商已做對，當範本）。
- C3 `ref_*` 是平台級共用、RLS 無 workspace 隔離，不走 entity hook 是合理的，廣播標 N/A。

## 4. 完整施工清單

> 格式：問題 → 修法 → file:line（信心）→ 風險。標「待親驗」者，動手前必走 page-scout + fix-safe 再確認。

### P0 — 上線前必修（建議今天動手）

**P0-1 請款單「修改 / 刪除」不刷新（V1 破口、已坐實）**
- 問題：`AddRequestDialog.edit-ops.ts` 的 `saveEditedRequest`(L71 收 onSuccess 沒呼叫) 與 `deleteEditedRequest` 都沒觸發 page 的 `refreshAll`（含 refreshListView）→ list-view key 不刷、金額卡舊值。
- 修法：存檔/刪除成功收尾呼叫 `_onSuccess?.()`（page.tsx:119 refreshAll 已備好，含 invalidatePaymentRequests + refreshListView）。
- file:line：`edit-ops.ts:182`(save 成功後)、`edit-ops.ts:239`(delete 成功後)（高信心，已親驗 save 路徑）
- 風險：極低（補呼叫既有 callback）。守門：補 P0-T1 測試。

**P0-2 廣播漂移：補表進 publication（V2 破口）**
- 問題：live DB 確認 `ai_products`/`brands`/`checks`/`journal_vouchers` 不在 supabase_realtime publication，但 entity hook 已訂閱 → 喇叭沒開。code↔production 漂移（migration 寫了沒 apply）。
- 修法：先確認各表是否真需多人即時（傳票/支票財務多人共看→該補；品牌/AI商品→評估）。需要的寫一個 idempotent migration `ALTER PUBLICATION supabase_realtime ADD TABLE ...`，走 `apply_migration`，apply 後查 `pg_publication_tables` 驗證。
- 風險：低（純加廣播、不改 schema）。注意：補完別忘 RLS 仍守 row 可見度（廣播只控推送、不繞 RLS）。

**P0-3 訂單成員編輯補撕小抄（V1/V2 半套、William 在意、已壞過）**
- 問題：`useMemberEditDialog.ts:148` 直接 supabase.update + setMembers、`useOrderMemberActions.ts:117` insert 後沒 invalidate、`:127` recalculate `skipInvalidate:true` 軟阻廣播。
- 修法：**保留辨識/驗證邏輯不動**，三處寫入收尾補 `await invalidateOrderMembers()`（並連動 invalidateOrders/invalidateTours）；檢討 skipInvalidate 設計。
- file:line：上述三處（高信心，subagent 親驗 95%，動手前再走 fix-safe）
- 風險：中（牽涉 recalculate 連動，需驗團/訂單金額同步）。守門：補 P0-T3 測試。

### P1 — 上線後第一週

**P1-1 訂單列表 useOrdersListView 補雙刷 + realtime**（C1 例外補代價）
- `useOrdersListView` 自訂分頁 key、無廣播；新增/刪除後是否 refreshListView 待親驗。比照請款補 refreshAll + 加 realtime 訂閱。
**P1-2 收款 / 出納列表補 realtime**（V2）：useReceiptsListView / useLinkedPaymentRequests 加廣播訂閱（自己改已靠 refreshAll，缺的是同事改同步）。
**P1-3 會計科目 / 傳票 / 支票頁改 entity hook 或補刷**（V1）：accounts/vouchers 頁從 D 型改 A 型；支票刪除/作廢補 invalidate。
**P1-4 字典節能**（V3）：會計科目加快取（別每開重撈）；人員下拉改 server filter（別 `all:true` 全撈）；銀行快取縮短 + 補廣播。

### P2 — 系統遷移（收斂存量、達成完整 SSOT）

**P2-1** 全面盤 127 個凍結違規（110 supabase-writes + 17 useswr，59 檔），按模組遷進分類表、清 baseline（`lint:swr-prune` 自動撤封條）。
**P2-2** roles/branches/workspaces REST API 評估是否該有 realtime（多人改頻率低，可標 N/A）。
**P2-3** notes 統一（移除 localStorage 雙來源、走 entity hook）。

## 8. 風險與回滾

- 動 RLS / publication：apply 前後查 `pg_publication_tables` 驗證；廣播只控推送、不動 RLS row 可見度（資安不受影響）。
- 動既有 component（成員編輯/會計頁）：**每項先 page-scout + fix-safe**、逐項拍板、不批次硬改。
- migration 破壞性低（純加廣播 / 加 invalidate 呼叫），但仍附 reverse SQL（Migration SOP）。
- 上線夜只動 P0 低風險項，避免製造新 regression。
