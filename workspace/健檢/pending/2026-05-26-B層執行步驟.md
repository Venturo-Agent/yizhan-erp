## B1 — 補強 entity hook filter API（解最大一群裸 useSWR 的根因）⭐ 最高槓桿

**問題**：`createEntityHook` 的 `usePaginated`/`useList` filter 只支援 `.eq()`（array→`.in()`）。因此 4 個列表 hook 自己刻裸 useSWR：
- `orders/_hooks/useOrdersListView.ts:89`、`finance/payments/_hooks/useReceiptsListView.ts:109`、`finance/requests/_hooks/useRequestsListView.ts:119`、`finance/treasury/_disbursement/_hooks/useLinkedPaymentRequests.ts:51`（需 OR / IS NULL / IS NOT NULL 複合條件）
- `tours/_hooks/useToursPaginated.ts:82`（需 keepPreviousData + 自訂 realtime）

**步驟**：
1. 在 `createEntityHook.ts` 的 filter 迴圈（`usePaginated` ~L498、`useList` ~L262）加 operator 語法：`{ status: { neq }, deleted_at: { is: null }, departure_date: { gte } }`、以及頂層 `or: 'a.eq.x,b.is.null'`。對映 PostgREST `.neq/.is/.gte/.lte/.like/.or`。
2. 保持向後相容：值是 scalar→`.eq`、array→`.in`（現況不變）、object→新 operator 分支。
3. 逐一把 4 個 ListView + useToursPaginated 改用擴充後的 `usePaginated`。**一次一個**：對照原 `select` 欄位 + filter 條件 → 改 → 手測該頁列表（filter/搜尋/分頁）→ 確認結果一致。
4. 刪掉該裸 useSWR hook、`npm run lint:swr-prune` 讓 `.eslint-suppressions.json` 對應 baseline 條目消失。

**驗證**：每頁列表行為對照遷移前；`lint:swr-prune` 顯示 useswr baseline 減少；`type-check`。
**風險**：中（filter 語意影響列表正確性）。逐一遷移、每個驗。
**William 在場**：B1 步驟 1（擴 filter API）可獨立做 + 單元測；步驟 3 遷移建議白天逐頁驗。

## B4 — 訊息送出改樂觀更新 + polling 後援；AI Hub key 一致 + 去重訂閱

**問題**（盤點 §5）：AI Hub + Channels 送出**非樂觀**（等 round-trip + refetch）；收訊息**全靠 realtime**（code 自承不穩）**無 polling 後援**；AI Hub 過濾 key 與 sidebar 未過濾 key 不一致；`AiConversationsTab` + `AiSidebar` 重複訂閱同一 broadcast channel（一事件 4× mutate）。

**步驟**：
1. **送出樂觀**：Channels 走 entity `create` 內建樂觀；AI Hub `ReplyComposer` 送出時用 `apiMutate` 的 `optimistic` 先 push 進 thread cache、POST 回來對帳（失敗回滾）。
2. **polling 後援**：訊息 thread 的 useSWR 加 `refreshInterval`（15–30s），realtime 斷線時保險。
3. **AI Hub key 一致**：conversation list 統一 SWR key（`AiConversationsTab` 與 `useMessagingConversations`/`AiSidebar` 用同一 key，或寫入後兩個 key 都 invalidate）。
4. **去重訂閱**：broadcast 訂閱集中到一個 hook/context，兩個 component 共用、不各開一條。

**驗證**：送訊息即時顯示、斷線時 polling 補上、切 channel filter 兩邊列表一致；`type-check`。
**風險**：中（高頻互動、樂觀回滾 + scroll 要處理）。
**William 在場**：建議白天，需實際收發訊息驗。
