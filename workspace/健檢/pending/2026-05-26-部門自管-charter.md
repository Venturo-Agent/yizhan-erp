## 1. 目標組織模型（William 2026-05-26 拍板）

| 維度 | 由誰管 | 現況 | 落差 |
|---|---|---|---|
| **品牌** | 漫途（平台 / 租戶管理）| ✅ `TenantBrandSection`（建租戶時設定）| 無 |

| **部門** | **租戶自己管** | ❌ 5/18 拔掉、無 API、無 UI | **本 charter 要補** |

## 2. 現況盤點（為什麼這是功能、不是改字）

- **DB**：departments 表存在（`20260510120000_phase_a1_departments_and_scope_columns`、`20260514140000_departments_add_branch_id` 把部門掛在分公司下）。
- **但 5/18 拔了**：`20260518700000_fix_scope_visible_drop_department_refs` 把部門從 `scope_visible()` 移除 → 部門目前**不參與資料 scope 過濾**。
- **API**：`/api/organization` 只有 `branches`、`brands`，**沒有 `departments`**。
- **UI**：租戶端組織管理只有（唯讀的）分公司，**沒有部門管理**。
- ⚠️ **本 charter 等於反轉 5/18「拔掉部門」的決定**，需 William 確認這是有意為之。

## 3. 關鍵設計岔路（決定工作量的大小）

**部門到底要不要參與「資料 scope」（像分公司那樣過濾單據/員工可見範圍）?**

- **(A) 部門只是「標籤 / 分組」**（員工掛部門、報表分組用，但不影響誰看得到哪些單據）
  - 工作量：小～中。不用碰 `scope_visible` / RLS。只要 departments CRUD API + UI + employee.department_id 關聯。
  - **不反轉 5/18 的 scope 決定**（5/18 拔的就是 scope 參與）。
- **(B) 部門是「第 3 個 scope 維度」**（像分公司一樣，部門 A 的人看不到部門 B 的單據）
  - 工作量：大。要把部門重新接回 `scope_visible()`（反轉 5/18）、RLS、所有業務表的 scope 過濾、員工指派、單據下拉。橫跨 6 層。
  - 風險高：動 RLS / scope = 紅線 A/H 領域，動完要跑 `tests/e2e/login-api.spec.ts` + cross-tenant 滲透測試。

👉 **這題 William 必須先答**，否則無法估工、也決定要不要碰 scope/RLS。

| 層 | 工作 | (A) | (B) 額外 |
