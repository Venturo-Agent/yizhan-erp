## 一、現況一句話

**有「值合不合法」的守門、沒有「轉換合不合法」的守門。**

- DB 有 CHECK constraint 擋「不在清單裡的值」（譬如 tours.status 必為 6 個值之一）
- 但**沒有任何**東西擋「從 closed 倒回 template」「從 template 直接跳 returned」
- 全站 28 處 `updateTour({status})` 直連 DB、想填什麼就什麼
- 原 service 有寫流水線規則、但只是 code 註解、沒人經過

**比喻**：餐廳有「菜單上才能點」的規則（CHECK），但沒有「點完才能上菜、上菜才能結帳」的規則（transition）。客人理論上可以結帳後再點菜。

## 四、現有可參考的 trigger 範本

DB 已有兩個範本可抄：

### 範本 A — 守門用（推薦）
```sql
-- guard_payment_doc_tour_status: 擋 receipts/payment_requests 在 proposal/template 團上開單
CREATE OR REPLACE FUNCTION public.guard_payment_doc_tour_status()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $function$
DECLARE
  v_tour_status text;
BEGIN
  IF NEW.tour_id IS NULL THEN RETURN NEW; END IF;
  SELECT status INTO v_tour_status FROM public.tours WHERE id = NEW.tour_id;
  IF v_tour_status IS NULL THEN RETURN NEW; END IF;
  IF v_tour_status IN ('proposal', 'template') THEN
    RAISE EXCEPTION '提案 / 模板狀態的旅遊團不可開立此單據（tour_id=%, status=%）',
      NEW.tour_id, v_tour_status
      USING ERRCODE = 'check_violation';
  END IF;
  RETURN NEW;
END;
$function$;
```

### Q1 — Tour 6 狀態轉換規則：嚴格還是寬鬆？

**嚴格版（原 service 寫的、單向不可逆）：**
```
template → proposal → upcoming → ongoing → returned → closed
                                              ↓
                                           closed（終點）
```

**邊角 case 該怎處理？**

| Case | 嚴格不允許 | 寬鬆允許 | 推薦 |
|---|---|---|---|
| upcoming → proposal（要重改） | ❌ 阻擋、走「取消團再複製」 | ✅ 允許、業務常見 | 🟡 看實務 |
| ongoing → upcoming（出發後發現要延期） | ❌ 阻擋 | ✅ 允許 | 🟢 阻擋（要走「取消重開」） |
| closed → returned（結案後發現帳沒對） | ❌ 阻擋、要走「強制重開」流程 | ✅ 允許 | 🟢 阻擋（保護財務月結） |
| 任何 → 同狀態 | 允許（no-op） | 允許 | 🟢 允許 |
| upcoming → returned（短團當天回） | ❌ 跳過 ongoing | ✅ 允許 | 🟡 看實務 |
| 取消團（archived=true）後再改狀態 | ❌ archive 鎖死 | ✅ 允許 | 🟢 阻擋 |

2. 短團（譬如機票團、出發即回）— 允許 upcoming → returned 跳過 ongoing？

### Q2 — Order 5 狀態轉換規則：

DB 值：`pending_review / hk / kk / hl / lk`
UI 中文：受理 / 待確認 / 已確認 / 候補 / 候補成功

**業務語意**：
- pending_review = 訂單剛進來、待處理
- hk = 待確認（OP 已受理、等資源確認）
- kk = 已確認（資源 OK、訂單成立）
- hl = 候補（資源滿、排候補）

**可能的轉換**：
```
pending_review → hk          (OP 受理)
hk → kk                       (確認 OK)
hk → hl                       (確認失敗、排候補)
hl → lk                       (候補有位)
lk → kk                       (候補成功 → 客人確認)
任何 → cancelled?             (沒看到 cancelled 狀態、是用 archive？)
```

### Q3 — 規則放哪裡？

**選 1：純 DB trigger（推薦）**
- ✅ 100% 強制、繞不過
- ✅ 攻擊面歸零（連直連 DB 都擋）
- ✅ 跟資安 #1 一致
- ❌ 後端 / 前端錯誤訊息要翻譯
- ❌ 改規則要寫 migration

**選 2：DB trigger + service 雙層**
- ✅ 雙保險
- ❌ 違反紅線 E（同表寫入只一處、不准 trigger + API 雙寫）

**選 3：API middleware（不在 DB）**
- ❌ Service Role / SQL 直連跳過
- ❌ 違反資安 #1 優先順位

**推薦**：選 1 — 純 DB trigger。錯誤訊息走 `@/lib/db-error-translate.ts` 翻譯（已存在的中央 module）。

## 七之二、最終規則矩陣

### Tour 5 階段（+ 取消是封存維度）

```
        模板 ──複製──> 提案 ──開團──> 即將出發 ─自動─> 旅行中 ─自動─> 未結案 ─結案按鈕─> 已結案
                       │              │                                              │
                       └──取消──┐    └──取消──┐                                     │
                                ↓              ↓                                     │
                              封存（archive=true、狀態不動）                          │
                                                                                     │
                                              ┌──主管強制重開按鈕（要 capability + log）
                                              ↓
                                            未結案
```

**轉換矩陣（從 → 到、✅ = 允許、❌ = 擋）**

| 從 \ 到 | 模板 | 提案 | 即將出發 | 旅行中 | 未結案 | 已結案 |
|---|---|---|---|---|---|---|
| 模板 | ✅ 同 | ✅ 複製 | ❌ | ❌ | ❌ | ❌ |
| 提案 | ❌ | ✅ 同 | ✅ 開團 | ❌ | ❌ | ❌ |
| 即將出發 | ❌ | ❌ | ✅ 同 | ✅ 出發 | ❌ | ❌ |
| 旅行中 | ❌ | ❌ | ❌ | ✅ 同 | ✅ 回程 | ❌ |
| 未結案 | ❌ | ❌ | ❌ | ❌ | ✅ 同 | ✅ 結案 |
| 已結案 | ❌ | ❌ | ❌ | ❌ | ⚠️ 主管 | ✅ 同 |

**封存（archive）正交**：任何階段都可 archive、不影響狀態欄

### Order 6 階段（含已取消）

```
待處理 ──> 待確認 ──> 已確認（終點、撤單走業務流程）
            │  │
            │  └──> 候補 ──> 候補成功 ──> 已確認
            │       ↑          ↓
            │       └──────────┘ (業務員可退候補)
            │
            ↓
          已取消（任何階段都可進、終點）
```

**轉換矩陣**

| 從 \ 到 | 待處理 | 待確認 | 已確認 | 候補 | 候補成功 | 已取消 |
|---|---|---|---|---|---|---|
| 待處理 | ✅ 同 | ✅ 受理 | ❌ | ❌ | ❌ | ✅ |
| 待確認 | ❌ | ✅ 同 | ✅ 確認 | ✅ 滿排候補 | ❌ | ✅ |
| 已確認 | ❌ | ❌ | ✅ 同 | ❌ | ❌ | ✅ |
| 候補 | ❌ | ❌ | ✅ 直接確認 | ✅ 同 | ✅ 候補到 | ✅ |
| 候補成功 | ❌ | ❌ | ✅ 客人確認 | ✅ 退候補 | ✅ 同 | ✅ |
| 已取消 | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ 同 |

### 主管「強制重開」按鈕

- **新 capability**：`tour.reopen_closed`（屬於主管 / 財務主管）

  - 檢查 capability
  - UPDATE tours SET status='returned'
  - INSERT tour_status_logs (from='closed', to='returned', changed_by, reason, is_force_reopen=true)
- **trigger 例外**：guard trigger 看到 is_force_reopen=true 跳過檢查
