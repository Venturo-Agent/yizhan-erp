# yizhan-erp 架構地圖

> 這份文件是 yizhan-erp 整個程式碼倉庫的「地圖」。目的是讓任何人（包括非工程背景）能看懂：一筆業務資料從使用者點按到存進資料庫，中間會經過哪些「層」、每一層放在哪個資料夾、誰是「唯一真相來源（SSOT）」。
>
> 本文件只講「架構長相」，不列待整理的問題（那些在另一份 backlog）。

---

## 一、整體分層圖

系統照「資料的流向」分成六層。從使用者點畫面，到資料寫進資料庫，會由上往下穿過這些層；資料讀回來則反方向。

```
                          使用者（瀏覽器）
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                        │
        ▼                       ▼                        ▼
┌─────────────────┐   ┌──────────────────┐    ┌────────────────────┐
│  路由 / 頁面      │   │  components 元件庫 │    │  stores 狀態        │
│  src/app/(main)  │◄──┤  src/components   │    │  src/stores         │
│  69 個 page.tsx   │   │  (ui 低階積木 +    │    │  (登入 session、    │
│  一個 module 一夾 │   │   editor/dialog/   │    │   側欄、主題)        │
│                  │   │   table 等組裝層)  │    │  Zustand（舊架構）   │
└────────┬─────────┘   └──────────────────┘    └─────────┬──────────┘
         │  (1) 頁面呼叫讀取 hook                          │ 共用登入狀態
         │  (2) 表單送出 → 打 API                          │
         ▼                                                 ▼
┌──────────────────────────────┐         ┌──────────────────────────────┐
│  讀取層 data                  │         │  modules 模組/權限 SOURCE      │
│  src/data                    │         │  src/modules/*.ts             │
│  createEntityHook 工廠         │         │  → codegen 產出 →             │
│  各表一份 useXxx() 讀取 hook   │         │  src/lib/permissions/*        │
└───────────────┬──────────────┘         │  (誰能進哪個 module/路由)      │
                │ 讀資料直接打 DB           └───────────────┬──────────────┘
                │                                          │權限資料供守門用
                ▼                                          ▼
┌─────────────────────────────────────────────────────────────────────┐
│  中央 SSOT / 防呆底座  src/lib                                          │
│  ─────────────────────────────────────────────────────────────────── │
│  auth/        登入檢查 + 權限守門 (requireCapability …)                 │
│  permissions/ @generated 權限常數 (來源是 src/modules)                  │
│  supabase/    連線工廠 (client / server / admin)                       │
│  swr/         快取 + 失效 + realtime (scoped-mutate / api-mutate)       │
│  codes.ts     單號產生器 (11 種編號)                                    │
│  db-error-translate.ts  DB 錯誤翻成中文                                │
│  types/core/constants/i18n  型別、常數、文案 (src/types, src/core …)    │
└───────────────────────────────────┬───────────────────────────────────┘
                                     │
                ┌────────────────────┴────────────────────┐
                │  API 伺服端進入點  src/app/api            │
                │  161 個 route.ts（薄殼）                  │
                │  解析請求 → 檢查登入 → 檢查權限           │
                │  → 寫 DB → 翻譯錯誤 → 記 audit            │
                └────────────────────┬────────────────────┘
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│  資料庫  supabase                                                      │
│  Postgres + RLS（資料列層級安全）                                       │
│  schema 真相 = supabase/migrations/ 864 個 .sql 依時間順序疊出來        │
└─────────────────────────────────────────────────────────────────────┘
```

**一句話讀法：**
- **路由（app）** 是門牌，使用者看到的頁面。
- **API（app/api）** 是後台的收發室，所有寫入都要先過這裡的守門。
- **讀取層（data）** 是「讀資料的統一窗口」，頁面要看資料就跟它要。
- **中央 SSOT（lib）** 是公用工具箱：登入、權限、連線、快取、單號、錯誤翻譯都集中在這。
- **DB（supabase）** 是真正存資料的地方，連 schema 都用一串遷移檔記錄歷史。
- **components / stores / modules** 不在主資料流的「上下游」，而是橫向支援：components 提供畫面積木、stores 保管登入狀態、modules 定義「誰能看到什麼」。

## 二、各子系統

### 1. 路由 / 頁面（`src/app/(main)`）

**職責**：Next.js 的「已登入區」。所有業務頁面都放在 `(main)` 這個群組底下，外面包一層共用框架（側欄＋頂列）和一道權限守門（ModuleGuard）。一個業務模組一個資料夾，`page.tsx` 就是那一頁的門牌；該頁專用的小元件、hooks 放在同資料夾的底線前綴子夾（`_components`、`_hooks`）裡。

**目錄分層**：
- `(main)/layout.tsx`：整個已登入區的入口，把所有頁面包進框架並掛上守門。
- 22 個頂層模組資料夾：dashboard、tours、orders、finance、accounting、hr、library、ai 等。
- 巢狀子路由：例如 `accounting/` 底下還有 reports、vouchers、checks 等子頁。
- 共 69 個 `page.tsx`。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/app/(main)/layout.tsx` | 已登入區唯一入口，掛載框架與守門 |
| `src/components/layout/main-layout.tsx` | 真正的畫面外殼（側欄＋頂列＋內容區） |
| `src/components/layout/sidebar-config.ts` | 側欄顯示的唯一設定（哪些模組進側欄、順序、icon） |
| `src/components/guards/ModuleGuard.tsx` | 路由守門：決定哪個模組能進，否則導到 no-access |
| `src/modules/_registry.ts` | 模組清單來源，側欄與路由都從這裡對齊 |

**依賴**：components/layout 框架、ModuleGuard 守門、modules/_registry 模組清單、lib/permissions 權限、lib/auth、data 讀取層、stores/auth-store 登入狀態。

---

### 2. API 伺服端進入點（`src/app/api`）

**職責**：後台 HTTP 收發室，161 個 `route.ts`。每個都是「薄殼」：解析請求 → 確認已登入 → 檢查權限 → 從登入狀態取得所屬公司（workspace）→ 寫資料庫（再被資料庫的 RLS 二次把關）→ 把錯誤翻成中文 → 記錄操作軌跡（audit）。涵蓋三類：後台 CRUD/業務、對外公開面（付款、webhook）、整合/AI/OCR。

**目錄分層**：
- 161 個 `route.ts`，散在 57 個資源資料夾。
- `src/app/api/lib/`：API 區的局部共用工具（權限查詢、員工 id 解析）。
- 上游真正的 SSOT 在 `src/lib`（auth、supabase、錯誤翻譯、audit）。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/lib/auth/require-capability.ts` | 最常用的權限守門（106 個 route 用它） |
| `src/lib/auth/get-api-context.ts` | 另一套一次完成登入＋權限的守門 helper |
| `src/app/api/lib/check-capability.ts` | 實際查員工角色與權限的底層 |
| `src/app/api/lib/resolve-employee.ts` | 把登入帳號換成員工 id（填寫操作人欄位用） |
| `src/lib/db-error-translate.ts` | 把資料庫錯誤翻成中文業務訊息 |
| `src/lib/audit/audit-helper.ts` | 記錄「誰做了什麼」 |

**依賴**：lib/auth 守門、lib/supabase 連線工廠、lib/db-error-translate、lib/audit、lib/permissions、Postgres + RLS。

---

### 3. 中央 SSOT / 防呆底座（`src/lib`）

**職責**：放給「全系統共用」的中央工具與技術正確性守門：單號產生、DB 錯誤翻譯、登入/權限守門、Supabase 連線工廠、快取與即時更新。它不是某個業務功能，而是「每張表都該借用、不准各頁自己亂寫」的公用底座。

**目錄分層**：
- 頂層單檔：`codes.ts`（單號）、`db-error-translate.ts`（錯誤翻譯）。
- `auth/`：登入檢查與權限守門。
- `permissions/`：自動產出的權限常數 + 執行期判斷。
- `supabase/`：連線工廠（browser / server / admin 三種）。
- `swr/`：快取、寫入後失效、即時更新。
- `finance/`、`status/` 等業務計算與狀態顯示。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/lib/codes.ts` | 單號產生唯一入口（11 種編號，各對應一支防搶號 RPC） |
| `src/lib/db-error-translate.ts` | DB 錯誤碼翻中文的唯一來源 |
| `src/lib/swr/scoped-mutate.ts` | 快取刷新的唯一來源 |
| `src/lib/swr/config.ts` | 每個使用者各自的快取空間設定 |
| `src/lib/swr/api-mutate.ts` | 寫入＋失效快取的統一入口 |
| `src/lib/supabase/admin.ts` | 高權限連線（每個請求新建一份） |
| `src/lib/auth/require-capability.ts` | API 主力權限守門 |
| `src/lib/auth/useLayoutContext.ts` | 前端權限的共享來源（一次抓回登入者能做什麼） |
| `src/lib/permissions/capabilities.ts` | 自動產出的權限常數清單 |

**依賴**：data/createEntityHook（與它共用連線與快取）、stores（登入狀態）、modules（權限常數的產生來源）、Supabase RPC、swr。

---

### 4. 讀取層（`src/data`）

**職責**：前端「讀資料的統一窗口」。核心是 `createEntityHook` 工廠，把快取、離線備援、即時更新、樂觀更新、公司隔離全包成一個統一的 hook 介面。每張資料表用工廠各生一份 `useXxx()`，頁面要看資料就叫它。這是「實體讀取 SSOT」的目標位置。

**目錄分層**：
- `src/data/index.ts`：對外總出口。
- `src/data/core/`（6 檔）：引擎本體（工廠、型別、快取、即時、CRUD、登記表）。
- `src/data/entities/`：53 個各表的讀取 hook。
- `src/data/hooks/`：6 個手寫 hook。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/data/core/createEntityHook.ts` | 讀取層工廠本體（list/detail/分頁/字典全在這） |
| `src/data/core/entityHookCrud.ts` | 寫入＋寫完自動刷新快取 |
| `src/data/core/entityHookRegistry.ts` | 記住每張表有哪些快取 key，刷新時逐個處理 |
| `src/data/core/entityHookCache.ts` | 公司隔離名單＋從本機讀登入資訊 |
| `src/data/core/entityHookRealtime.ts` | 訂閱資料庫變動，別人改了就自動刷新 |
| `src/data/entities/index.ts` | 53 個讀取 hook 的對外出口 |

**依賴**：lib/supabase/client、stores/auth-store、本機快取（IndexedDB）、lib/swr/scoped-mutate、lib/auth/enforce-workspace-scope、swr。

---

### 5. 模組清單 + 權限 SSOT（`src/modules`）

**職責**：全系統「有哪些模組、各模組有哪些功能、誰能用」的單一來源。`src/modules/*.ts` 是手寫的 SOURCE，一個模組一個檔；`_registry.ts` 集中成總清單；再用 codegen 腳本自動產出 `src/lib/permissions` 底下的三份權限產物。權限碼採三段式（`模組.分頁.動作`），1 對 1 對應資料庫的權限資料表。

**目錄分層**：
- `src/modules/`：手寫 SOURCE（扁平 `.ts` 檔），約 24 個 active 模組。
- `src/lib/permissions/`：自動產出的權限常數＋執行期判斷 hooks。
- `scripts/codegen-permissions.ts`：從 modules 反向產出權限產物的引擎。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/modules/_define.ts` | 模組型別根＋宣告函式（這層的核心） |
| `src/modules/_registry.ts` | 模組總清單，側欄與權限稽核都讀它 |
| `scripts/codegen-permissions.ts` | 自動產出三份權限產物的引擎 |
| `src/lib/permissions/capabilities.ts` | 自動產出的權限常數，80+ 個 API 用它守門 |
| `src/lib/permissions/module-tabs.ts` | HR 勾選權限的 UI 來源 |
| `src/lib/permissions/features.ts` | 租戶功能開關＋側欄路由對映 |

**依賴**：codegen 腳本、DB 權限資料表、subscription-plans（建租戶方案）、ModuleGuard 與側欄（消費者）。

---

### 6. 元件庫（`src/components`）

**職責**：展示層，224 個檔。`src/components/ui/` 是低階積木庫（按鈕、輸入框、卡片、表格…），其餘是依功能切分的組裝層（編輯器、對話框、選擇器、列表頁框架等）。設計風格（顏色、邊框等 token）的真正來源是 `src/styles/tokens.css`。

**目錄分層**：
- `src/components/ui/`：約 48 個低階積木＋ 5 個子資料夾（含表格 `enhanced-table/`）。
- 功能組裝層：`editor/`、`canvas-renderer/`、`dialog/`、`selectors/`、`layout/`、`table-cells/`、`guards/` 等。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/styles/tokens.css` | 設計風格 token 的真正來源 |
| `src/components/ui/input.tsx` | 輸入框積木（含全半形、IME、算式計算），95 檔用 |
| `src/components/ui/card.tsx` | 卡片積木 |
| `src/components/ui/enhanced-table/EnhancedTable.tsx` | 表格 SSOT |
| `src/components/dialog/form-dialog.tsx` | 表單彈窗（46 個地方用） |

**依賴**：styles/tokens.css、tailwind 色盤、lib/utils、radix 等底層 UI、ModuleGuard、各 domain 的 data 讀取 hook。

### 7. 狀態 + 通用 hooks（`src/stores`、`src/hooks`）

**職責**：`src/stores` 是 Zustand 狀態層（舊架構，向後相容），主要保管前端的登入 session、UI 偏好（主題、側欄）。`src/hooks` 是一批跨頁共用的通用 React hooks（表單、debounce、breadcrumb、OCR 等）。官方方向是「新功能讀資料改走 `@/data`」，stores 僅保留。

**目錄分層**：
- `src/stores/index.ts`：總出口（明標 legacy）。
- `src/stores/auth-store.ts`：前端登入/權限狀態的真正 SSOT。
- 其他：user-store、accounting-store、calendar-store、theme-store 等。
- `src/hooks/`：19 個通用 hook（功能型 hook 多 colocate 在各頁的 `_hooks/`）。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/stores/auth-store.ts` | 前端 session/權限的真正 SSOT，全 app 取「所屬公司」的入口 |
| `src/stores/index.ts` | stores 總出口（標 legacy） |
| `src/stores/core/create-store.ts` | 舊架構的 zustand 資料讀寫工廠 |
| `src/hooks/index.ts` | 通用 hooks 出口 |

**依賴**：@/data 讀取層、@/lib/swr、Supabase client、@/types、workspace 相關 helper。

---

### 8. 型別 + 核心 + 常數 + 多語（`src/types`、`src/core`、`src/constants`、`src/i18n`）

**職責**：型別與共用基礎設施。`src/types` 放領域型別（`*.types.ts`）與從 Supabase 衍生的型別；`src/core` 放極少量共用 primitive（錯誤、base service、common 型別）；`src/constants` 放 UI 文案與支付列舉；`src/i18n` 走 next-intl，字典在 repo 根的 `messages/`。

**目錄分層**：
- `src/types/`：17 個 `*.types.ts` + `database/`（Supabase 衍生型別）+ `tour/` 子目錄 + barrel。
- `src/core/`：僅 3 檔（common 型別、app-errors、base service）。
- `src/constants/` 與 `src/lib/constants/`：兩處常數。
- `src/i18n/request.ts` + repo 根 `messages/`。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `src/types/index.ts` | 型別總出口 |
| `src/types/database/index.ts` | Supabase 衍生型別層（領域型別的 DB 來源） |
| `src/types/base.types.ts` | 最底層共用型別 |
| `src/core/types/common.ts` | core 共用 primitive |
| `src/i18n/request.ts` | next-intl 設定（字典在根 messages/） |

**依賴**：Supabase（型別來源）、next-intl、被幾乎所有上層以 `@/types`、`@/core`、`@/constants` 消費。

---

### 9. 資料庫遷移 + schema（`supabase`）

**職責**：全部 DB schema 的真相來源。一個扁平的 `supabase/migrations/` 目錄，864 個帶時間戳的 `.sql`，用「本地寫檔 → git commit → 上 production」流程演進 schema、RLS、seed 資料。沒有獨立的 schema 宣告檔，schema 真相靠把整串遷移依時間順序「重播」推導出來。

**目錄分層**：
- `supabase/config.toml`：Supabase CLI 設定。
- `supabase/migrations/`：864 個時間戳 `.sql`（扁平）。
- `supabase/migrations-pending/`：草稿區。
- `supabase/functions/workspace-api/`：唯一 edge function。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `supabase/config.toml` | Supabase CLI 設定 SSOT |
| `supabase/migrations/20260513024000_rls_helper_procedures.sql` | RLS 中央程序定義處（三個 setup_*_rls） |
| `supabase/migrations/`（整串） | schema 真相＝依時間順序重播全部遷移 |

**依賴**：Supabase CLI / PostgREST、src/lib/permissions（seed 須與權限 SSOT 對齊）、src/lib/codes.ts（依賴此層的單號 RPC）、auth.users / employees。

### 10. 工具鏈 / 設定 / 根目錄（`scripts`、`eslint-rules`、`.husky`、根設定）

**職責**：開發期工具鏈與專案配置層（非執行期應用碼）。負責把規範落地成可跑的檢核：`scripts/` 約 40+ 支 audit/codegen/seed/smoke 腳本；`eslint-rules/` 自訂 lint 規則；`.husky/` git hooks；`.github/workflows/` 6 個 CI；根目錄全部 build/test/lint/部署設定。

**目錄分層**：
- 根設定：`package.json`、`eslint.config.mjs`、`.eslint-suppressions.json`、tsconfig、next.config 等。
- `eslint-rules/venturo-design-system.js`：自訂 lint 規則。
- `.husky/`：pre-commit / pre-push。
- `.github/workflows/`：6 個 CI。
- `scripts/`：52 個扁平檔 + 6 個子目錄（audit / codegen / codemod / smoke 等）。

**關鍵 SSOT 檔案**：

| 檔案 | 角色 |
|---|---|
| `package.json` | 所有指令入口（audit / codegen / lint / test） |
| `eslint.config.mjs` | ESLint 設定，掛載自訂規則 |
| `eslint-rules/venturo-design-system.js` | 全部自訂 lint 規則實作 |
| `scripts/audit-rls-blueprint.ts` | 最大的稽核（RLS 六層檢核引擎） |

## 三、跨層資料流：一筆業務資料怎麼穿過各層

以「業務人員在訂單頁新增一筆收款」為例，看一筆資料怎麼往下寫、又怎麼讀回畫面。

### 寫入（使用者按下「儲存」）

1. **使用者點側欄 → 進頁面**
   側欄由 `sidebar-config.ts` 渲染，點進 `(main)/orders/...` 的 `page.tsx`。外層 `layout.tsx` 先過 `ModuleGuard` 守門：確認這個人有權限進這個模組，沒有就導到 no-access。

2. **填表單 → 送出 → 打 API**
   表單用 `components/dialog/form-dialog.tsx` 之類的彈窗。送出時呼叫 `src/app/api/...` 底下的 `route.ts`。

3. **API 守門 → 寫資料庫**
   route 先用 `lib/auth/require-capability.ts` 確認登入＋權限，從登入 session 取得「所屬公司」（workspace_id，不信任前端傳來的），用 `lib/supabase` 連線寫入。需要單號時叫 `lib/codes.ts`。

4. **資料庫二次把關**
   寫到 Postgres 時，RLS（資料列安全）再擋一次，確保只能動自己公司的資料。

5. **錯誤翻譯 + 記軌跡**
   出錯時用 `lib/db-error-translate.ts` 翻成中文回給前端；同時用 `lib/audit/audit-helper.ts` 記下「誰、做了什麼」。

### 讀取（畫面要顯示資料）

6. **頁面呼叫讀取 hook**
   頁面用 `src/data` 的 `useXxx()`（由 `createEntityHook` 工廠生出）。

7. **讀取層撈資料 + 快取**
   工廠內部直接向 Supabase 撈該表資料，套上「公司隔離」，並寫進本機快取（IndexedDB）。下次再進畫面時先給舊快照當暫時值，體感更快。

8. **即時更新**
   `entityHookRealtime` 訂閱資料庫變動。當別人（或剛才那筆寫入）改了同一張表，畫面會自動刷新——這就是為什麼一存完，列表立刻就更新。

9. **快取失效串起寫與讀**
   寫入成功後，`entityHookCrud` 會透過 `lib/swr/scoped-mutate` 通知讀取層「這張表的快取過期了」，讀取層重撈最新資料。這樣「寫」和「讀」雖然分在兩條路徑，卻能保持一致。

### 橫向支援

- **stores/auth-store** 全程提供「登入者是誰、屬於哪家公司」，是上面好幾步取 workspace_id 的來源。
- **modules + lib/permissions** 提供「這個人能不能進這個模組」的判斷依據，被第 1 步的 ModuleGuard 和第 3 步的 API 守門共用。
- **types/constants** 提供整條路徑用到的型別與文案。
- **supabase/migrations** 則是這一切的地基：第 4 步那張表長什麼樣、RLS 怎麼擋，全由這串遷移檔定義。
