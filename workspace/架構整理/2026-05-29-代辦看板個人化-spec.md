# 代辦看板個人化 + 指派任務收件欄 — spec（待 William 拍板）

> 2026-05-29 起草。脈絡：William 發現「代辦看板欄位整個 workspace 共用」。
> 查證：`todo_columns` 自 4/12 建立起就只有 `workspace_id`、從無 user 維度 → 原始設計即共用、非回歸。
> 目標：看板欄位改成「每人自己一組」、再加一個系統「指派給我」收件欄（不可自行新增/刪除）。

---

## 1. 現況

| 物件 | scope | 說明 |
| --- | --- | --- |
| `todos`（卡片） | **個人**：`created_by` / `assignee` / `visibility[]` / `is_public` | 代辦頁已依 user 過濾（isCreator ∥ isAssignee ∥ inVisibility） |
| `todo_columns`（看板欄） | **全 workspace 共用**：只有 `workspace_id` | 任何人改欄位、全公司一起變 |
| `todos.column_id` | → `todo_columns.id`（共用欄） | 一張卡只指一個共用欄 |

`todo_columns` 現有欄位：`id, workspace_id, name, color, sort_order, created_at, updated_at`（**無 `is_system`、無 owner**）。

---

## 2. 目標模型（William 2026-05-29 定案）

- **一般欄位 = 每人自己設計**（per-user、自己建/改/排序；**不自動 seed**、新使用者一開始沒有自訂欄）。
- **唯一一個「任務指派」欄 = 內建虛擬欄**：
  - UI 永遠固定渲染、**不存 DB**、使用者**不能新增/刪除/改名**。
  - 顯示「指派給我」的任務（`assignee = 我` 且為配對的被指派卡）。
- 指派 = 產生配對的兩張卡（§3）；被指派卡固定在對方的「任務指派」虛擬欄、不佔個人欄。

白話：個人看板（自己設計的欄）+ 一個固定的「任務指派」虛擬欄接別人丟來的任務。

> 註：P1 已加的 `todo_columns.is_system` / `system_kind` 因「任務指派」改虛擬欄、用不到（留著無害、不另建 DB 系統欄）。

---

## 3. 核心模型（William 2026-05-29 拍板）★

**指派 = 產生「配對的兩張卡」**（建立者一張、被指派者一張），不是單一來源。定調：

- **雙向同步、雙向操作**：兩張卡用 `linked_group_id` 配對；任一方改**內容**（標題/說明/子任務/備註/期限/優先級）→ 另一方跟著變。雙方都能編輯（不是只有建立者）。
- **完成各自獨立**：一方按「完成」只從**自己**板上消失（進自己完成區）、另一方仍顯示、要他自己按完成才消失 → `completed` / `status` 各卡自己的、**不同步**。
- **欄位各自獨立**：每張卡 `column_id` 是自己 owner 的個人欄（含被指派者的「指派給我」起始欄）；拖卡不影響對方。

→ 用「**兩 row + linked_group_id 配對**」：沿用 `todos.column_id` / `completed`（自然變每卡個人的）、不需 placement join 表；
複雜度集中在「內容欄位的雙向同步」（哪些欄同步、避免無限迴圈）。

| 欄位類別 | 範例 | 同步? |
| --- | --- | --- |
| 內容 | title / description / sub_tasks / notes / deadline / priority | ✅ 雙向 |
| 個人狀態 | completed / status / column_id / sort | ❌ 各自 |
| 指派/歸屬 | created_by（建立者卡）/ owner（被指派卡）/ assignee / linked_group_id | 固定、不變 |

---

## 4. Schema 變更（兩 row + linked 模型）

1. `todo_columns` 加 `owner_employee_id UUID REFERENCES employees(id)`（個人擁有）+ `is_system BOOLEAN DEFAULT false` + `system_kind TEXT NULL`（`'assigned_inbox'`）。
   - UNIQUE(workspace_id, owner_employee_id, system_kind) → 每人最多一個 assigned_inbox。
2. `todos` 加 `linked_group_id UUID NULL`（配對兩張卡；同一指派的建立者卡 + 被指派卡共用同一個值）。
   - `column_id` / `completed` / `status` 維持、但語意變成「每張卡自己的」（因為一張卡只屬一個 owner）。
3. RLS：`todo_columns` 改「先過 workspace_id、再過 `owner_employee_id = 我`」（紅線 H）；todos 既有個人過濾不變。
4. 自動 seed：使用者首次進代辦 → 建預設個人欄 + `assigned_inbox` 系統欄。
5. **內容同步機制**：更新內容欄（title/description/sub_tasks/notes/deadline/priority）時 mirror 到同 `linked_group_id` 的另一張卡。
   - 實作擇一（§9 待定）：API 層同步 vs DB trigger。
   - 防迴圈：只在「內容欄」真的變動時寫、且同步寫入要避免再觸發（trigger 用 `WHEN` 比對 / API 層直接雙寫不回呼）。

---

## 5. 既有資料 backfill

1. 每位 active employee：建個人預設欄 + `assigned_inbox` 欄（既有 workspace 共用欄可複製名稱/顏色、或直接砍掉重建 → §9）。
2. 既有 `todos`：
   - `created_by = assignee`（或無 assignee）→ 純個人卡、把 `column_id` 重映到該 owner 的同名個人欄即可。
   - `created_by ≠ assignee`（已指派）→ 補一張「被指派卡」給 assignee（複製內容、放他的 `assigned_inbox`）、兩張卡填同一個 `linked_group_id`。
3. 驗證：每個指派關係都成對、linked_group_id 正確。
4. 全綠後 drop 舊 workspace 共用欄（破壞性、附 rollback）。

---

## 6. API / UI 變更

- `/api/todo-columns`：GET/POST/PUT/DELETE 改依 **owner = 當前 user** scope；POST/DELETE/改名擋 `is_system`。
- 指派任務（設/改 assignee）：自動建/搬「被指派卡」到對方 `assigned_inbox`、填 `linked_group_id`；取消指派 → 刪對方那張卡（§9 待定）。
- 編輯內容：寫入後同步 linked 卡（§4.5）。
- 完成 / 移動欄位：只動**當前這張卡**（per-card、不同步）。
- 代辦頁：欄位來源改個人欄；`assigned_inbox` 欄特殊渲染（無新增鈕、不可刪、可把卡拖到自己其他欄）。

---

## 7. 風險 / 紅線

- **紅線 H（RLS 過 workspace_id）**：todo_columns 改個人隔離後、RLS 仍須先過 workspace_id 再過 owner，不可只靠 owner。
- **同步迴圈**：內容雙向同步必須防無限觸發（trigger/API 各有防呆寫法）、且只同步「內容欄」、不同步 completed/column_id。
- **資料不漏**：backfill 後每個指派關係都成對、linked_group_id 正確；退役舊共用欄前先比對。
- **個資**：個人看板隔離後、不可因查詢漏掉 owner 條件導致看到別人的欄/卡。
- **孤兒卡**：取消指派 / 刪一方卡時、linked 另一張要一起處理（不可留斷鏈孤兒）。

---

## 8. 分階段計畫（建議）

| 階段 | 內容 | DB? |
| --- | --- | --- |
| P1 schema | todo_columns 加 owner/is_system/system_kind；todos 加 linked_group_id；RLS | 是 |
| P2 seed | 使用者首次進代辦自動 seed 個人欄 + assigned_inbox | 否（code） |
| P3 backfill | 既有共用欄 → 每人個人欄；已指派 todos 補配對卡 + linked_group_id | 是 |
| P4 指派 + 同步 | 指派建配對卡、內容雙向同步、完成/移動 per-card | 否（code）* |
| P5 讀取收斂 | 代辦頁改讀個人欄、assigned_inbox 特殊欄 | 否（code） |
| P6 退役 | drop 舊共用欄（附 rollback） | 是（破壞性） |

\* 內容同步若走 DB trigger 則含 DB。
