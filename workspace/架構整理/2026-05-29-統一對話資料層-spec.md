# 統一對話資料層 — 目標方案 + 遷移計畫（spec，待 William 拍板）

> 2026-05-29 起草。脈絡：B6 釐清 bot/messaging 時發現「表面整合、底層兩套」。
> 對應記憶：讀取層不是真 SSOT、bots-as-employees smell、事件驅動整合模型。
> **目標**：LINE / FB / IG / 未來官網 真的共用一套對話資料、單一 SSOT、加通路 = 加一個 `channel_type`。

---

## 1. 現況（問題）

對話資料**有兩套寫入路徑**，讀取層用 synthetic 合成假裝成一套：

| 通路 | 寫入去哪 | 狀態 |
| --- | --- | --- |
| FB / IG | `inbox_conversations` + `inbox_messages`（統一收件匣） | ✅ 正解 |
| LINE | `line_conversation_messages` + `line_user_profiles`（自己舊一套） | ❌ 沒搬 |

- AI Hub 對話 tab 讀 `/api/messaging/conversations` → read 端把兩套 **synthetic 合成** 一個列表 → 看起來整合了、其實寫入不是 SSOT。
- 額外：LINE bot 被當成**假員工**（`workspace_line_settings.bot_employee_id`、`inbox_messages.sender_employee_id`），但統一模型已有正規 `sender_type='ai_agent'` → 新舊定義打架。

---

## 2. 目標模型（單一 SSOT）

保留統一收件匣、**所有通路只寫這兩張表**：

```
workspaces
 ├─ workspace_line_settings        每租戶 LINE OA 金鑰/設定（不變；FB/IG 各有對應 settings）
 ├─ inbox_conversations            ★唯一對話 thread
 │    channel_type ∈ {line, facebook, instagram, website*}   (*未來加)
 │    external_user_id, display_name, picture_url
 │    customer_id → customers       對話接回 ERP 客戶
 │    bot_paused / unread_count / last_message_*
 │    UNIQUE(workspace_id, channel_type, external_user_id)
 │    └─ inbox_messages            ★唯一訊息表
 │         direction ∈ {inbound, outbound}
 │         sender_type ∈ {contact, agent, ai_agent, system}
 │         sender_employee_id → employees（僅真人 agent 用）
 │         source_id（webhook 去重；UNIQUE(conversation_id, source_id)）
 │         content / message_type / raw_event
 └─ customers / employees
```

**概念**：租戶 → 統一收件匣 → 訊息。`channel_type` 區分通路、`customer_id` 接回客戶、`sender_type` 區分 客戶/真人/AI/系統。

### 官網怎麼接（驗證模型對不對的試金石）
1. `channel_type` CHECK 加 `'website'`
2. 寫一支官網 webhook/接收端 → 呼叫既有 `lib/inbox/inbox-service` 寫 `inbox_*`
3. 完。AI Hub 不用改、自動多一個通路。

## 5. 風險 / 紅線

- **紅線 H**：`inbox_*` RLS 必過 `workspace_id`、不准 `auth.role()`。動表前確認既有 RLS、新欄位不破壞。
- **紅線 E（單一寫入）**：P2 後 LINE 只能有一條寫入路徑，禁 line_conversation_messages 與 inbox_messages 雙寫。
- **去重**：backfill 必須正確填 `source_id`，否則撞 `UNIQUE(conversation_id, source_id)` 或產重複。
- **別斷 related_order_id**：LINE bot 建單的反向連結不能在遷移中遺失（P1 先補欄位）。
- **破壞性 DROP（P5）**：末尾附 rollback、先比對 source/目標筆數、確認 read 端已不依賴舊表。
- **資料現況**：目前僅自家公司一租戶在用、AI 修好前不對外 → backfill/退役風險低，但仍照 SOP。
