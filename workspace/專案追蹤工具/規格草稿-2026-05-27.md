# 專案作戰室（Project War Room）— 規格草稿

> 撰寫者：Logan ｜ 日期：2026-05-27 ｜ 狀態：**草稿、待 William 拍板**
> 本文件為純規劃文件，**未動任何 code、未建表、未跑 migration**。

## 1. 一句話定位

> **「一個公司大專案的作戰室：左手追進度、中間三方（你 / AI / 同事）對話、右手讓 AI 自動幫你復盤現在做到哪、誰卡住、下一步推什麼。」**

這個工具同時是三個東西（三個角色）：

| 角色 | 白話 | 解決的痛點 |
| ---- | ---- | ---------- |
| **(A) 進度追蹤板** | 看板（Kanban）+ 階段一關關推，誰負責哪張卡、做到哪一目了然 | 「現在到底進行到哪一關？」一眼看不出來 |
| **(B) 三方溝通平台** | William、AI 機器人（bot）、真人同事都能在每張卡底下留言、報進度 | LINE 上三方訊息混在一起、洗版、過幾天找不到 |
| **(C) 🌟自動復盤機（靈魂）** | AI 讀完整個專案的所有卡片 + 留言，自動生成「現況復盤報告」 | 每次要跟 AI 報告現況，還得人工把 LINE 訊息整理一遍餵進去 |

> 🌟 **(C) 是這個工具的靈魂、是它贏過 LINE 群組的唯一理由。** 沒有 (C)，這只是一個比較整齊的 LINE。有了 (C)，它變成「會自己整理、會自己提醒、會自己復盤」的作戰指揮所。

## 2. 為什麼是這個、不是甘特圖（需求收斂結論）

William 在多輪對話中**明確否決**了「綁日期、自動往後順延」那一套排程引擎（甘特圖那套）。原因與結論：

- **不綁日曆日期**：不要「這張卡 6/1 開始、6/10 結束」這種日期框。任務什麼時候做、做多久，現實會變，硬綁日期只會一直在改日期。
- **不要自動順延**：甘特圖最核心的「前一個任務晚了，後面全部自動往後推」這個排程引擎，**不要**。這只會製造一堆假精確的日期。
- **不要甘特圖 library**：既然不綁日期、不順延，就**完全不需要**引入任何甘特圖套件（frappe-gantt / SVAR / dhtmlx 等都用不到）。省授權費、省 bundle 體積（對齊 8 維度第 6 條：組建優化）。
- **比甘特單純**：階段就是「一關一關推」，靠的是「這一關該做完的事做完了沒」，不是「日期到了沒」。

**取而代之的時間感**：用「**狀態燈 + 最近活動時間**」取代日期框。不是「這張卡該在 6/10 完成」，而是「這張卡 3 天沒人動了、亮紅燈、該推一下」。這是「**盯動態**」不是「**盯死線**」。

## 3. 核心概念模型：專案 ＞ 階段 ＞ 任務卡 ＞ 活動紀錄

四層階層結構，每一層的白話定義：

| 層 | 名稱 | 白話 | 範例 |
| -- | ---- | ---- | ---- |
| 第 1 層 | **專案（project）** | 一個公司大任務，作戰室本身。第一版 UI 只聚焦「一個」大專案，但資料結構支援多專案。 | 「2026 公司數位轉型」 |
| 第 2 層 | **階段（phase）** | 專案底下的關卡，序列推進、一關關過。前一關沒結束不進下一關（階段鎖）。 | 「第一階段：基礎建設」→「第二階段：上線推廣」 |
| 第 3 層 | **任務卡（task）** | 階段底下的具體要做的事，有負責人、有狀態、有性質（一次性 / 持續）。 | 「架設官網」「業務開發」 |
| 第 4 層 | **活動紀錄（activity）** | 每張卡底下的一條條留言 / 進度更新 / 事件，是復盤的素材。 | 「Logan 6/1 更新：官網首頁完成」「AI 提醒：這卡 3 天沒動」 |

### 階層示意

```
專案：2026 公司數位轉型（project）
│
├─ 階段一：基礎建設（phase, order=1, 進行中）  ←─ 鎖：只看「一次性任務」做完沒
│   │
│   ├─ 任務卡：架設官網（task, type=once, status=完成 ✅）   ← 一次性，做完了
│   ├─ 任務卡：導入 ERP（task, type=once, status=進行中)    ← 一次性，還沒完
│   │   └─ 活動紀錄串：
│   │       ├─ [真人] Logan 5/25：ERP 框架搭好
│   │       ├─ [AI]   bot 5/27：偵測到此卡 2 天沒更新
│   │       └─ [真人] William 5/27：催一下進度
│   └─ 任務卡：業務開發（task, type=ongoing, 狀態燈=正常 🟢） ← 持續型，永遠不卡關
│
└─ 階段二：上線推廣（phase, order=2, 未開始 🔒）  ←─ 鎖住，因為階段一的一次性任務「導入 ERP」還沒完
```

> **關鍵理解**：階段二上鎖，是因為階段一的**一次性任務**（架設官網 ✅、導入 ERP ⏳）還沒全部做完。「業務開發」這種**持續型任務**永遠不會「完成」，所以**不拿來算卡關條件**。詳見 §5.1。

## 4. 關鍵設計點詳述

### 4.1 階段鎖（Phase Lock）— 只看一次性任務、持續型不卡關

**業務白話**：階段一關關推。前一關沒「過關」，下一關就是鎖住的（灰掉、進不去 / 不能設成進行中）。

**「過關」怎麼判定？** —— 這是最容易做錯的地方：

- ✅ **只看該階段的「一次性任務」是否全部完成。**
- ❌ **「持續追蹤型任務」永遠不會完成，所以絕對不能拿來當卡關條件**，否則階段永遠卡死、過不了關。

**判定規則（建議預設）**：

```
階段 N 可推進到「已完成 / 可進下一關」的條件 =
   該階段底下「所有 type='once' 且未軟刪」的任務卡，status 都是「完成」
   （type='ongoing' 的任務卡完全不參與此判定）
```

**邊界情況處理（spec 待 William 確認，§12 列出）**：

- 一個階段底下**全部都是持續型任務、一張一次性任務都沒有** → 建議：此階段視為「可手動推進」（因為沒有任何「該做完的事」，由人決定何時關掉這一關）。
- 推進方式：建議**人工按「結束本階段」按鈕**觸發（系統檢查一次性任務都完成才放行），而不是「最後一張一次性卡完成的瞬間自動跳關」。理由：給人一個「確認收尾」的動作，避免誤觸。

**技術落點**：對齊建表 SOP 的 **L4 狀態守門**（`is_row_editable`）。階段的「可否推進」是一個 DB function（譬如 `can_advance_phase(phase_id)`），前端按鈕 `disabled` 對齊，API 寫入時再驗一次（雙保險）。

### 4.2 任務兩種性質：一次性 vs 持續追蹤

William 親口的關鍵區分：

| | **一次性任務（type='once'）** | **持續追蹤型任務（type='ongoing'）** |
| -- | -- | -- |
| 白話 | 做完就關閉、就沒事了 | 永遠在跑、要一直盯 |
| 範例 | 「架設官網」弄好就結束 | 「業務開發」一直打電話、一直寄 email，不會「完成」 |
| 進度衡量 | **完成度**（status：待辦 / 進行中 / 完成）；可選進度 % | **狀態燈**（正常 🟢 / 需注意 🟡 / 卡住 🔴）+ **最近更新時間** + 可選累積計數 |
| 算不算卡關 | ✅ 算（一次性全完成才能推進階段） | ❌ 不算（永遠不會完成，不可拿來卡關） |
| 「完成」這件事 | 有終點，會被標記為完成 | 沒有終點，只有「最近有沒有在動」 |

**持續型任務的進度衡量法（建議預設，§12 待確認）**：

- 主指標：**狀態燈**（normal / attention / stuck 三段）。
- 輔指標：**最近活動時間**（最後一條活動紀錄的時間，多久沒動）。
- 可選：**累積計數欄位**（譬如「本週聯絡 N 家客戶」），讓持續型任務也有「在動」的量化痕跡。第一版可做成一個自由欄位（`progress_note` 文字 + 可選 `counter` 數字），不要做成複雜的指標系統。
- ❌ **不要求完成度 %**：持續型任務沒有「完成」概念，逼它填 % 是違反業務本質。

### 4.3 活動紀錄串（Activity Log）— 復盤素材、標人 / AI 發話者

**業務白話**：每張任務卡底下，有一條「誰幾時做了什麼」的時間軸。這是整個工具的**記憶**，也是 AI 復盤的素材來源。

**和現有 todos 的差異**（探查確認）：現有 `todos.notes` 是 jsonb 的**附件清單**，不是真正的多人留言串。所以**活動紀錄要新做一張表**，搬 channels 的 `channel_messages` pattern。

**活動紀錄的類型（建議）**：

- `comment`：人或 AI 的留言（最常見）
- `progress_update`：進度更新（譬如把卡從「進行中」推到「完成」，系統自動寫一條）
- `status_change`：狀態 / 階段變更事件（系統自動寫）
- `ai_review`：AI 復盤產出（可選，把復盤結果也存成一條活動，方便回溯）

**標示發話者是真人還是 AI（搬 channels pattern）**：

對齊 `channel_messages` 的做法 —— 用**兩個互斥的欄位**標示發話者身分：

- `author_employee_id`（FK → `employees(id)`）：真人員工發的
- `author_agent_id`（FK → `ai_agents(id)`）：AI 機器人發的
- 兩者其一為 NULL：一條活動要嘛是真人發、要嘛是 AI 發（CHECK constraint 守「至少一個、且通常只一個」）

> 這樣前端就能在每條活動旁畫個小標籤：「👤 Logan」或「🤖 復盤 bot」，一眼分清誰說的。

**不變式（搬 channels I3）**：

- 活動紀錄**一旦發出不可改、不可實刪**（要更正走新增一條更正活動、或軟刪 `deleted_at`）。理由：這是復盤的證據鏈，不准回頭塗改（呼應紅線 D 的精神——已發生的事留痕、要改走補正）。

### 4.4 🌟自動復盤機（靈魂功能）

**William 的痛點（原話精神）**：

> 「LINE 訊息一直洗、過幾天找不到、沒法回頭看進行到哪、每次要跟 AI 報告現況還得人工整理。」

**這工具怎麼解**：因為**每張卡的活動紀錄全部留存在 DB**（不像 LINE 訊息會洗掉、找不到），所以 AI 機器人（bot）可以隨時讀「某專案的所有卡片 + 所有活動紀錄」，自動生成一份**進度復盤報告**。

**AI 讀什麼（輸入）**：

- 該專案的所有階段（哪關完成、哪關進行中、哪關還鎖著）
- 每個階段底下的所有任務卡（負責人、狀態、性質、狀態燈、最近活動時間）
- 每張卡底下的活動紀錄串（誰幾時做了什麼、留言內容）

**AI 生成什麼（輸出復盤報告，建議結構）**：

1. **整體進度**：目前在第幾關、這一關的一次性任務完成幾 / 總幾，整個專案大致到哪。
2. **哪些卡住了**：列出狀態燈紅的卡、超過 X 天沒活動的卡、status 長期停在「進行中」沒動的卡。
3. **誰慢了**：哪些負責人的卡好幾天沒更新（呼應 §4.5）。
4. **下一步該推什麼**：建議優先處理哪幾張卡才能讓這一關過關、進到下一關。
5. **持續型任務的健康度**：哪些持續型任務最近沒在動（狀態燈黃 / 紅）。

**怎麼觸發（分期）**：

- **v1（第一版）：on-demand（要的時候生）。** William / 有權限的員工在復盤面板按一個「生成復盤」按鈕，AI 即時讀資料、生成報告。
- **v2（之後）：定期自動生**（每日 / 每週排程，譬如每週一早上自動產出上週復盤，推到 channels 公告或站內通知）。v2 才接排程（cron）。

**技術落點**：

- AI 讀資料：透過一個**唯讀的彙整 API**（譬如 `GET /api/projects/{id}/digest`），把整個專案的卡片 + 活動紀錄結構化吐出來，餵給 AI（bot 已有的 LLM 管道）。
- 復盤結果：v1 可只回傳給前端顯示；也可選擇把結果存成一條 `ai_review` 類型的活動紀錄（存在專案層的活動，或一個專案層的 `project_reviews` 表，§6 給判斷）。
- 對齊既有 AI bot 基礎建設（探查確認系統已有 `ai_agents` 表 + bot 管道）。

### 4.5 「誰慢了」判定（自動 + 人工並存）

**建議預設（§12 待確認）**：兩者並存。

- **系統自動標**：某張卡超過 **X 天**沒有任何活動紀錄 → 自動亮紅燈（「停滯」）。
  - `X` 可**每張卡個別設定**（譬如重要的卡設 1 天、不急的設 7 天）。
  - **預設 X = 3 天**。
  - 實作：用「最後一條活動紀錄時間」跟「now」比較，前端算 / 或一個 DB view 算（不需要排程，純粹讀的時候計算）。
- **人工手動標**：負責人 / William 可手動把某張卡標成「重點關注」或手動標「停滯」，覆蓋自動判定。

> 兩者並存的好處：自動的抓「明顯沒人理的」，人工的抓「系統看不出來但人知道有問題的」。

## 5. 不要做的事（明確邊界）

- ❌ **不做甘特圖**（任何形式的甘特圖 UI / library）
- ❌ **不綁日曆日期**（不要「開始日 / 結束日」這種硬日期框；deadline 第一版不做，或做成純參考、不參與任何自動邏輯）
- ❌ **不做「時間到自動往後順延」排程引擎**
- ❌ **不做「管 N 個專案的通用專案管理系統」的完整 UI**（資料結構支援多專案，但第一版 UI 聚焦單一大專案）
- ❌ **不做跨 workspace 共享**（這功能做在「角落」租戶內，不動租戶架構）

## 6. 資料結構草案（對齊建表 SOP 與紅線）

> 全部新表都過建表 SOP 的 Phase 0 分類 + L1-L6。下面標出每張表的資料屬性、RLS pattern、審計欄位 FK、軟刪。

### 6.0 複用既有 todos 表，還是新建？——判斷與理由

**探查發現**：現有 `todos` 表已有 `task_type`、`tour_id`、`status`、`assignee`、`column_id`、`visibility` 等欄位（比 todos-spec 文件記載的更豐富），且 realtime、entity hook、Kanban 拖拉 UI 都現成。

**判斷**：

| 層 | 建議 | 理由 |
| -- | ---- | ---- |
| 專案層（project） | **新建 `projects` 表** | todos 沒有「專案」這個上層概念，必須新建 |
| 階段層（phase） | **新建 `project_phases` 表** | todos 沒有「序列推進 + 階段鎖」概念，必須新建 |
| 任務卡層（task） | **新建 `project_tasks` 表**（建議，非複用 todos） | 見下方詳述 |
| 活動紀錄層（activity） | **新建 `project_task_activity` 表** | todos.notes 是 jsonb 附件清單、不是多人留言串，必須新建（搬 channels pattern） |

**任務卡層為何建議「新建」而非「複用 todos」（重要判斷，給 William 拍板）**：

- **複用 todos 的誘惑**：todos 已有看板拖拉、realtime、entity hook，複用省力。
- **複用的代價（為什麼不建議）**：
  1. todos 沒有 `phase_id`（要掛到階段底下）、沒有「一次性 / 持續」這個會影響卡關邏輯的 `task_nature`、沒有「狀態燈」。硬加欄位會讓 todos 表越長越雜（todos 是給「個人 / 團隊待辦」用的，塞專案概念進去會污染它的語意）。
  2. 階段鎖的 L4 狀態守門邏輯如果寫進 `is_row_editable('todos', ...)`，會跟 todos 既有語意打架。
  3. 兩種任務（個人待辦 vs 專案作戰任務）的生命週期、權限、復盤需求不同，混在一張表，未來改一邊容易弄壞另一邊（違反 8 維度第 7 條：不過度耦合）。
- **新建的好處**：`project_tasks` 語意乾淨、欄位專為作戰室設計；**UI 看板組件（Kanban）可直接複用 todos 那套拖拉組件**（複用「UI 程式碼」而非複用「資料表」，這才是對的複用層級）。

> **結論建議：資料表新建 4 張（project / phase / task / activity），但 UI 看板拖拉組件、entity hook 工廠、apiMutate、realtime 機制全部複用既有基礎建設。** 詳見 §7 盤點表。

### 6.1 `projects`（專案層）— 🏢 業務主表

| 欄位 | 型別 | 說明 |
| ---- | ---- | ---- |
| `id` | uuid PK | |
| `workspace_id` | uuid NOT NULL FK→workspaces ON DELETE CASCADE | 租戶隔離（紅線 H）。一旦設不可改（搬 channels I1）。 |
| `code` | text | 專案編號（走 `@/lib/codes`，新增 `generate_project_code` RPC，紅線 L6） |
| `name` | text NOT NULL | 專案名 |
| `description` | text | 專案說明 |
| `status` | text DEFAULT 'active' | active / archived（封存後唯讀，呼應紅線 D：已封存不回頭改） |
| `owner_employee_id` | uuid FK→employees ON DELETE SET NULL | 專案總負責人 |
| `created_by` | uuid FK→**employees(id)** ON DELETE SET NULL | 紅線 B |
| `updated_by` | uuid FK→**employees(id)** ON DELETE SET NULL | 紅線 B |
| `created_at` / `updated_at` | timestamptz | updated_at 走 `set_updated_at` trigger |
| `deleted_at` | timestamptz | 軟刪（8 維度第 4 條） |

- **RLS**：`CALL setup_workspace_scoped_rls('projects')`（USING + WITH CHECK 都過 `workspace_id = get_current_user_workspace()`，紅線 H）
- **Index**：`(workspace_id, created_at DESC)`、`(workspace_id, code) UNIQUE WHERE deleted_at IS NULL`、`(deleted_at) WHERE deleted_at IS NOT NULL`

### 6.2 `project_phases`（階段層）— 🏗 業務子表（inherit project scope）

| 欄位 | 型別 | 說明 |
| ---- | ---- | ---- |
| `id` | uuid PK | |
| `project_id` | uuid NOT NULL FK→projects ON DELETE CASCADE | 走 parent scope |
| `name` | text NOT NULL | 階段名（譬如「基礎建設」） |
| `phase_order` | int NOT NULL | 階段序號，決定推進順序 |
| `status` | text DEFAULT 'pending' | pending（未開始/鎖）/ active（進行中）/ completed（已過關） |
| `started_at` / `completed_at` | timestamptz | 進入 / 結束本階段時間（記錄用，不參與自動順延邏輯） |
| `created_by` / `updated_by` | uuid FK→**employees(id)** ON DELETE SET NULL | 紅線 B |
| `created_at` / `updated_at` / `deleted_at` | timestamptz | 軟刪 |

- **RLS**：`CALL setup_inherited_rls('project_phases', 'projects', 'project_id')`（繼承 project 的 workspace scope）
- **Index**：`(project_id, phase_order)`
- **L4 狀態守門**：階段推進靠 DB function `can_advance_phase(phase_id)` —— 檢查該階段所有 `task_nature='once'` 的 `project_tasks` 都 `status='done'`。`project_tasks` 屬於已 completed 階段時不可改（`is_row_editable`）。
- **不變式**：`(project_id, phase_order)` 唯一；序列推進，前一階段未 completed，後一階段不可設 active。

### 6.3 `project_tasks`（任務卡層）— 🏗 業務子表（inherit project scope）

| 欄位 | 型別 | 說明 |
| ---- | ---- | ---- |
| `id` | uuid PK | |
| `project_id` | uuid NOT NULL FK→projects ON DELETE CASCADE | 走 parent scope |
| `phase_id` | uuid NOT NULL FK→project_phases ON DELETE CASCADE | 掛在哪一階段 |
| `title` | text NOT NULL | 任務名 |
| `description` | text | 說明 |
| `task_nature` | text NOT NULL DEFAULT 'once' | **'once'（一次性）/ 'ongoing'（持續追蹤）** ← 關鍵欄位，決定卡關與進度衡量 |
| `status` | text DEFAULT 'todo' | 一次性：todo / in_progress / done；持續型：用 status_light 為主 |
| `status_light` | text | **持續型專用：normal / attention / stuck（狀態燈）** |
| `progress_percent` | int | 一次性可選進度 %（持續型不用） |
| `progress_note` | text | 持續型的自由進度描述（譬如「本週聯絡 N 家」） |
| `counter` | int | 持續型可選累積計數（搭 progress_note，§4.2） |
| `assignee_employee_id` | uuid FK→employees ON DELETE SET NULL | 負責人（可每卡不同人） |
| `stale_threshold_days` | int DEFAULT 3 | 幾天沒活動算停滯（每卡可設，§4.5，預設 3） |
| `is_flagged` | boolean DEFAULT false | 人工手動標重點 / 停滯（§4.5） |
| `column_id` | uuid FK→todo_columns? | **看板欄位（待確認是否複用 todo_columns，§12）** |
| `last_activity_at` | timestamptz | 最後一條活動時間（給「誰慢了」算，由活動寫入時更新或 view 算） |
| `created_by` / `updated_by` | uuid FK→**employees(id)** ON DELETE SET NULL | 紅線 B |
| `created_at` / `updated_at` / `deleted_at` | timestamptz | 軟刪 |

- **RLS**：`CALL setup_inherited_rls('project_tasks', 'projects', 'project_id')`
- **Index**：`(phase_id, status)`、`(project_id, task_nature)`、`(assignee_employee_id)`、`(deleted_at) WHERE ...`
- **L4 狀態守門**：屬於已 completed 階段的卡不可改（除非走補正流程，呼應紅線 D）。
- **CHECK**：`task_nature IN ('once','ongoing')`；`status_light IN ('normal','attention','stuck')`

### 6.4 `project_task_activity`（活動紀錄層）— 🏗 業務子表（搬 channel_messages）

| 欄位 | 型別 | 說明 |
| ---- | ---- | ---- |
| `id` | uuid PK | |
| `project_id` | uuid NOT NULL FK→projects ON DELETE CASCADE | 走 parent scope（也方便 AI 一次撈整個專案的活動） |
| `task_id` | uuid FK→project_tasks ON DELETE CASCADE | 掛在哪張卡（可為 NULL = 專案層活動，譬如 AI 復盤產出） |
| `activity_type` | text NOT NULL DEFAULT 'comment' | comment / progress_update / status_change / ai_review |
| `content` | text | 留言內容 / 事件描述 / 復盤報告內文 |
| `author_employee_id` | uuid FK→**employees(id)** ON DELETE SET NULL | **真人發話者**（搬 channels sender_employee_id） |
| `author_agent_id` | uuid FK→ai_agents ON DELETE SET NULL | **AI 機器人發話者**（搬 channels sender_agent_id） |
| `metadata` | jsonb | 結構化附帶資料（譬如 status_change 的 from/to、復盤的統計數字） |
| `created_at` | timestamptz | 活動時間（不變式：不可改） |
| `deleted_at` | timestamptz | 軟刪（更正走新增，不實刪，呼應紅線 D） |

- **RLS**：`CALL setup_inherited_rls('project_task_activity', 'projects', 'project_id')`
- **Index**：`(task_id, created_at DESC)`、`(project_id, created_at DESC)`（AI 復盤一次撈整專案）
- **CHECK**：`author_employee_id IS NOT NULL OR author_agent_id IS NOT NULL`（至少一個發話者）
- **不變式**：一旦寫入 `content` 不可改（搬 channels I3）；要更正走新增更正活動或軟刪。
- **Realtime**：加入 `supabase_realtime` publication（搬 todos / channel_messages 做法），entity hook 走內建 `useRealtimeSync()`，留言即時刷新。
- **寫入更新 `project_tasks.last_activity_at`**：⚠️ 對齊**紅線 E**（同表寫入 SSOT）——`last_activity_at` 的更新**只放一個地方**。建議走 **DB trigger**（純技術 housekeeping，不可能漏），並把這對「activity INSERT → task UPDATE」加進 `audit:writes` 的 ALLOWLIST 並註明理由；**不要**同時在 API code 也手動 update（會雙寫撞車）。

### 6.5 （可選）`project_reviews`（復盤報告層）— 待 §12 判斷

AI 復盤產出可以：
- (a) 存成 `project_task_activity` 的一條 `ai_review`（task_id = NULL，掛專案層）—— 較輕量，建議 v1 用這個。
- (b) 獨立 `project_reviews` 表（含產出時間、產出 agent、報告全文、統計快照）—— v2 定期復盤需要查歷史時再建。

> **建議 v1 用 (a)、v2 視需要升級到 (b)。** 寫進 §12 給 William 拍板。

## 11. 分期計畫

### Phase 1 — 最小可跑核心（MVP，能展示）

- L3 三維組織篩選（按部門看專案）。

- 跟既有 todos 互通（把專案任務同步到個人待辦）。

## 13. 風險與注意（救護車式排序：先講會死人的）

### 🔴 會死人的（資安 / 資料外洩）
1. **跨租戶活動紀錄洩漏**：活動紀錄是子表，RLS 必須**確實繼承 project 的 workspace scope**（`setup_inherited_rls`）。channels 在 SEC-005 踩過「重建 policy 忘了補 SELECT policy 導致全擋 / 或 workspace 隔離不夠」的雷——**必須 defense-in-depth：子表 RLS 一定要能追到 workspace_id，且 INSERT WITH CHECK 也過 workspace**（紅線 H）。上線前用 service_role 模擬不同 workspace user 驗證讀不到別人的。
2. **AI 復盤彙整 API 越權讀取**：`/api/projects/{id}/digest` 一次撈整個專案所有卡 + 活動，**必須 `requireCapability('projects.read')` + RLS 雙保險**，且 workspace_id 從 session 取（`getCurrentWorkspaceId()`），**不准信 client 傳的 workspace_id / project_id**（紅線 H）。否則 AI 管道變成資料外洩破口。

### 🟡 會弄壞功能的
3. **階段鎖把持續型任務算進去 → 階段永遠卡死**：這是最容易做錯的點。`can_advance_phase` 必須**只數 `task_nature='once'`**，寫單元測試守住。
4. **`last_activity_at` 雙寫撞車**：activity INSERT 後更新 task 的 `last_activity_at`，**只能放一個地方**（建議 trigger）。若 trigger + API 各做一份會 race condition（紅線 E）。動前跑 `npm run audit:writes`。
5. **審計欄位 FK 設錯**：`created_by` 等若誤指 `auth.users(id)` 而非 `employees(id)`，insert 直接炸（紅線 B）。前端寫 `created_by: currentUser?.id || undefined`，**不是 `|| ''`**。
6. **封存 / 已過關階段開後門**：不准做 `reopenPhase` / `unlockProject` 這種解鎖 API（紅線 D）。要改已封存的走補正（新增活動 / 新階段），不回頭塗改。

### 🟢 體質 / 維護性
7. **不要把 project_tasks 塞進 todos**（§6.0），避免兩種任務語意糾纏，未來改一邊弄壞另一邊。
8. **不要引入甘特圖 library**（bundle 體積、授權費，且用不到）。
9. **UI 不准 Tailwind 預設色**，全走 morandi / status token（UI 紀律紅線），commit 前自己 audit。
10. **5 SSOT 漏 module-tabs.ts**（5/12 channels 踩過）：commit 前 grep 自查 5 個 SSOT 都動了。
