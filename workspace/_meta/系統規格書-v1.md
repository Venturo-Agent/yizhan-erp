## 1. 願景

**多租戶、多產業的 SaaS ERP。** 從旅行業起家、地基設計成「換產業只換上層、地基不動」。
- 第一塊多產業地基已落地：`workspaces.industry / sub_industry`（tourism / general…）。
- 終局：同一套地基地基服務各行業、各租戶資料完全隔離。

## 2. 核心原則（= 100 分的定義）

| 順位 | 原則 | 白話 |
|---|---|---|
| #1 | **資安** | 洞 = 客戶資料外洩 = 商業終結。RLS 守在資料庫最底層、是地基保證。|
| #2 | **效能（不浪費）** | 不隨便浪費、**不全撈所有資料**、讀取走 server 端過濾 + 快取。|
| #3 | **SSOT** | 單一真相來源。讀/寫/同步/身分 各只有一條路。|
| — | **完整性 / 統一性** | 不留補丁、不留散刻、不留 dead code。新功能天生繼承全部。|
| — | **強迫症乾淨** | 投資人（工程師）翻開 code 會想投資的乾淨度。|

> 「完整 vs 保守」一律選完整（沒有 production 包袱時不為避險而最小化）。
> 唯一不打折：**正確性**（type-check / 守門 / 邏輯對 = 每筆 100 分）。

## 4. 資料地基（體制級、競爭核心）

**一個原則貫通：DB 是唯一真相 → Realtime 主推 → 單一身分認證 → 本地只是鏡像 → 寫入樂觀。**

- **身分**：一次登入 → JWT → 同一份身分貫通 API-RLS / realtime / client（auth-sync SSOT）。
- **讀**：entity hook（`createEntityHook`）。**server 端 filter、不全撈**。
- **寫**：apiMutate（寫入 + 失效 SSOT）。**寫了即現、刪了即消、不用 F5**。
- **同步**：Supabase Realtime 主推（entity hook 內建 useRealtimeSync）、跨使用者即時、訂閱失敗可觀測。
- **快取**：SWR 鏡像 DB、realtime 沒到時 mutate-revalidate fallback。

**升級路線（William 拍板 🅐）**：黏合升級、Realtime 當主軸、**不導入重型同步引擎**（PowerSync/ElectricSQL）。理由：辦公室 ERP 不需離線優先/協作編輯/差量同步那些重型強項；Supabase 原生 + 統一地基 = 輕、快、好維護、撞牆能逃。重型引擎留到「離線優先變核心」才評估。

**地基健康度（落地中）**：
- ✅ realtime auth race / channel 撞號 / 訂閱觀測（5/20 修）
- ✅ 4 張漏網表進 publication（5/24 止血）
- ⏳ ~110 處散刻寫入 → 收進 apiMutate（完整性最大一塊）
- ⏳ finance 表單不全撈（單筆 filter / 批次跨團 分流）
- ⏳ publication 完整性 CI 守門

## 5. 權限模型（純角色 capability、零旗標）

**能力決定一切、零旗標、零特權。**（紅線 #0：無超級管理員）
- 員工 → 一個職務(role) → 職務的能力(role_capabilities)。**唯一真相。**
- 指派候選 = 「有對應能力的人」（不是員工身上的固定標籤）：
  - 業務 = 能新增/編輯訂單（orders.create/edit.write）
  - 團控 = 能寫團員名單（tours.members.write）
  - 代墊人 = 有可代墊款能力（finance.advance_payment.write）
  - 模板 = 能建立模板（tours.template.write）
  - 助理 = **取消、不存在**
- 「沒人有某能力 → 對應按鈕/欄位消失」(hide-if-none)。
- 跨大小公司：差別只在「給哪個角色哪些能力」、系統不框死職能。
- ✅ 已落地（2026-05-24、`2026-05-24-員工職能-capability簡化方向.md`）：eligibility 旗標系統整套移除。

## 6. 資安契約（紅線摘要、完整見 CLAUDE.md）

- 每張業務表 RLS 過 `workspace_id = get_current_user_workspace()`（紅線 H）
- 無超級管理員、唯一訪問路徑 = workspace_features + role_capabilities + RLS（紅線 #0）
- 審計欄位 FK → employees(id)（紅線 B）；admin client per-request（紅線 C）
- 月結/結團封存不可回頭改、走沖正（紅線 D）
- per-user SWR cache key 防跨帳號污染（紅線 G）
- DB trigger × API 同表寫入不雙寫（紅線 E）

**達標定義升級（2026-05-25 攻擊式稽核拍板、「牆立起來」不夠、要「牆夠厚」）**：
- **RLS 強度**：不只「有 RLS」、policy 不准寫死 `true` / `auth.role()='authenticated'` 當隔離（那只擋未登入、不擋跨租戶）。INSERT `WITH CHECK` 必過 workspace。（攻擊式稽核抓到 11 條寬鬆 policy、已修）
- **客戶 PII 檔案**（護照/證件）儲存桶**不可 public**、一律走 signed URL 短效授權。（passport-images 曾 public=true、護照可無登入下載、已止血）
- **無 SECURITY DEFINER view 繞 RLS**（用建立者權限跑會穿透查詢者隔離）。
- **API 永不信任 client 傳的 workspace_id**、一律從 session 取。（/api/integrations/usage 曾信任、已修）
- **內建顧問接進常駐健檢**：Supabase advisor 抓得到上述（rls_policy_always_true / security_definer / 公開桶）、自製 audit:rls 抓不到 → 兩者並用。

## 7. 效能契約

- 列表預設 20、分頁固定 15、不給「每頁筆數」選擇器
- **不全撈再前端 filter**（egress 殺手）→ entity hook server 端 filter；報表「只撈選取範圍」+ 自動翻頁防 1000 筆截斷
- **全站不准 stale-read（達標線、2026-05-25 拍板）**：寫入後必失效「所有讀同份資料的快取」（entity / list-view / report / store 四種對齊）、操作完即現、不用 F5。根因型 bug、解法＝中央失效機制（invalidateFinanceAll→推廣）+ 補完未接統一資料層的模組（會計最大缺口）。
- **外鍵全索引**：FK 無覆蓋索引 = 關聯查詢/cascade 全表掃描、規模化線性慢（已補 198 個、缺索引 0）。
- **持續性槓桿 > 一次性**：後端優化（不全撈/索引/RLS 每列重算）= 每次查詢都省、隨資料量放大；前端 bundle = 一次性下載後快取。優先打持續性的。
- SWR `revalidateOnFocus: false`、快取一次撈到處重用、寫入才失效
- 大 library 動態 import、route 預設 server component
- 防連點：寫入按鈕 `disabled={loading}`

## 8. 五句競爭優勢（對投資人 pitch）

1. **概念**：資料庫是唯一真相、即時推給所有人、本地只是鏡像、一次登入身分貫通全程。
2. **抽象層**：讀/寫/同步/身分 四件事統一在一塊地基、不散刻。
3. **優化了什麼**：寫了即現、刪了即消、不用 F5、跨人同步、不全撈不浪費、資安守在最底層。
4. **跟市面差別**：別人要嘛手動刷新（爛 UX）要嘛上昂貴複雜同步引擎；我們原生 realtime + 統一地基 = 輕快好維護。
5. **優勢**：多租戶資安是地基保證、效能是天生性質、新功能一接就繼承全部 → 開發快、bug 少、好維護、好擴張到各產業。

---

## 9. 多產業擴展邏輯

- 地基（身分/資料/同步/資安/效能）**與產業無關** → 換產業不動地基。
- 產業差異吃 `workspaces.industry/sub_industry` + `workspace_features`（買哪些功能）+ role_capabilities（誰能用）。
- 旅行業特化（團/訂單/團控）是「上層模組」、可被其他產業模組替換、地基不變。
