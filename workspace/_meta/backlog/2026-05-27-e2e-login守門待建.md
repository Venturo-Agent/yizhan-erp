# Backlog：E2E 登入守門待建（2026-05-27 停用、未來 SaaS 化再建）

> **狀態**：CI 的 `e2e-smoke` job 2026-05-27 William 拍板**停用**（改 `if: ${{ vars.E2E_SMOKE_ENABLED == 'true' }}`、預設 skip、定義保留不刪）。本卡記錄它是什麼、為什麼停、未來怎麼建。

## 它是什麼（白話）

「上線前的自動試登入員」。每次推 code 上線前，自動跑兩條命脈路：①員工登入 ②開新租戶，確認改完沒把登入/建租戶弄壞。壞了舉紅旗擋上線。

對應 spec：`tests/e2e/login-api.spec.ts`（守 4 個 workspace 被 admin client 找到、FORCE RLS / singleton 紅線）+ `tests/e2e/admin-login-permissions.spec.ts`（守 schema 改動不炸登入 + 權限路徑）。

緣由：防 4/20「改 RLS 後全員登不進去」事故重演。

## 為什麼 2026-05-27 停用

1. **從沒真正運作過**：CI 的 e2e-smoke job 需要 Supabase 連線 secrets（URL/ANON_KEY/SERVICE_ROLE_KEY/TEST_PASSWORD），但 repo 從沒設 → 每次 build/start 取不到 Supabase client 就 fail。`injecting env (0)` + `Your project's URL and Key are required`。
2. **port 撞號**：ci.yml 手動 `npx next start &` + playwright.config 的 `webServer`（CI 時 `reuseExistingServer: false`）又起一個 → `EADDRINUSE :::3000`。
3. **開發階段非必要**：漫途自用、員工少、改動可人工驗（登入用著就知道）、RLS 另有獨立 CI 守門（`audit:rls` / RLS 6 層守門）兜底。這隻一直紅著也沒釀禍 = 現在非關鍵。
4. 一直紅誤導人（以為 CI 有問題）。

## 未來要啟用時怎麼建（三條路 + 必修 port）

**先決：啟用開關** — 設 repo variable `E2E_SMOKE_ENABLED=true`（job 的 `if` 就會跑）。

**DB 來源（三選一）**：
1. **升 Supabase Pro** → 解鎖 branching → 複用 `pr-supabase-branch.yml`（目前是 Free tier 佔位空殼、Pro 才能真跑）。最乾淨、要月費。
2. **CI 跑本地 Supabase stack**（`supabase start` + migrations + seed 測試資料）。不花錢、工程不小、要保證 schema/seed 跟 production 同步。
3. **開一個免費「測試專用」Supabase project**（對它跑、不碰 production、它的 service_role 暴露也不怕；要套 schema + seed 4 workspace + TESTUX/E001 員工+密碼）。← **建議**：不花錢、不暴露 production 萬能鑰匙、隔離乾淨。

**必修 port 撞號**：二選一，不要兩個 server 都起 ——
- ci.yml 手動 `next start` + 把 playwright `webServer` 拿掉（或 `reuseExistingServer: true`）；或
- 不手動 start、讓 playwright `webServer` 自己起（但要改成 `next start` production build、非 `npm run dev`）。

**secrets（啟用時設）**：依選的 DB 來源，設 `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` / `SUPABASE_SERVICE_ROLE_KEY` / `TEST_PASSWORD`（指向 test DB、不是 production）。

## 什麼時候該做

未來 SaaS 化、多客戶在用、多人同時改 code、沒人每次手動驗登入時 — 那時這道自動守門才重要。現在（單租戶自用）不急。
