## 範圍拍板（2026-05-22 William）

**首發只做兩個產品**：

1. ✅ **一般信用卡刷卡**（EPOS、TransMode=0、走 URL 付款頁 HPP iframe、不分期 / 不紅利 / 不 QRCode）
2. ✅ **豐收款**（虛擬帳號匯款、走永豐金融 API）

**不做**：

- ❌ 信用卡分期 / 紅利折抵 / QRCode
- ❌ Apple Pay / Google Pay / Samsung Pay（Token Pay API）
- ❌ 銀聯卡
- ❌ 後台代刷（API 元件直連、PCI 負擔重）

**架構拍板**：

- 資料模型走 **B 方案**（kind + provider 兩維、kind 維持業務分類、provider 開放擴充）
- **line-payment-bot 併入 ERP**、sinopac 模組設計成獨立 lib、ERP 帳單 / channel bot（LINE/FB/IG）都能呼叫
- 沙盒走「本機 Supabase + Supabase branch」混用、不另開 staging project

## 目前狀態（截至 2026-05-22）

### 收款方式現況

- 走 `src/constants/payment-method.ts`、固定 5 種 kind（William 2026-05-18 拍板）：
  - `wire_transfer`（匯款）
  - `card`（刷卡）
  - `cash`（現鈔）
  - `check`（支票）
  - `other`（其他）
- 每個 kind 是「**分類**」、實際 name / 帳號 / 手續費由客戶自己填（譬如「玉山銀行匯款」「中信刷卡機」）
- ERP **完全沒有任何銀行 API 接入**、grep 0 行 sinopac / EPOS / 永豐
- `card` kind 目前只用來「**記錄**」刷卡手續費分錄、不會真的去發授權

### 既有的永豐相關產物（在 ERP 外面）

- `~/Projects/_archive-misc-2026-05/資料夾/永豐EPOS_ERP整合技術規格書.docx`（2025/7、v1.0）— 整理永豐 EPOS 三通道技術規格、規劃文件、未實作

## 未來目標

### 商業模型

| 角色                  | 跟永豐關係                                          | 做的事                                                |
| --------------------- | --------------------------------------------------- | ----------------------------------------------------- |
| **VENTURO 平台**      | 跟永豐簽合約、拿到平台級特店號（ShopNo / API 憑證） | 一次性把刷卡 / 虛擬帳號 / Webhook 接通                |
| **客戶（SaaS 租戶）** | 不直接面對永豐                                      | 在 ERP 內勾選「永豐豐收款」「永豐線上刷卡」、自動可用 |
| **永豐銀行**          | 平台對接、不認個別客戶                              | 提供 EPOS / 虛帳號 API                                |

> 跟「客戶自己跟銀行簽、自己拿 API key」的模式相反、走「**平台轉售**」概念、類似 Stripe Connect。

### 客戶面操作流程

1. 客戶進 `/finance/settings` → 收款方式 tab
2. 「新增收款方式」→ kind 選單多出兩個選項：
   - 永豐豐收款（虛擬帳號收款）
   - 永豐線上刷卡（信用卡授權）
3. 選定後不用填銀行帳號 / API key（平台層處理）、只填內部顯示名、會計分錄科目
4. 儲存 → 此 method 出現在帳單 / 收款單的「收款方式」下拉
5. 業務開帳單時選此方式 → 系統自動觸發對應流程：
   - 豐收款 → 系統開虛擬帳號、客戶匯款後 webhook 自動完成收款
   - 線上刷卡 → 系統發送付款連結（URL 付款頁 / iframe）給客戶刷卡

## 永豐兩個產品的差異

| 比較項       | 永豐豐收款（虛擬帳號）          | 永豐線上刷卡（EPOS）              |
| ------------ | ------------------------------- | --------------------------------- |
| **技術通道** | 永豐金融 API、開立虛擬帳號      | EPOS 系統三通道擇一               |
| **客戶體驗** | 拿到專屬虛帳號 → ATM / 網銀匯款 | 點付款連結 → 線上輸入卡號         |
| **入帳時間** | 對方匯款後即時 webhook          | 即時授權、後續請款                |
| **手續費**   | 通常一筆固定 NT$10-30           | 通常 2%-3%                        |
| **適用**     | B2B 大額、客戶習慣匯款          | B2C / 小額 / 急件                 |
| **PCI 負擔** | 無（不碰卡號）                  | 看通道：URL 付款頁低 / API 元件高 |
| **3D 認證**  | 不適用                          | 可選                              |
| **退款處理** | 走人工退匯                      | API 退貨 / 沖正                   |

> 兩個都要做、不是二擇一。客戶各種帳單情境會用到不同產品。

## 資料模型影響（待拍板）

### 選項 A：擴充 PaymentMethodKind enum

```ts
export type PaymentMethodKind =
  | 'wire_transfer'
  | 'card'
  | 'cash'
  | 'check'
  | 'other'
  | 'sinopac_collect' // 永豐豐收款
  | 'sinopac_card' // 永豐線上刷卡
```

**優點**：直觀、最少改動。
**缺點**：未來加台新、玉山、Stripe 都要加 enum、PaymentMethodKind 會膨脹。

### 選項 B：拆 kind + provider 兩欄

```ts
// kind 維持 5 種（分類）
kind: 'card' | 'wire_transfer' | ...

// 新增 provider（誰來處理）
provider: 'manual' | 'sinopac_collect' | 'sinopac_card' | 'taishin' | ...
```

**優點**：擴充性好、kind 維持業務分類、provider 管金流通道。
**缺點**：兩欄要同時管、邏輯複雜一點。

> 我建議選項 B、SaaS 長期路線比較乾淨、但要 William 拍板。

### 平台層需要的新 table

- `platform_payment_providers`（平台層註冊的金流商、跨 workspace 共用）
  - provider_code（sinopac_collect / sinopac_card / ...）
  - 永豐特店號、HashID、API endpoint（**只存在平台 workspace、加密**）
  - enabled
- `workspace_payment_provider_settings`（每個客戶開了哪些 provider）
  - workspace_id
  - provider_code
  - 客戶端設定（譬如返回 URL 客製、收款人顯示名）
- `payment_transactions`（每筆刷卡 / 虛帳號交易紀錄）
  - workspace_id
  - provider_code
  - 對應的 ERP 收款單 id
  - 永豐回傳的 ApproveCode / TransNo
  - 狀態（authorized / captured / refunded / failed）
  - 原始 webhook payload

## 接入流程（誰做什麼）

### Phase 0：平台對外簽約（William + 永豐業務）

- 簽約、拿到測試環境特店號
- 確認費率、結算頻率、爭議處理流程

### Phase 1：平台層串接（VENTURO 工程）

- 建上述 3 個 table
- 寫 `src/lib/payment-providers/sinopac/` module：
  - 豐收款：開虛擬帳號 / 查詢 / 對帳
  - EPOS 刷卡：授權 / 請款 / 退貨 / 查詢
- Webhook endpoint：`/api/payment-webhooks/sinopac/[type]`
- 加密 / Nonce / Sign / AES-CBC（參考既有 line-payment-bot 的 sinopac.js）

### Phase 2：客戶端 UI（VENTURO 工程）

- 收款方式設定加 kind / provider 選單
- 帳單 / 收款單流程接入：
  - 豐收款 → 開單時生成虛帳號、顯示給客戶
  - 線上刷卡 → 開單時產生付款連結、傳給客戶
- 對帳介面：每日跟永豐 API 對帳、抓出差異

### Phase 3：權限 / 計費（VENTURO 商業）

- workspace_features 加 `payment_gateway` feature flag
- 開通邏輯：平台同意 + 簽客戶補充協議
- 計費模式：平台抽手續費 X% / 按筆收費 / 月費（待 William 拍板）
