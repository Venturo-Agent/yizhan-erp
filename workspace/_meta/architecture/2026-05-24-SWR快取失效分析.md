# SWR 快取「刪除後卡住」整體分析（William msg 458 要求、2026-05-24）

## 根因（已找到 + 確認已修）

entity hook 早期設 `dedupingInterval: Infinity`（設計上想靠 Supabase realtime 推送更新、不主動重抓）。但 realtime 因 supabase-js + ssr cookie 不穩 → **寫入後的 mutate-revalidate 被 Infinity dedupe 砍掉** → UI 卡 stale、要 F5。
**5/18 已修**：`createEntityHook.ts:94` dedupingInterval Infinity → 2000ms、mutate 強制 revalidate 不再被砍。= William 說的「已經優化」。

## 現況：主資料路徑健康

- `entityHookCrud.ts`：create / update / **remove(delete)** 全部 call `invalidateEntity`（SWR + IndexedDB 一起清）。走 entity hook 的刪除會即時更新、不卡。
- `apiMutate`（`src/lib/swr/api-mutate.ts`）= 寫入 + 失效 SSOT。

## 系統性殘留風險（還會偶發卡住的真正來源）

- 全站約 **57 處直接 `.delete()`** + ratchet 凍結的 **53 supabase-writes + 18 useswr** 違規 = 繞過 entity hook / apiMutate 的直接寫/刪。
- 這些寫了 DB 但**沒通知快取** → 刪除後畫面不更新、要 F5/換頁。
- 已被 ratchet（`.eslint-suppressions.json`）列管凍結、新的 CI 擋、不會再長。
