## 二、業務概念

### 代轉是什麼

旅行業「代收轉付收據」：旅行社代旅客付給航空公司 / 飯店 / 地接的代收款項憑證。
跟「發票」是兩回事 —— 代轉收據是代收代付的錢（非旅行社營收），發票是旅行社賺的服務費。
現有骨架命名「電子收據 / travel_invoice」偏發票思維，**建議正名為「代轉管理」**。

### 紙本 vs 電子（二選一）

|                | 紙本代轉                           | 電子代轉                                |
| -------------- | ---------------------------------- | --------------------------------------- |
| **載體**       | 公司買進的實體收據本 / 套印單      | 藍新平台開立的電子收據                  |
| **字軌來源**   | 公司自己的收據本，**ERP 內部編號** | 跟藍新「申購」的字軌，**藍新自動配號**  |
| **開立動作**   | ERP 記錄 + 列印套印                | 打藍新 API 開立 + 取回收據號 / 展示網址 |
| **外部依賴**   | 無                                 | 藍新測試/正式會員 + IP 白名單           |
| **財政部上傳** | 公司自行處理（媒體申報）           | 藍新平台代上傳（每單月一日排程）        |

> 「公司可能會進一套新的紙本」→ ERP 要能登記一本新收據本的字軌起號，之後開一張流水 +1。

## 三、廠商串接事實（藍新科技 NewebPay — travelinvoice 平台）

### 環境

|          | 平台網站（人工操作）               | API 串接網址                                  |
| -------- | ---------------------------------- | --------------------------------------------- |
| **測試** | https://cwww.travelinvoice.com.tw/ | https://capi.travelinvoice.com.tw/invoice_xxx |
| **正式** | https://www.travelinvoice.com.tw/  | https://api.travelinvoice.com.tw/invoice_xxx  |

- 測試環境：登入測試會員 → 申購測試字軌 → 測試付款選 WebATM 不需真付款
- 串接前要先跟客服申請 **IP 白名單**（最多 10 組）

### 字軌申購（重要 — 不是 API、是平台後台人工）

電子字軌在藍新平台網站「【電子收據管理 / 申購待轉收據】」手動申購。
**ERP 不管申購**，只管「開立 + 記錄收據號」。開立時藍新自動從已申購字軌池配號，回應裡帶 `Surplus`（剩餘張數）→ ERP 可顯示「字軌還剩幾張」。

### 開立收據 API（invoice_issue）

POST 兩個外層參數：

- `MerchantID_`：旅行社統編（8 碼）
- `PostData_`：AES 加密後的下列欄位（加密方法見規格書附件一）

PostData\_ 內含欄位（節錄關鍵）：

| 參數                            | 說明                                    | 對應 ERP                             |
| ------------------------------- | --------------------------------------- | ------------------------------------ |
| `Version`                       | 固定 `1.1`                              | 常數                                 |
| `TimeStamp`                     | Unix 秒數                               | 即時產生                             |
| `MerchantOrderNo`               | 自訂編號（英數+底線，30 碼）            | **我方代轉單號**（走 `@/lib/codes`） |
| `Status`                        | 1=即時開立 / 2=預約開立                 | 預設 1                               |
| `Category`                      | B2B（統編）/ B2C（個人）                | 收據種類                             |
| `BuyerName` / `BuyerUBN`        | 買受人名稱 / 統編（B2B 必填）           | 客戶                                 |
| `BuyerEmail`                    | 買受人信箱（**必填**，寄收據通知）      | 客戶                                 |
| `SellerName`                    | 經辦人名稱（**必填**）                  | 開單員工                             |
| `TotalAmt`                      | 收據總額（=明細小計總和）               | 金額                                 |
| `ItemName/Count/Unit/Price/Amt` | 商品明細（多項用 `\|` 分隔，最多 7 項） | 代轉明細                             |
| `TourName/TourNo/TourDate`      | **團名 / 團號 / 出團日**（原生綁團）    | **綁團三欄**                         |
| `TaxNoted`                      | 申報註記 0/1                            | 申報狀態                             |

開立回應（string、`&` 分隔、結尾 `EndStr=##`）：

| 回應欄位         | 說明                               | ERP 要存 |
| ---------------- | ---------------------------------- | -------- |
| `Status`         | SUCCESS / 錯誤碼                   | 開立結果 |
| `InvoiceNumber`  | 收據號碼（9 碼）= 電子字軌號       | ✅       |
| `InvoiceTransNo` | 開立流水號（20 碼，作廢/觸發必用） | ✅       |
| `RandomNum`      | 防偽隨機碼（8 碼）                 | ✅       |
| `Surplus`        | 字軌剩餘張數                       | 顯示用   |
| `CheckCode`      | 驗證回應合法性（附件二）           | 驗證後存 |
| `DisplayURL`     | 收據展示網址（可直接給客戶看）     | ✅       |

### 其他 API（同 MerchantID* + PostData* 模式）

| 動作          | endpoint（測試）                                            |
| ------------- | ----------------------------------------------------------- |
| 作廢收據      | `capi.travelinvoice.com.tw/invoice_cancel`（待細讀 p24-27） |
| 開立折讓      | `.../allowance_issue`（待細讀 p32-35）                      |
| 變更收據      | `.../invoice_alter`（買受人電話/地址/備註，p40-44）         |

| 觸發/取消預約 | `.../invoice_touch_issue`（p20-23）                         |

> 加密：AES（PostData\_）+ CheckCode（SHA256 類）—— 藍新標準模式，實作時細讀附件一/二，封裝進 `src/lib/payment-providers/neweb-travelinvoice/`。

## 六之二、紙本字軌作廢／重印機制（William 2026-05-23 提問、拍板方向）

### 問題（William 描述）

紙本用點陣撞針印表機 + 套印單、實體對位容易跑掉。實務上會發生：

- 按列印 → 配到 1 號 → 印出來位置歪了、作廢
- 重印 → 配到 2 號 → 又歪、作廢
- 第 3 次 → 3 號終於印對

→ 中間 1、2 號要作廢。William 原本想的是「列印後回填最後成功號（3），讓中間（1、2）自動變作廢」。

### 為什麼不採「回填讓中間自動作廢」

- ❌ **連鎖錯風險**：填錯一個號、後面全部跟著錯位（William 自己也擔心這點）
- ❌ **無痕跳號違法遵**：代轉收據字軌號是法定憑證、媒體申報要能交代「哪些號作廢、為什麼」。讓號「自動消失」＝沒有 audit trail、稅務/審計查不到
- ❌ 違反紅線 D 精神（不開「回頭塗改」後門）：作廢要正規留痕、不是讓資料消失

### 採用方案：單張作廢制 + 試印校準 + 作廢留痕

**核心：每一個配出的紙本號 ＝ 一筆獨立記錄、有自己的狀態（待確認/已開立/已作廢），不靠「回填」連動。**

| 步驟        | 動作                                      | 系統行為                                                                                                            |
| ----------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| 1. 開立     | 員工選團+填資料、按開立                   | 從 active 字軌 advisory lock 配 `current_no+1` → 建 travel_invoices row（medium=paper、status=`pending`待列印確認） |
| 2. 列印確認 | 印出來、位置 OK → 按「確認列印成功」      | row status → `issued`、綁定該代轉內容                                                                               |
| 3. 印壞作廢 | 位置歪/印糊 → 對「這一張」按作廢 + 選原因 | row status → `voided`、寫 `travel_invoice_voids`（號、原因、誰、何時）；**該號就停在 voided、不重用**               |
| 4. 重印     | 作廢後按重印                              | 配「下一個號」（current_no+1）、回到步驟 1 的新 row                                                                 |

- **號只增不減**：current_no 持續遞增、作廢的號保留為 voided row、不回收不重用 → 字軌連續可稽核
- **每號狀態獨立**：1=voided、2=voided、3=issued，各自一筆 row、互不連動 → 填錯一張只錯一張、不連鎖
- **作廢留痕**：沿用既有 `travel_invoice_voids` 表（號/原因/voided_by→employees/voided_at），對齊紅線 B、D

### 試印校準頁（治本：減少作廢真實號）

點陣印表機位置跑掉是「實體對位」問題、不該靠一直作廢真實號碼解決。

- UI 加「**試印校準頁**」按鈕：印一張**不佔號**的測試頁（印格線/定位標記/範例字）
- 員工拿它對印表機位置、對準了再正式開立（步驟 1）
- 校準頁不建 row、不動 current_no → 對位歸對位、開立歸開立

### Schema 影響（補在第五節）

- `travel_invoices` 主表的 `status` enum 加 `pending`（待列印確認）：`pending` / `issued` / `voided`
- 紙本作廢沿用既有 `travel_invoice_voids` 表（不新增表）
- `travel_invoice_paper_tracks.current_no` 配號走 `@/lib/codes` advisory lock（防兩人同時開立撞號）
