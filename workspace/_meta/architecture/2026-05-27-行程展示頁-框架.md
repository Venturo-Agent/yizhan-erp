## 〇、交付定位（2026-05-27 William 拍板、專案交給 Robin）

**一句話交付目標**：做出「**行程 AI 畫的感覺**」——同業把手上的行程丟進來，**按一鍵就變成能直接給客人看的漂亮網頁**。這是賣給**同業**的 SaaS 賣點。

**賣點命脈（決定產品成不成）**：同業手上的行程**七成是陽春的（每天只有一句話）**。所以關鍵不在「資料完整時排得多漂亮」、而在「**陽春行程一鍵也能變體面網頁**」。→ 核心原則第 3 條的「優雅降級」**不是技術細節、它就是賣點本身**。

**經濟學（為什麼這條路可行）**：題目（版型 + 規則 + JOIN 景點庫補料）做得越滿、AI 剩下的事越小 → 用越便宜的小模型（MiniMax）就夠、又快又穩又不幻覺。慢工花在前面做模子、後面每團每次點 AI 都享受便宜又準。**走「完整的 B」（規則排版 + AI 潤色），不賭 A（AI 自己排整頁）。**

## 一、核心原則（這套框架的脊椎）

> William 2026-05-27 從「總覽要可選」推導出的通則。

1. **預設順序固定** — 下面的章節表是骨架、決定客人從上往下讀的動線、不隨團變。
2. **每塊積木可開關** — 總覽要不要、航班有沒有含機票、飯店有沒有資料……一團一團不同、靠開關不靠改版。
3. **資料不足自動降級** — 有圖大卡 / 沒圖小卡 / 根本沒資料整塊隱藏。**不開天窗**。

譬喻：這是一本雜誌的**固定欄目表**、但每期可抽掉沒料的單元、不留空頁。

## 三、餐廳穿插 vs 飯店統一（設計邏輯）

兩者處理不同、是刻意的：

- **餐廳屬於「某天某餐」**：跟當天行程綁死（早午晚）、穿插在 `day` 裡才有脈絡 → **小卡 inline**、流水帳裡的點綴。
- **飯店是「賣點門面」**：過夜地、團質感象徵（米其林餐廳飯店、溫泉）→ 統一拉出來**大卡 gallery**、當壓軸 showcase。

譬喻：餐廳是路邊隨手拍的風景、飯店是進門前的全家福（一個隨流程走、一個要正式擺拍）。

## 四、風格整合：manhattan 外殼 + B+C 內容物

「B+C 協調、manhattan 簡約」怎麼整合——關鍵是**作用在不同層、不衝突**：

- **manhattan = 版面骨架與節奏**（外殼）：大圖封面、章與章之間留白呼吸、克制文字、雜誌式分章。
- **B+C = 資訊卡**（內容物）：航班卡、每日行程卡、飯店卡（含米其林標）、餐廳卡、領隊卡。

譬喻：**manhattan 是雜誌的開本與留白、B+C 是雜誌裡每篇報導的資訊圖卡。**

→ 「簡約 vs 資訊充足」不是二選一：**用留白做章節間隔（簡約感）、用資訊卡裝內容（充足感）**、分屬不同層。

## 七、兩層料來源（為什麼「不可能多爛」— William 2026-05-27 校正）

別被「行程自己存的很陽春」嚇到。料有**兩層**：

1. **行程自存**（`daily_itinerary` jsonb）：常陽春。實測 17 團、**7 成第一天只有一句 title**（如「上海 → 武康路 → 安福路」）、3 成有結構化景點。
2. **景點庫**（`attractions` 表）：**豐富**。圖、描述、tags 都在。行程的景點靠 `activities[].attraction_id` 連回庫（`/api/itineraries/[id]/route.ts` 已有「用 attraction_id 補描述」機制）。

→ **一鍵產出的料策略**：先盡量從**景點庫 JOIN 撈滿**（圖/描述/tags）、撈不到才降級。**降級是最後手段、不是常態。有景點庫當靠山、成品不會爛。**

**唯一缺口**：那句 title 裡的景點若**沒連 `attraction_id`**（純打字）→ JOIN 不到 → 靠「景點名比對景點庫」搶救、或引導業務補連結。是要補的小缺口、不是死穴。

**餐廳 / 飯店**：餐廳庫（`restaurants`、米其林屬性已整合進來、不獨立）有料、但行程餐食目前**沒連 `restaurant_id`**（純文字）→ 要享受料倉得先補「餐食連餐廳庫」。飯店同理。

## 十、風格池 + 排印規範（2026-05-27 William 補充、修正圓桌）

### 風格依主題、不是一種走天下（修正圓桌「克制留白」）

- **留白不是鐵律**——是「沉靜主題」（文化古蹟 / 合掌村 / 溫泉）的語言。
- **熱鬧主題**（親子樂高 / 繁華都市 / 夜市美食）**該飽滿、繽紛、密集**，硬留白反而不對味。
- **把所有參考抓進池子**：`ui-ux-pro-max` 50+ 風格 + `awesome-design-md` 71 品牌全納為參考池、依團的 **tags / category** 選對應風格。
- 系統支援**多套風格 token**（不只「優雅」一套；呼應早期 A minimal / B explorer / C elegant 三套主題可全納）。editorial 精品只是「沉靜主題」的 baseline 之一。

### 排印規範（所有版型必守）

- **分隔符一律用「・」**（全形中黑點 U+30FB、居中、**貼緊無空格**，如 `飯店入住・21世紀美術館`）。
- ❌ 不用小圓點「·」(U+00B7)、不用 bullet「•」、不用句號當分隔。
- **句末一律不收句號「。」**（標語 / 卡片 / eyebrow / 內文段落**結尾**都拿掉）；**句中句號**（分隔段落內部兩句的）**保留**、否則文字會糊在一起。（William 2026-05-27 拍板）
- 中文排印續守：禁新細明體、中文不偽斜體、思源宋體 + 蘋方（見 memory `venturo-chinese-typography-rules`）。

## 十一、資料庫地基 + 相輔相成（2026-05-27 William 定調）

**天花板原則**：資料庫品質 = 整個系統的天花板。標籤（景點 `tags` / 餐廳 `cuisine_type`・`specialties` / 飯店 `highlights`）→ 聚合辨識「**這團主題**」→ AI 選風格 + 凸顯亮點。

**地基體檢（2026-05-27 實測）**：

| 庫 | 總數 | 有標籤 | 有描述 | 有圖 |
|---|---|---|---|---|
| 景點 | 2467 | 55% | 99% | 40% |
| 餐廳 | 312 | 8% | 85% | 39% |
| 飯店 | 490 | 9% | 17% | 6% |

→ **景點庫強壯**（主力靠山、系統現在就立得住）；餐廳標籤弱、飯店近空殼。

**相輔相成（William 拍板）**：內容填充（補標籤 / 描述 / 圖，尤其飯店）由**公司後續補齊、不歸 code 側管**。系統側「**當料會齊**」設計版型、料還沒到就優雅降級。**料補一分、畫面好看一分**——兩邊各自推進、互相成就、誰都不卡誰。

**兩種工分流**：寫 code（版型 / AI / JOIN，Robin）｜建資料（補內容，公司）。互不阻擋。
