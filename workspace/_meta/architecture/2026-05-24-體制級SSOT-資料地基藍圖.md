## 1. 現況：地基是三套搶當真相的拼接

今天挖出的事實（證據在 `2026-05-24-SWR快取失效分析.md`）：

| 機制                             | 做什麼                           | 問題                                                                                                                                                   |
| -------------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SWR 本地快取                     | 猜要不要重抓（dedupingInterval） | 各 hook 各自調（2s/5min/30min/1hr）、靠記憶力                                                                                                          |
| 手動失效（apiMutate/invalidate） | 寫完通知快取                     | **~110 處直接寫/刪繞過它** → 刪了畫面不動                                                                                                              |
| Realtime 推送（useRealtimeSync） | DB 變動推給 client               | **4 張表沒進 publication**（bank_accounts/contracts/expense_categories/payment_methods）→ 靜默失效；auth 靠 3 處 setAuth + library bug workaround 黏著 |

→ 三套各做各的、沒有一個說了算 → **沒有單一真相來源 = 卡住的病根。**

**身分線也是拼接的**：登入 → JWT（昨天剛換 getClaims）→ 要分別 setAuth 給 realtime（auth-sync.ts + client.ts + entityHookRealtime.ts 三處）+ API 各自驗。任何一處沒同步、realtime 就收不到 RLS 保護的推送。

## 2. 商業級 SaaS 地基：不該有 / 該有 / 最小可行

### ❌ 不該有

- 多套搶當真相的機制（本地快取 vs 手動失效 vs realtime 各自為政）
- 散刻寫入繞過中央失效（刪了畫面不動）
- 每個 hook 各自手調 dedup、靠人記憶維護
- realtime 靜默失效（沒進 publication / auth 沒帶到）卻無信號
- auth / realtime / 快取 用 workaround 互黏、動一個崩一片

### ✅ 該有

1. **一條身分線**：登入 → JWT → **同一份身分**貫通 API-RLS / realtime / client。一個 identity SSOT、一處設定、處處生效。
2. **DB 是唯一真相、Realtime 是主推送**（不是事後貼）。本地快取只是 DB 的鏡像、不是競爭真相。
3. **寫入樂觀更新 + 失敗自動還原**。
4. **失敗可觀測**：realtime 斷線/沒訂閱要有信號（UI 提示或 log）、不是默默卡住。
5. **租戶隔離是地基保證**：workspace_id RLS 守門（紅線 H）是地基性質、不是每張表各自記得守。

### 🎯 最小可行地基（MVP foundation）

上面變成系統的「**性質**」——任何新功能天生就有：

> 「寫了即現、刪了即消、不用 F5、跨使用者同步、RLS 安全、出錯看得到」

而不是每個功能各自努力達成。**這就是商業級 SaaS 跟「能跑的 side project」的分界線。**

## 3. 統一地基設計（我主導的方向）

**一個原則貫通：DB 是真相 → Realtime 主推 → 單一身分認證 → 本地鏡像 → 寫入樂觀。**

```
登入 ──► identity SSOT（JWT）
            │  一處設定、處處生效
            ├──► API client（RLS）
            ├──► Realtime client（setAuth、token 刷新自動跟）
            └──► 讀取層

DB（唯一真相）
  │  變動
  ├──► Realtime 主推 ──► 所有 client 的本地鏡像即時更新
  └──► 寫入：樂觀更新 → apiMutate → realtime 確認（雙保險）

本地快取 = DB 鏡像（不是競爭真相）
  - dedup 統一策略（不是每 hook 各調）
  - realtime 沒到時 mutate-revalidate 當 fallback
```

## 4. 升級路線：兩條路 + 我的推薦

### 🅰 黏合升級（unify what exists）— **我推薦這條**

把現有的 SWR + entity hook + realtime + auth 收斂成一套一致機制：

1. **identity SSOT**：把 3 處 setAuth 收成一個 auth 模組、token 刷新自動同步到 realtime（formalize auth-sync）。
2. **Realtime 升為主推送**：每個 entity hook 都訂閱、realtime 更新快取為主、mutate 當 fallback。
3. **publication 完整性 = CI 守門**：`audit:realtime` 強制「每個 entity hook 的表都在 publication」、缺就擋 PR（4 張漏的先補）。
4. **滅散刻寫入**：110 處直接寫/刪 → 收進 apiMutate（ratchet 已列管、逐步清）。
5. **可觀測性**：realtime 連線健康信號。

- **成本**：中、漸進、可一塊塊上、低風險（不換框架）。
- **風險**：低（沿用熟悉的 Supabase 原生能力）。
- **為什麼推薦**：商業級夠用、不賭大遷移、團隊（含 Claude）已熟、撞牆能 escape（薄抽象、符合既有「不過度抽象」原則）。

### 🅱 導入同步引擎（PowerSync / ElectricSQL / Replicache）

專門的 local-first 同步層、離線 + 即時 + 衝突解決全內建。

- **成本**：高（新框架、學習曲線、遷移所有讀寫）。
- **風險**：高（厚抽象、撞牆難 escape、跟 Supabase RLS 整合要驗）。
- **何時才值得**：離線優先是核心需求、或規模到單純 realtime 撐不住。**現在不到這個點。**

→ **我的推薦：🅐**。把地基黏成一套一致的、Realtime 當主軸（正中 William 的直覺）、不賭大遷移。🅑 留作未來「真的需要離線優先」時再評估。
