# 官網與展示行程 — 資訊架構 + 設計框架

> 這份文件不重複既有 websites 模組規格（Max 的 5 SSOT / 6 層 / 紅線對齊）、也不重複 canvas-editor 的 C→B→A 路徑卡。
> 這份只專注三件 William 親口要的：
> 1. **官網 8 區**該長什麼樣、哪些是組件、AI 在每區做什麼
> 2. **展示行程 5 項**的資料來源、組件設計、AI 填或抓
> 3. **「亮點抓取」機制**（William 最在意）——行程資料庫裡的米其林/音樂劇/特殊體驗、怎麼自動冒到 Hero 跟本次特色區

## 0 · 救護車式重點（先講會死人的、再講怎麼治、最後講原理）

**這次最該先拍板的一件事**：亮點抓取「**用 AI 抽文字**」還是「**讀結構化欄位**」？

我查了 production 資料庫（不是猜的），結論很關鍵——

> **漫途的行程資料庫，米其林/特殊住宿/特色這些「亮點」大部分已經是「打勾的欄位」、不是埋在一段文字裡。**
> 餐廳表有 `michelin_stars`（米其林星數）、`bib_gourmand`（必比登）、`signature_dishes`（招牌菜）、`chef_name`（主廚）；
> 飯店表有 `has_michelin_restaurant`、`star_rating`（星級）、`awards`（得獎紀錄）、`butler_service`（管家服務）、`highlights`（亮點清單）；
> 行程表有 `features`、`focus_cards`（本次特色卡，已是 jsonb 結構）。

**所以建議是「結構化欄位優先、AI 只做排序 + 潤色」——不是「AI 從一段文字硬�mining」。** 理由：欄位是打勾的、抓出來 100% 準（米其林就是米其林、不會幻覺）；AI 從自由文字抽容易抓錯、抓漏、或編出一個資料庫裡根本沒有的「米其林」。AI 的價值放在「**這團 10 個亮點，挑哪 3 個放 Hero、用什麼一句話講**」，這才是 AI 擅長且不會出大錯的事。

**第二件該拍板的**：要不要新增一個「**亮點標記**」的人工開關欄位？建議**加一個輕量的 `is_highlight` 標記 + `highlight_label`**（讓業務手動「釘選」某個亮點到首頁），但**不是必填、不擋自動抓**。理由見第四章。

其餘都是設計細節、不急。

## 一、設計定位

### 一句話定位

> **漫途官網 = 「台灣旅行社的資訊密度」與「國際精品旅遊的編輯質感」之間，站在偏編輯質感那一側、但保留台灣人要的『看得到價格、看得到日期、馬上能問』。**

白話：不要做成雄獅那種「滿屏紅底黃字、一頁塞 50 個團、像賣場 DM」；也不要做成 Black Tomato 那種「全屏黑底電影感、連價格都不給、要 enquire」——前者俗、後者台灣客戶會嫌「問了老半天不知道多少錢」。漫途站中間偏質感：**大圖留白 + 宋體標題的編輯感（質感那側），但每張商品卡看得到出發日、天數、價格帶、一鍵詢問（實用那側）。**

### 為什麼站這裡（對比兩端）

| 維度 | 台灣本土（雄獅 / 東南 / 五福 / KKday / 易遊網） | 國際精品（Black Tomato / Kuoni / Airbnb） | 漫途該怎麼融合 |
|---|---|---|---|
| **Hero** | 輪播 banner + 大搜尋框、資訊滿載 | 單張全屏攝影 + 極簡一句標題、留白 | **單張大圖 + 一句宋體標題 + 一條精簡搜尋**（質感打底、但搜尋不藏） |
| **資訊密度** | 高（一頁塞滿團、價格紅字跳動） | 低（一屏一個重點、慢慢滾） | **中**：列表頁密一點（客戶要比價）、首頁/詳情頁鬆一點（要嚮往感） |
| **商品列表** | 格狀密排、紅色促銷標、價格最大 | 大圖卡、文字克制、價格低調或不給 | **大圖卡 + morandi 標籤 + 價格帶清楚但不跳紅**（看得到、不俗艷） |
| **行程詳情** | 表格式逐日、文字密、圖小 | 編輯式分鏡、大圖、留白、攝影主導 | **編輯式分鏡為主**（這正是現有 Canvas 15 積木的氣質）**+ 可摺疊的逐日明細**（給要看細節的客戶） |
| **價格揭露** | 永遠最大最紅 | 常常藏起來要詢問 | **揭露但克制**：價格帶用 morandi 中性色、不用 status-danger 紅 |
| **CTA** | 「立即報名」滿屏 | 「Enquire」單一克制 | **「諮詢專屬業務」**（扣漫途 B2B2C / 業務帶單模式，已有 `?ref=E001` 機制） |

### 視覺氣質（沿用既有、不另起爐灶）

漫途已經有一套成形的旅遊提案視覺氣質（現有 Canvas 渲染器），這套就是官網的基底、**不要重新發明**：

- **色**：morandi 色系（`morandi-gold` 主品牌色 / `morandi-primary` / `morandi-cream` / `morandi-container`）；狀態走 `text-status-*`。編輯器外殼沿用既有暖棕 `#2D1F18`。
- **字**：中文標題走思源宋體（編輯質感、扣現有 Canvas 紅銅宋體風）、內文蘋方 / 思源黑體。**禁用** Inter / Roboto / Arial / 新細明體、中文不准偽斜體（對齊 memory「中文字型紀律」）。
- **攝影主導**：旅遊賣的是「對一趟旅程的嚮往」，圖一定要大、要好。Hero 與商品卡都以攝影為主角、文字克制。
- **反 AI slop**：不做漸層紫 / teal 發光 / 千篇一律的「hero + 3 卡 + CTA」罐頭、不做聊天泡泡感。

### 參考案例（誰做得好 / 爛）

- **學 Black Tomato**（國際精品）：全屏攝影 Hero、destination mega-menu、克制 CTA、暗色電影感的氣場。**但不學它藏價格**。
- **學 Kuoni**（國際精品）：清晰導航、destination 篩選、詳細目的地導覽、aspirational 文案。
- **學 Airbnb**（消費級質感）：暖色、大量留白、攝影信任感勝過字重（display 只用 500/600 不用 700+）、pill 圓角搜尋。**漫途的搜尋條可借這個 pill 質感**。
- **學台灣本土的「實用」、不學它的「俗」**：雄獅/KKday 的「看得到日期+價格+一鍵問」是台灣客戶剛需，保留；但它們的滿屏紅促銷、密到喘不過氣的格狀，丟掉。

## 二、官網資訊架構（8 區逐一）

> 每區格式：**作用 / 內容 / 旅遊業最佳實踐 / 組件（可重用？對應現有 Canvas 積木）/ AI 在這區做什麼（輔助點，非生整頁）**

對齊既有 websites 模組規格的 8 個 component type（`hero` / `header` / `about` / `services` / `tour_listings` / `testimonials` / `contact` / `footer`），這裡把 William 的 8 區對映進去並補齊。

### 區 1 · Hero 頁面

- **作用**：3 秒內讓人「有感覺、想往下滾」。官網的門面。
- **內容**：單張全屏攝影（或低調 ken-burns 慢推，不是浮誇輪播）+ 宋體大標一句（品牌主張，如「為每一次出發，找到值得的理由」）+ 副標一句 + 一條精簡搜尋條疊在底部 1/3 + 一顆克制 CTA（「探索行程」）。
- **旅遊業最佳實踐**：精品端用單張靜態大圖（Black Tomato）、本土端用搜尋框（雄獅）。漫途兩者疊加：**圖是主角、搜尋條浮在下緣不搶戲**。Hero 圖優先用 `next/image` + priority、避免拖慢首屏。
- **組件**：新組件 `hero`（variant：全屏攝影 / 影片背景 / 分割版）。**這是官網新積木、Canvas 既有的 `cover`（C01）是它的近親**——cover 是單團封面、hero 是整站封面，可共用底層樣式 token。
- **AI 輔助點**：① 從「本站精選團」自動挑一張最好的攝影當 Hero 背景（依圖片解析度/構圖打分，或直接讓業務選）；② **把全站近期最強的亮點（米其林 X 家、特殊體驗 Y）凝練成 Hero 副標一句**（這就是亮點抓取的全站版，見第四章）；③ 潤標題文案。**AI 不生整個 Hero 版面，只填內容 + 凸顯重點。**

### 區 2 · 搜尋欄位

- **作用**：讓「已經知道想去哪/什麼時候」的客戶直接命中。台灣客戶的剛需。
- **內容**：目的地（國家/城市）+ 出發月份/日期區間 + 天數 + 價格帶（可選）。pill 圓角、morandi 中性底。
- **旅遊業最佳實踐**：欄位不要超過 4 個（雄獅塞太多反而難用）；要支援「空查」直接看全部；搜尋結果走 server-side filter（對齊地方法律：列表 server 端篩、跟 PostgREST query string 對齊）。
- **組件**：`search_bar`（可獨立、也可嵌在 Hero）。資料來源 = `tours` 表 `is_public_listed = true` 的團，吃 `departure_date` / `country_id` / `days_count` / `selling_price_per_person`。
- **AI 輔助點**：① 自然語言搜尋（「過年想帶爸媽去日本溫泉、預算 5 萬內」→ 解析成結構化 filter）——這是 v2 加值、v1 先做傳統下拉；② 搜尋無結果時，AI 推薦「相似的可選團」。**AI 不取代搜尋邏輯、只加一層自然語言入口。**

### 區 3 · 標題區（區段標題 / Section Heading）

- **作用**：把首頁切成幾段（精選 / 主題 / 關於），每段一個有質感的引導標題。
- **內容**：eyebrow（小標，如「FEATURED」）+ 宋體主標 + 一句引言。
- **旅遊業最佳實踐**：精品站靠這種 editorial section heading 製造節奏感與呼吸；本土站常常省略、直接堆商品（漫途要保留節奏）。
- **組件**：`section_heading`（純展示小積木、高重用，每段之間插一個）。**對應 Canvas 的 `feature_hero`（C09）的輕量版**。
- **AI 輔助點**：依該段內容自動草擬 eyebrow + 引言文案（業務一鍵採用或改）。**純文案輔助。**

### 區 4 · 特色商品區（精選 / 主打團）

- **作用**：把「漫途最想推的 3-6 個團」用大版面凸顯，不淹沒在列表裡。首頁的business重點。
- **內容**：3-6 張大卡（或 1 大 + 數小的 bento 排法），每張：大攝影 + 團名 + **本次亮點標籤**（米其林/特殊體驗，來自亮點抓取）+ 出發日 + 價格帶 + 詢問鈕。
- **旅遊業最佳實踐**：精選區是「編輯選的」不是「全部」，數量克制（≤6）才有「精選」感；每張卡要有一個「為什麼選它」的鉤子——**這正是亮點抓取的用武之地**。
- **組件**：`featured_products`（variant：大卡橫排 / bento 不對稱 / 雜誌雙欄）。底層卡片 = `tour_card`（與區 5 共用，只是尺寸大、加亮點標籤）。
- **AI 輔助點**：① **核心——從每個精選團抓「本次亮點」生成卡上標籤**（米其林一星、百年溫泉旅館、限定音樂劇，見第四章）；② 草擬每張卡的一句 hook 文案。**這區是亮點抓取在官網側的主舞台。**

### 區 5 · 商品列表（全部行程）

- **作用**：可瀏覽/篩選/比較的完整行程庫。台灣客戶比價、找日期的地方。
- **內容**：格狀或列狀卡片，每張：攝影 + 團名 + 目的地 + 出發日 + 天數 + 價格帶 + 狀態（熱賣/即將成團/已滿，走 `text-status-*`）+ 詢問鈕。配合區 2 的篩選。
- **旅遊業最佳實踐**：**列表預設 20 筆、分頁固定 15 筆、不給「每頁筆數」選擇器**（地方法律效能紀律，省 Supabase egress）；server-side filter；卡片資訊密度可比首頁高（客戶在比價狀態），但仍走 morandi、不跳紅。狀態色克制。
- **組件**：`tour_listings`（既有 websites 規格已列）。卡片 = `tour_card`（SSOT 共用組件）。資料 = `tours` where `is_public_listed = true`，走 entity hook（紅線 F、不散刻 useSWR）。
- **AI 輔助點**：① 每張卡自動掛 1-2 個亮點 micro-badge（輕量版亮點抓取）；② 排序建議（「最多人問的」「即將成團」）。**AI 不改列表邏輯，只加標籤與排序提示。**

### 區 6 · 關於我們

- **作用**：建立信任。旅遊是高信任決策（一趟幾萬塊、出國交給你），這區很關鍵。
- **內容**：品牌故事（宋體長文+攝影）+ 數字佐證（成立年份/服務團數/滿意度）+ 理念。
- **旅遊業最佳實踐**：精品站用 editorial 長文+攝影講「我們為什麼不一樣」；本土站常常只放冷冰冰的公司簡介。漫途走前者，有溫度。
- **組件**：`about`（既有規格已列，variant：圖文左右 / 全幅攝影壓字 / 數字統計帶）。對應 Canvas `spotlight`（C13）兩欄式的氣質。
- **AI 輔助點**：依業務給的關鍵字（成立年/特色/理念）草擬一版品牌故事文案，業務改。**純文案輔助，故事真實性業務把關。**

### 區 7 · 員工介紹

- **作用**：把「人」秀出來。漫途是業務帶單模式（`?ref=E001`），客戶信任的是「那個業務」不只是公司。這區直接扣商業模式。
- **內容**：業務/領隊團隊卡——照片 + 姓名 + 專長路線 + 一句個人介紹 + 聯絡鈕。
- **旅遊業最佳實踐**：本土旅行社很少做（可惜），精品定制旅遊（如 Black Tomato 的 travel designer）都會放——「你的專屬旅遊設計師」是信任放大器。**這是漫途相對台灣同業的差異化亮點。**
- **組件**：`team`（新組件，variant：網格頭像 / 編輯式人物特寫）。資料 = `employees` 表（display_name / avatar_url / 專長），扣既有 `?ref=` 業務機制——點某業務卡可帶 ref 進詢問。
- **AI 輔助點**：依業務的服務路線/年資草擬一句個人簡介。**純文案輔助。**

### 區 8 · Google 內容（評論 / 地圖）

- **作用**：第三方背書（Google 評論）+ 實體信任（地圖/門市位置）。台灣客戶很吃 Google 評分。
- **內容**：Google 評論輪播（星等+評論摘錄）+ 嵌入地圖（公司位置）+ 整體評分。
- **旅遊業最佳實踐**：評論要真實（接 Google Places API，不要自己編，否則信任反噬）；地圖用靜態圖+點擊展開（不要一進頁就載互動地圖拖慢速度）。
- **組件**：`google_content`（新組件，variant：評論輪播 / 評分摘要 + 地圖）。資料 = Google Places API（secrets 已有 `$GOOGLE_PLACES_API_KEY`）。
- **AI 輔助點**：① 從一堆評論中挑「最有代表性的 3-5 則」摘錄凸顯；② 把長評論精煉成一句不失真的摘錄（**要標示「摘錄自原評論」、不竄改原意**，避免造假爭議）。**AI 只做篩選與摘錄，不生成假評論。**

### 官網 8 區 → 組件總表

| William 的 8 區 | 對應組件（type） | 是否新組件 | 與現有 Canvas 積木關係 | AI 在這區的角色 |
|---|---|---|---|---|
| 1 Hero | `hero` | 新 | cover（C01）近親、共用 token | 挑圖 + 全站亮點凝練成副標 + 潤標題 |
| 2 搜尋 | `search_bar` | 新 | 無 | v2 自然語言搜尋（v1 傳統下拉） |
| 3 標題區 | `section_heading` | 新（輕量） | feature_hero（C09）輕量版 | 草擬 eyebrow + 引言 |
| 4 特色商品 | `featured_products` + `tour_card` | 新 | route_card（C04）卡片家族 | **亮點抓取主舞台**：生卡上亮點標籤 + hook |
| 5 商品列表 | `tour_listings` + `tour_card` | 既有規格已列 | — | 亮點 micro-badge + 排序提示 |
| 6 關於我們 | `about` | 既有規格已列 | spotlight（C13）氣質 | 草擬品牌故事 |
| 7 員工介紹 | `team` | 新 | 無（扣 `?ref=` 機制） | 草擬業務個人簡介 |
| 8 Google 內容 | `google_content` | 新 | 無 | 篩選+摘錄評論（不造假） |

> 既有 websites 規格列了 8 個 type（hero/header/about/services/tour_listings/testimonials/contact/footer）。
> 本框架建議調整為扣 William 8 區：保留 hero/header/about/tour_listings/footer；
> `services`→併入 about 或保留為選用；`testimonials`→升級為 `google_content`（真實 Google 評論勝過自填見證）；

## 四、亮點抓取機制（重點章節）

> William 最在意。核心問題：行程資料庫裡的「米其林餐廳、音樂劇、特殊體驗」這些亮點，怎麼**自動冒到 Hero / 本次特色區**？

### 4.1 關鍵發現（查過 production DB，不是猜的）

**漫途的亮點，大部分已經是「打勾/填好的結構化欄位」，不是埋在一段自由文字裡。** 這徹底改變了機制設計——

**餐廳（restaurants 表）已有的「亮點訊號」欄位**：
- `michelin_stars`（米其林星數 integer）、`michelin_guide_year`、`bib_gourmand`（必比登 boolean）、`michelin_plate`、`green_star`（綠星）
- `chef_name`（主廚）、`chef_profile`、`signature_dishes`（招牌菜 ARRAY）、`specialties`（特色 ARRAY）
- `awards`（得獎 ARRAY）、`rating`（評分）、`dining_style`、`highlights`（亮點 ARRAY）、`is_featured`（精選 boolean）

**飯店（hotels 表）已有的「亮點訊號」欄位**：
- `star_rating`（星級）、`hotel_class`、`has_michelin_restaurant`（boolean）
- `awards`（ARRAY）、`certifications`（ARRAY）、`highlights`（ARRAY）
- `butler_service` / `concierge_service` / `airport_transfer`（boolean 服務）、`amenities`（ARRAY）、`is_featured`

**景點（attractions 表）**：
- `category`、`type`、`tags`（ARRAY）——**注意：景點沒有 highlights 欄位，「音樂劇/特殊體驗」目前要靠 `category`/`type`/`tags` 判斷，或 `description` 文字**。這是唯一需要 AI 補文字判斷的地方。

**行程（itineraries 表）**：
- `features`（jsonb，已存特色）、`focus_cards`（jsonb，**已是「特色卡」結構**）、`highlights`（透過 daily 各日的 highlight）

### 4.2 建議機制：三層漏斗（結構化抓取 → AI 排序 → 人工釘選）

```
第 1 層【結構化抓取・規則引擎・100% 準】
  掃這團 daily_itinerary 的所有餐廳/飯店/景點 id，
  讀它們的「打勾欄位」，命中即記一條 highlight signal：
    餐廳 michelin_stars > 0      → 「米其林 N 星・餐廳名」
    餐廳 bib_gourmand = true     → 「米其林必比登・餐廳名」
    餐廳 chef_name 有值          → 「主廚 X 親自掌杓」
    餐廳 signature_dishes 有值   → 「招牌菜：…」
    飯店 star_rating >= 5        → 「五星・飯店名」
    飯店 has_michelin_restaurant → 「館內米其林餐廳」
    飯店 awards 有值             → 「得獎旅宿・飯店名」
    飯店 butler_service          → 「管家服務」
    景點 category/type/tags 命中 → 「特殊體驗・X」（音樂劇/秀/體驗）
  → 產出一個「亮點候選清單」（每條有 type/label/來源 id/權重）

第 2 層【AI 排序 + 潤色・低風險・可控】
  把候選清單丟給 AI，要它做三件事（不是生內容、是「挑+講」）：
    a. 排序：哪 3-5 個最該放 Hero / 本次特色（米其林 > 特殊體驗 > 五星 > 一般）
    b. 凝練：把「米其林一星・〇〇壽司」講成一句有感的 hook
    c. 去重：同類太多時合併（「3 晚五星飯店」而非列 3 條）
  → AI 只在「已驗證為真的候選」裡挑與潤，不會幻覺出不存在的米其林

第 3 層【人工釘選・業務最終把關】
  業務在展示行程編輯器 / 官網精選設定裡：
    - 看到自動抓出的亮點清單
    - 可手動「釘選」某個亮點到 Hero（is_highlight = true）
    - 可改 AI 潤的文案、可刪不想凸顯的、可加資料庫沒有的（如「含羽田直飛」）
  → 最終顯示 = 自動抓 ∪ 人工釘選，業務說了算
```

### 4.3 建議：AI 抽 vs 標記欄位 —— 兩者並用，但以「結構化欄位」為主

**直接回答 William 的問題**：

| 方案 | 適用範圍 | 建議 |
|---|---|---|
| **讀結構化欄位**（規則引擎） | 米其林/星級/得獎/主廚/服務——**這些都已是打勾欄位** | ✅ **主力**。100% 準、零幻覺、零 AI 成本、即時 |
| **AI 從文字抽** | 景點的「音樂劇/特殊體驗」——目前只在 category/type/tags/description | 🔶 **輔助**。只用在「沒有打勾欄位」的景點亮點，且要人複核 |
| **新增人工標記欄位** | 業務想釘選/補資料庫沒有的亮點 | ✅ **建議加，但輕量、非必填** |

**為什麼不純靠 AI 抽文字**：米其林是「事實」不是「感覺」，資料庫已經打勾了，再讓 AI 去一段文字裡抽，等於「明明有戶口名簿，卻派人去問鄰居這家有幾個小孩」——準確度反而下降、還可能幻覺出資料庫沒有的米其林（造假風險、商業信任反噬）。AI 該做它擅長的「在一堆真亮點裡挑最強的 3 個、用一句話講得動人」。

**為什麼還是要 AI**：① 景點的「音樂劇/特殊體驗」目前沒有專屬打勾欄位，需要 AI 從 category/type/tags/description 判斷（過渡方案，見 4.4 建議補欄位）；② 排序與文案凝練是 AI 的主場。

### 4.4 對現有 schema 的影響（給 William 拍板）

**建議 A（輕量、低風險、推薦做）——加「人工釘選亮點」能力**：

在 `itineraries` 表加（或用既有 `focus_cards` jsonb 擴充，**不一定要新欄位**）：
- 一個 `pinned_highlights` jsonb（業務手動釘選/補的亮點清單），或直接複用 `focus_cards`
- 影響：純加 jsonb 欄位、不破壞既有、idempotent migration、無資料風險

**建議 B（中等、解景點亮點缺口）——景點補「亮點訊號」欄位**：

`attractions` 表目前沒有 michelin 等級的結構化亮點欄位。若「音樂劇/秀/特殊體驗」要像米其林一樣準確抓取，建議補：
- `experience_type` text 或擴充 `tags`（如 'musical' / 'show' / 'unique_experience'）
- `is_signature_experience` boolean（這景點是不是「招牌體驗」）
- 影響：純加欄位 + 業務建檔時順手勾。**但這要業務願意維護資料**——若業務不勾，就退回 AI 從文字判斷（建議 A 的 AI 輔助層兜底）。

**建議 C（不建議現在做）——專門的 highlights 彙總表**：
- 不需要。亮點是「衍生資料」（從餐廳/飯店/景點算出來），不該另存一張表造成 SSOT 分裂（對齊紅線 E：同一真相不要兩個地方寫）。**亮點即時算 + 業務釘選存 jsonb 即可。**

**技術落點**：
- 亮點抓取邏輯放 `src/lib/canvas/extract-highlights.ts`（新檔，規則引擎），吃 enriched daily（`enrich-itinerary.ts` 已經把餐廳/飯店/景點補完，正好接上）。
- AI 排序/潤色：擴充既有 AI 助理（`applyAiPatch`），加一個 `highlight_curation` 動作（吃候選清單、回排序+文案 patch）。**對齊既有「AI 生 block JSON 不生 HTML」的共同前提。**
- 官網 Hero / 特色區與展示行程「本次特色」**共用同一個 extract-highlights**（SSOT，一處算、兩處用）。

> ⚠️ 紅線提醒：亮點抓取**只讀不寫核心業務資料**（餐廳/飯店是共享資料表，不改）；只在 `itineraries` 寫「釘選/文案 override」。寫入走 API + entity hook + apiMutate（紅線 F），不散刻。AI 文案要可被業務改、且標示來源，避免「AI 編出假米其林」的信任與法務風險。

## 五、AI + 組件化 + 輔助的協作分工（William 核心人格）

> 「AI + 組件化 + 輔助三者並存、系統不是一口氣用 AI 生成整個頁面」。
> 這章把這句話拆成**明確的職責邊界**，並對照現有 Canvas 15 積木 + 官網要新增的組件。

### 5.1 三者各做什麼（職責邊界，不可越界）

| 角色 | 做什麼 | **不做什麼**（界線） | 為什麼 |
|---|---|---|---|
| **組件化（積木）** | 預定義所有版型積木（官網 8 區組件 + Canvas 行程積木）。每塊有固定結構、morandi 樣式、防呆預設值。 | 不讓 AI 自由生 HTML/CSS、不讓 AI 發明新版型 | 視覺可控、CIS 一致、永遠不會醜。積木保底。 |
| **拖拉 / 選擇（人主導）** | 業務挑積木、排順序、填內容、釘選亮點、選圖。**人決定結構。** | 不需要從零寫 code、不需要懂設計 | 業務最懂這團的賣點與客戶，結構交給人最可控。 |
| **AI（副手 / 輔助）** | ①**抓亮點**（從結構化欄位匯總+排序）②**填內容**（草擬文案、補描述）③**潤飾**（改語氣、凝練 hook）④**配圖建議**（從圖庫挑/打分） | **不生整頁、不發明版型、不竄改事實（米其林/價格/退費）、不寫假評論** | AI 生整頁不可控、會翻車、會幻覺。AI 當副手「凸顯重點+填內容+潤飾+配圖」最穩。 |

**一句話**：**積木打底（不會醜）+ 人拖拉選擇（結構可控）+ AI 填血肉（內容有亮點）。AI 永遠是副手、永遠在「已驗證為真的素材」裡工作、業務永遠能改能否決。**

### 5.2 對照 canvas-editor 的 C→B→A 路徑

這完全吻合既有 canvas-editor 方向卡的結論（C→B→A 分階段）：
- **C（模板驅動 + AI 填充）= 現在的核心人格落地點**：選模板（組件化打底）→ AI 自動填+抓亮點（AI 輔助）→ 業務微調（拖拉）。**這就是「三者並存、非一口氣生成」。**
- **B（拖拉 + 側邊 AI）= 加強人主導**：開放拖積木、AI 局部增強。
- **A（AI 起草）= 未來**：AI 生整棵 block 樹當草稿，但**人仍用 B 的拖拉兜底**——即使到 A，也不是「一口氣生成不可改」，是「AI 起草、人精修」。

→ **官網 + 展示行程都吃同一套 Canvas JSON 地基**，C→B→A 一路長上去，不浪費工。

## 七、待 William 拍板的關鍵點

1. **亮點抓取主力是「讀結構化欄位」對嗎？**（我建議：是。米其林/星級/得獎已是打勾欄位、規則引擎抓 100% 準；AI 只做排序+潤色，不從文字硬抽，避免幻覺假米其林。）

2. **要不要加「人工釘選亮點」能力？**（我建議：要，但輕量——複用既有 `itineraries.focus_cards` jsonb 或加 `pinned_highlights`，純加欄位無風險。讓業務能釘選/補資料庫沒有的亮點如「羽田直飛」。）

3. **景點的「音樂劇/特殊體驗」要補結構化欄位嗎？**（景點表目前沒有 michelin 級的打勾欄位，靠 category/type/tags/description。選項：A 補 experience_type 欄位讓業務勾【準但要維護】；B 先用 AI 從文字判斷兜底【免維護但需複核】。我建議先 B 後 A。）

4. **官網 8 區的組件調整同意嗎？**（既有 websites 規格的 testimonials 升級為真實 Google 評論 `google_content`、新增 search_bar/section_heading/featured_products/team。)

5. **AI 的界線同意嗎？**（AI 只做：抓亮點+填內容+潤飾+配圖；不做：生整頁/發明版型/竄改事實/寫假評論。這是「非一口氣生成」核心人格的具體界線。）

6. **落地順序同意嗎？**（Phase 0 亮點地基 → Phase 1 展示行程補完 → Phase 2 官網 C 模式 → B → A。先做亮點，因為官網與行程都靠它。）
