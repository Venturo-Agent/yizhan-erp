# 設計方向提案 · Canvas 編輯器升級成「拖拉式建站 + AI 整合」

> 寫給 William 選方向用 · 方向卡 3 選 1
> 設計師：Venturo Design（漫途設計）
> 日期：2026-05-26
> 主題：把現有「展示行程編輯器」升級成「拖拉式建站編輯器 + AI 整合」、並為未來 websites 模組（多客戶官網系統）鋪路

---

## 0 · 先講結論（救護車式排序）

**這題的核心不是「要不要做拖拉」、而是「用戶主要靠誰把第一版生出來」。**

- 拖拉、AI、模板——三者都會在最終產品裡同時存在。
- 真正要選的、是「**進門第一步是什麼**」：對話描述（AI 起草）／空白畫布拖元件（人起草）／挑模板填資料（模板起草）。
- 這個第一步決定了整個編輯器的氣質、組件庫長相、AI 按鈕擺哪、以及——**要動多少現有 code**。

**三方向一句話**：

| 方向 | 一句話 | 第一步是 | AI 角色 | 工程量 |
|---|---|---|---|---|
| **A · AI 起草、拖拉微調** | 「你說、AI 畫、你再調」 | 對話描述 | 起草引擎（生整份 canvas） | 大 |
| **B · 拖拉為主、AI 側邊助手** | 「你蓋、AI 在旁邊遞工具」 | 拖元件到畫布 | 局部增強（潤稿 / 配色 / 補圖） | 中 |
| **C · 模板驅動、AI 換裝填充** | 「選樣板、AI 自動填、你微調」 | 挑模板 | 填充 + 套風格引擎 | 中小 |

**主持人建議（先講、細節在最後一節）**：
- **真選擇** = A 與 B（兩種不同的世界觀、都值得做）。
- **最保守、最快出貨** = C（且現有 code 已經做了 C 的一半）。
- **最有未來性** = A（AI 越強、A 的價值越被放大、扣 William 的遠見）。
- **我的拍板建議：分階段走 C → B → A**，理由見文末。

## 1 · 共同前提（三方向都吃這個、不可動搖）

> 這是已對齊的關鍵設計觀察、三張方向卡都建立在它上面。

**AI 不生 raw HTML、AI 生／改的是 Canvas 的 block JSON。**

白話：以前在本地請 Claude 寫「整頁 HTML」、那是一坨死字串、人改不動、拖拉也吃不進去。現在 AI 的產出要跟「人手動拖拉編輯」**吃同一個結構**（就是現有 `Canvas.sections[]` 那棵樹）。

這樣帶來三個好處：
1. **AI 生一版、人再拖拉微調**——因為兩邊都改同一份 JSON、無縫接力、不會「AI 一版、手動一版、兩套打架」。
2. **渲染只有一條路**——`CanvasRenderer` 那個 `switch(type)`、不管 block 是 AI 生的還是人拖的、長相完全一致。
3. **存檔、發布、版本快照全部免改**——`tour_display_overrides`（草稿 canvas + published_canvas 快照分離）那套照用。

技術語（放括號）：AI 的輸出 schema = 現有 `types.ts` 的 `CanvasSection` / `CanvasDayBlock` discriminated union。AI 回傳的不是文字、是「結構化的 patch / 整棵 sections 樹」、由前端 validate 過 schema 再塞進 state。

> ⚠️ 現況落差（重要、影響工程量估算）：
> 目前的 AI 助理（`AiAssistDialog` + `applyAiPatch`）**只會填既有欄位的文字**（封面副標、某天概述、附錄費用清單）、**不會生新 block、不會重排結構**。它的 `AiPatch.target` 只認 `cover` / `day_header` / `appendix` 三種、且都是改現有欄位。
> 也就是說——**「AI 生 block JSON」這個共同前提、現在的 code 還沒有**。三個方向各自要把這塊補到什麼程度、就是它們工程量的主要差異。

## 2 · 現有技術基礎盤點（不是從零做、這是底氣）

我讀過 code、確認以下都已存在、可重用：

| 已有的東西 | 在哪 | 三方向都重用？ |
|---|---|---|
| 結構化 Canvas JSON（typed 樹） | `src/components/canvas-renderer/types.ts` | ✅ 三方向共用、是地基 |
| `CanvasRenderer`（`switch(type)` 純渲染、exhaustive） | `src/components/canvas-renderer/CanvasRenderer.tsx` | ✅ 三方向共用 |
| 15+ 種 block 組件（封面 / 時間軸 / 景點卡 / 飯店 / 航班 / 餐廳 / spotlight / jp_note…） | `src/components/canvas-renderer/sections/` | ✅ 這就是現成的「組件庫」素材 |
| 全螢幕編輯器（toolbar + 主畫面即時渲染 + 右 panel） | `src/app/(main)/tours/[code]/display-editor/page.tsx` | ✅ 外殼共用 |
| 右側結構化 panel（樹狀清單 + 逐 block 表單） | `…/_components/EditorPanel.tsx` | ✅ A/B/C 都保留為「屬性面板」 |
| 不可變更新 helper（找 / 改 / 刪 block、永遠回新物件） | `…/_components/canvas-utils.ts` | ✅ 拖拉的「移動 / 插入」也走這層擴充 |
| 從團資料自動生 canvas（`buildCanvasFromTour`） | `src/lib/canvas/canvas-from-tour.ts` | ✅ **這就是方向 C 的引擎雛形** |
| 草稿 / 發布快照分離 + 公開頁 | `tour_display_overrides` + `/p/tour/[code]/canvas` | ✅ 三方向共用 |
| AI 對話框（掃空位 → 勾選 → 一次 call → 套用） | `…/_components/AiAssistDialog.tsx` | 🔶 A/B 要大改、C 小改 |

**白話總結**：地基（JSON + 渲染 + 存檔 + 公開頁）已蓋好。三方向都不動地基、差別在「**蓋哪一層樓、怎麼進門**」。

**現在缺的三塊**（決定工程量）：
1. **左側組件庫 + 拖拉到畫布**（B 必需、A 部分、C 輕量）——目前完全沒有、加 block 只能靠右鍵概念、且沒有 drag。
2. **畫布內直接選取 / 排序**（B 必需）——目前選取只能從右側樹點、畫布本身不可點不可拖。
3. **AI 能生整棵 / 多 block 結構**（A 必需）——目前只會填三種既有欄位的文字。

## 方向 A · AI 起草、拖拉微調

### 一句話定位
**像 Framer AI / v0——你用講的、AI 把整頁畫出來、你再拖拉修。** 拖拉是「修正筆」、AI 是「起草引擎」。

### 用戶主要操作流程
1. 業務進編輯器、看到的不是空白畫布、是一個**對話輸入框**（置中、大、像跟人講話）：「東京＋仙台六日、強調美食和溫泉、14 人小團、走精品低調風」。
2. AI 讀這段 + 團的結構資料（出發日 / 天數 / 已綁的景點飯店）→ **生出整棵 Canvas sections 樹**（封面文案、每日標題、景點描述、spotlight 選哪幾個重點、附錄費用）。
3. 主畫面**即時渲染出整份提案**（3-5 秒）、業務看到一個「已經有八成完成度」的版本。
4. 業務拖拉微調：把某張景點卡上移、刪掉一個 spotlight、換封面圖、點某段文字叫 AI「再寫感性一點」。
5. 滿意→發布。

### 組件庫 / 畫布 / 屬性面板
- **組件庫（左側）**：存在、但**收起來、次要**。預設摺疊成一條窄欄、需要時展開拖一個新 block 進去。因為 A 的主軸是「AI 已經幫你擺好了」、左庫只是補漏。
- **畫布（中間）**：即時渲染、每個 block 可點選（hover 浮出邊框）、可上下拖移排序。是主舞台。
- **屬性面板（右側）**：保留現有 `EditorPanel` 的逐 block 表單。但每個欄位旁多一顆「✦ 讓 AI 改寫」的小按鈕（不是 emoji、是 lucide 的 `Sparkles` 細線 icon）。

### AI 的角色與整合機制
- **AI 是起草引擎、坐在最前面。** 進門第一個畫面就是 AI 輸入框。
- **整合機制**：AI endpoint 回傳的是**完整 `Canvas.sections[]` 或一組 block patch**（不是文字、不是 HTML）。前端 validate schema → 直接 `setCanvas(next)` → Renderer 重畫。
- **AI 按鈕擺哪**：(a) 進門大輸入框（生整份）；(b) 每個 block hover 時浮出「叫 AI 改這塊」；(c) 屬性面板每欄位旁的小 `Sparkles`。
- **互動長相**：對話式、但**結果是視覺**——不是聊天室滾動、是「你說一句、畫面變一次」。可保留一條對話歷史側欄（「剛才的指令」、可回退）。

### 視覺感覺方向（編輯器本身的 UI 氣質）
沉穩、克制、像專業攝影師的後製軟體。深色 toolbar（沿用現有 `#2D1F18` 暖棕、不是科技黑）、主畫面留白大、AI 輸入框用 morandi-gold 細描邊聚焦、不要發光不要漸層。**絕不能做成「ChatGPT 包一層」那種藍紫對話泡泡感**——那是 AI slop。

### 對現有 Canvas 架構的技術影響：**大**
- **必做**：AI 能生「整棵樹 / 多 block patch」——要把現有 `AiPatch`（只認三種 target、只改文字）**升級成能 insert / reorder / 生整 section**。這是 prompt 工程 + schema validation + 一套「AI 回傳的不合法 block 怎麼降級」的防呆。
- **必做**：畫布內 hover 選取 + 拖移排序——畫布從「純展示」變「可互動」、要在 Renderer 外包一層 overlay（不污染 Renderer 本身、A/B 共用這層）。
- **必做**：對話歷史 / 回退 UI。
- **風險點**：AI 生的結構不可控（生出怪版型、塞錯天數）；要靠 schema validate + 「人最後一定看得到、改得動」兜底。

### 適合誰 / 什麼場景
不懂設計、要快、相信 AI 的業務。一個人要一天出五份不同團的提案、懶得從零排。**這是 SaaS sales pitch 最性感的版本**（「你用講的、官網自己長出來」）。

### 優點
- 進入門檻最低、業務上手最快。
- 最貼 William 遠見（UI 越來越值錢 = AI 起草能力越強越值錢）。
- Demo 最炸、最好賣。

### 風險
- 工程量最大、AI 不可控性最高（生壞版型 = 業務信任崩）。
- 「AI 生整份」如果品質不穩、業務反而要花更多時間收尾、體感變慢。
- 最依賴 AI API 成本與穩定度。

## 方向 B · 拖拉為主、AI 側邊助手

### 一句話定位
**像 Wix + AI / Notion AI——你自己蓋、AI 站在旁邊遞工具。** 拖拉是主幹、AI 是「順手幫個小忙」。

### 用戶主要操作流程
1. 業務進編輯器、看到**左側豐富組件庫**（封面 / 行程總覽 / 每日標題 / 景點卡 1-2-3 排 / 地圖 / 飯店卡 / 航班卡 / 餐廳卡 / spotlight / 日文注解 / CTA…）。
2. 從左邊**拖一個 block 到中間畫布**、放在想要的位置（拖放時畫布出現插入線）。
3. 點該 block → 右側屬性面板填內容（沿用現有逐 block 表單）。
4. 局部需要時、點該 block 上的「AI 幫忙」：「這段文案潤一下」「幫這天配個適合的重點」「補張圖」。AI 只動**這一塊**、不碰其他。
5. 一塊一塊蓋完→發布。

### 組件庫 / 畫布 / 屬性面板
- **組件庫（左側）**：主角、永遠展開。分類清楚（封面類 / 行程類 / 住宿交通類 / 特色介紹類 / 結尾類）。每個 block 有縮圖預覽（不是文字清單、是看得到長相的小卡）。
- **畫布（中間）**：可拖放接收區、block 間有插入線、選取後可拖移 / 複製 / 刪除。是主操作區。
- **屬性面板（右側）**：現有 `EditorPanel` 表單、**幾乎照用**、只是改「block 怎麼被選到」（從畫布點、不只從樹點）。

### AI 的角色與整合機制
- **AI 是側邊助手、局部增強。** 不生整份、只在「你選中某塊、明確叫它」時出手。
- **整合機制**：AI 回傳**單一 block 或單一欄位的 patch**（吃現有 `applyAiPatch` 的擴充版）。範圍小、可控、validate 簡單。
- **AI 按鈕擺哪**：(a) 選中 block 時、block 工具列浮出「✦ AI」；(b) 屬性面板欄位旁小按鈕。**沒有**進門大輸入框（那是 A 的）。
- **互動長相**：右側滑出一個窄 AI 小窗、給 2-3 個快捷動作（潤稿 / 換語氣 / 補圖）+ 自由輸入。動完即時預覽、接受 / 退回。

### 視覺感覺方向
工匠的工作台。左庫像「材料架」、中畫布像「桌面」、右面板像「工具盒」。沿用 morandi、左庫卡片用 `morandi-container` 底、選中態 `morandi-gold` 描邊。氣質是「掌控感、踏實」、不是「魔法感」。

### 對現有 Canvas 架構的技術影響：**中**
- **必做**：左側組件庫 UI + 每個 block 的「空白範本」工廠（拖出來時給一個合理預設 data）。
- **必做**：畫布拖放排序 + 插入線——同 A 的畫布 overlay 層。
- **小改**：AI 只生單 block / 單欄位、`applyAiPatch` 擴充幅度小、最可控。
- **幾乎不改**：右側屬性面板、存檔、發布、渲染全照用。
- **風險點**：拖放 UX 細節多（手機怎麼拖？插入位置判定？）、但都是前端互動工程、不碰資料正確性。

### 適合誰 / 什麼場景
想自己掌控、對成品有審美要求、不放心全交給 AI 的業務 / 設計型員工。也適合「客戶官網」這種需要精雕的場景（websites 模組）。

### 優點
- AI 範圍小最可控、品質最穩、不會「AI 生壞整份」。
- 工程量中等、且**最大化重用現有屬性面板**。
- 拖拉是經典熟悉的心智模型、教育成本低。

### 風險
- 「從零拖」對不懂設計的業務仍有門檻（白畫布恐懼）。
- Demo 不如 A 性感（沒有「一句話變官網」的魔法時刻）。
- 拖放互動工程瑣碎、要打磨。

## 方向 C · 模板驅動、AI 換裝填充

### 一句話定位
**像 Canva 套模板——選一個漫途精美樣板、AI 自動把這團的資料填進去、你輕量微調。** AI 是「填充 + 套風格」引擎、拖拉只用於輕量調整。

### 用戶主要操作流程
1. 業務進編輯器、先看到**模板選擇畫面**：漫途預設 4-6 套精美版型（精品低調 / 攝影大圖 / 雜誌編排 / 親子活潑…）、每套有縮圖。
2. 選一套→ AI（其實大半是 `buildCanvasFromTour` 的升級版）**自動把這團的行程 / 飯店 / 圖 / 餐廳填進模板的對應 slot** + 套該模板的風格 token。
3. 主畫面渲染出「已經填好的整份」（這步現有 code 已經做到八成）。
4. 業務輕量微調：換區塊順序、改幾句文案、換封面圖、刪一兩塊不要的。
5. 發布。

### 組件庫 / 畫布 / 屬性面板
- **組件庫（左側）**：**不是主角**。預設不顯示、或只給「加一塊」的精簡清單。因為 C 的哲學是「模板已經幫你決定好結構、你不該從零拼」。
- **畫布（中間）**：渲染填好的模板、可上下拖移排序、可刪、可點選編內容。**不能拖入任意新 block 重組**（那會破壞模板的設計完整性、刻意限制）。
- **屬性面板（右側）**：現有 `EditorPanel` 表單照用。
- **多一個「模板切換」入口**：toolbar 上可隨時「換一套模板」、資料自動重填進新模板。

### AI 的角色與整合機制
- **AI 是填充 + 套風格引擎。** 把「團資料 → 模板 slot」做智慧映射（哪個景點當封面圖？哪一餐值得做 spotlight？哪天該 bleed 底色？）+ 生補充文案。
- **整合機制**：模板 = 一個「帶 slot 標記的 Canvas 骨架」。AI / 規則引擎把團資料填進 slot、輸出標準 `Canvas.sections[]`。**這本質是現有 `buildCanvasFromTour` 的多模板、加 AI 文案、加風格 token 版本。**
- **AI 按鈕擺哪**：(a) 選模板後自動跑（不用按）；(b) 個別欄位旁「✦ 重寫」小按鈕（沿用現有 `AiAssistDialog` 的「掃空位 → 生文案」邏輯、幾乎不改）。
- **互動長相**：最少對話、最多「自動發生」。業務感覺是「我選了樣式、東西自己長好了」。

### 視覺感覺方向
策展人選相框。模板選擇畫面像精品圖庫（大縮圖、留白、morandi 中性底）。編輯態氣質低調、強調「模板本身夠美、你只要微調」。風格 token 化（每套模板 = 一組 `tokens.ts` 變體）、是漫途 CIS 最容易守住一致性的方向。

### 對現有 Canvas 架構的技術影響：**中小（最接近現況）**
- **必做**：把 `buildCanvasFromTour` 從「一種版型」擴成「多模板、可選」——抽出「模板定義」概念（slot + 風格 token 變體）。
- **必做**：`tokens.ts` 從單一 Yongcheng 風、擴成多套主題變體（現在 `Canvas.theme` 已經是 `'classic'`、就是為這個留的伏筆）。
- **小改**：AI 文案沿用現有 `AiAssistDialog`、擴充覆蓋的欄位即可。
- **不需要**：複雜拖放、不需要 AI 生整棵樹（規則引擎填 slot 為主、AI 只補文案）。
- **風險點**：模板做得不夠多 / 不夠美 = 大家長一樣（這是設計工作量、不是工程風險）。

### 適合誰 / 什麼場景
要快、要美、要規模化的場景——一個業務一天出十份團、每份都要好看但沒時間精雕。最適合「大量團快速產官網」。

### 優點
- **離現況最近、最快出貨**（`buildCanvasFromTour` 已是雛形）。
- 成品美感下限最高（模板保底、不會醜）。
- CIS 一致性最好守（風格 token 化、客戶官網不會跑掉品牌）。
- AI 範圍最小最穩。

### 風險
- 自由度最低、進階用戶會嫌「綁手綁腳」。
- 模板數量 / 品質決定天花板、是持續的設計投入。
- 沒有「拖拉建站」的爽感與賣點、Demo 平淡。

---

## 3 · 主持人建議

### 哪個是真選擇、哪個保守、哪個最有未來

- **C 是保守、保險、最快**。它幾乎是把現有 `buildCanvasFromTour` + `AiAssistDialog` 補強成「多模板」、工程風險最低、成品下限最高。**先出貨、先讓業務有得用、先驗證市場**、C 是最穩的第一步。
- **A 與 B 是真正的兩種世界觀**——A 是「AI 主導、人修正」、B 是「人主導、AI 輔助」。這兩個不是「做不做拖拉」的差別、是「**進門第一步交給誰**」的根本分歧。
- **A 最有未來性**。扣 William 的遠見：AI 越強、「一句話生整份官網」的價值越被放大。但它工程量最大、AI 不可控性最高、現在硬上風險高。

### 建議路徑：分階段 C → B → A

**為什麼不一次到位 A**：A 的核心（AI 生整棵合法的 block 樹）目前 code 完全沒有、且品質不穩會反噬信任。在沒有「人能輕鬆兜底修正」的工具前、AI 起草是空中樓閣。

**為什麼這個順序**：

1. **Phase 1 · 先做 C（模板驅動）**——最快出貨、複用最多現有 code、讓業務馬上有「會自動長好的官網」。同時逼我們把 `tokens.ts` 主題變體、模板 slot 概念建起來——**這套東西 A 和 B 之後都要用**。
2. **Phase 2 · 疊 B（拖拉 + 側邊 AI）**——在 C 的模板基礎上、開放「拖元件微調」+「選中塊叫 AI 局部增強」。這步把「畫布可互動 overlay」+「左組件庫」+「AI 改單 block」建起來——**這三樣是 A 的前置零件**。
3. **Phase 3 · 升級到 A（AI 起草）**——零件齊了（畫布可拖、組件庫在、AI 能生 block、人能輕鬆修正兜底）、才把 AI 推到最前面、開「一句話生整份」。此時 AI 生壞了、業務也能用 B 的拖拉秒修、信任不崩。

**一句話**：C 讓你「有得賣」、B 讓你「賣得專業」、A 讓你「賣得性感」。三層疊加、每層都用上一層的零件、不浪費工。

> 對 websites 模組（多客戶官網系統）的意義：這條路徑天然支援它——C 的「模板 + 風格 token 變體」就是「不同客戶不同 CIS」的基礎、B 的拖拉是客戶官網精雕的工具、A 是未來「客戶自己用講的生官網」的賣點。同一套 Canvas JSON 地基、三階段一路長上去。

## 4 · 顧問圓桌視角摘要

> 六個顧問各從自己角度檢視這題（純檢視、不畫設計）。

- **🔍 UX Researcher（受眾痛點）**：旅遊業務的真實痛點是「時間碎、團多、要快要美但不會排版」。最痛的不是「沒有拖拉工具」、是「面對白畫布不知從何開始」。這指向 **C / A**（給起點）優於純 B（給空白）。B 若獨立做、要解決「白畫布恐懼」。

- **🏗️ UX Architect（資訊密度與互動模式）**：三方向的互動模式根本不同——A 是「對話→視覺」（低密度入口、高魔法）、B 是「三欄工作台」（高密度、高掌控）、C 是「選擇→自動→微調」（漸進揭露）。現有右側 `EditorPanel`（樹 + 表單）三方向都該保留為「屬性面板」、它是已驗證的可控編輯模式、別丟。

- **🎯 Brand Guardian（競品站位）**：Wix/Framer/v0 全在「炫技、漸層、魔法」那一側。漫途要站對面——**克制、編輯感、像精品提案軟體而非建站玩具**。C 的「模板 + 風格 token」最能守住客戶 CIS 不跑掉。三方向都必須拒絕「AI slop 視覺」（漸層紫、發光、聊天泡泡）。

- **✨ Whimsy（驚喜元素）**：驚喜時刻在 A 最強（「我講一句、整頁長出來」是 wow moment）。但驚喜要克制——別做煙火動畫、做「填字機打字般、文案一行行浮現」「block 落位時輕微的紙張 settle 感」這種有質感的微動效。Whimsy 服務於「專業感」、不是「玩具感」。

- **♿ Inclusive（a11y / 文化）**：拖拉互動（B/A）對鍵盤 / 觸控 / 輔助技術不友善——必須提供「不靠拖拉也能排序」的替代路徑（右面板的上移 / 下移按鈕、就是現有樹狀清單可擴充的）。AI 生成內容要避免文化刻板（「日文注解」block 已是好的文化細節範例、延續這種尊重）。對話式 A 對不善表達需求的用戶反而是門檻、需引導式提示。

- **📖 Visual Storyteller（品牌情感）**：旅遊提案賣的是「對一趟旅程的嚮往」。編輯器本身的氣質會滲進產出——若編輯器像冰冷的科技工具、業務做出來的東西也冷。三方向的編輯器都該帶**溫暖、策展、有人味**的氣質（沿用 `#2D1F18` 暖棕 toolbar + morandi-gold + 宋體標題、就是對的方向）。讓業務在「美的工具」裡工作、他才做得出「美的提案」。
