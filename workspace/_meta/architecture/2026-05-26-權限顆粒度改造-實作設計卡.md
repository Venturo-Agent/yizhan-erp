## 0. 一句話 + 定案速查

把「誰能看到、改到哪些資料」從現在的**兩檔**（只看自己分公司 / cross_branch.read 全看）升級成 **集團 ⊃ 分公司 ⊃ 部門 ⊃ 個人 四階**。範圍**綁職務**、同時管看與改、跟組織複雜度自動長。

| # | 定案 |
|---|---|
| 1 | 「能看什麼功能」(capability) 與「能看多大範圍」(scope) 拆兩個獨立維度 |
| 2 | 範圍綁**職務**、不綁每個 capability |
| 3 | 四階：group ⊃ branch ⊃ department ⊃ self |
| 4 | 一個範圍**同時管看與改/刪**；廢 `can_write_others` |
| 5 | 五關階梯，「建立人/負責業務永遠看得到自己的」是保底線 |
| 6 | 部門這輪完整蓋（表+員工欄位+UI+RLS） |
| 7 | 業務資料帶 department_id、預設總部；分部門後歷史資料留總部 |
| 8 | 一人只隸屬一間分公司（單值 branch_id） |
| 9 | 職務組跟組織複雜度長：建分公司補「分公司負責人」、建部門補「部門主管」 |
| 10 | seed 走租戶開通**程式流程**、不走 DB trigger（紅線 E） |
| 11 | 共用資料（設定/主檔）不吃範圍；否決全局模板庫（紅線 #0） |
| 12 | 每個預設職務 = 名字 + 預配好的 capability + 範圍，開箱即用 |
| 13 | 最小權限鐵則：預設職務取「能完成工作的最小範圍」，只有系統管理員是 group |
| 14 | 個人級「自己」：團認 controller_id、訂單認 sales_id、其餘 created_by；**負責欄位可留空、空值 fallback created_by、不強制必填** |
| 15 | 附屬明細（子表）跟主單走（inherited RLS），不自己判斷範圍 |

## 1. 範圍模型

### 四階階梯（外層涵蓋內層）

```
group       集團級    所有分公司、跨部門
  branch    分公司級  只看自己所屬分公司（跨部門、限本店）
    department 部門級  只看自己部門
      self  個人級    只看自己經手的
```

### 五關階梯（單一 row 可見性判斷，依序）

```
① row 的歸屬人 = 我？              → 看得到（保底線、永遠成立）
② 我的職務 scope = group？         → 看得到
③ 我的職務 scope = branch？        → row.branch_id 為空 或 = 我的 branch → 看得到
④ 我的職務 scope = department？    → row.department_id 為空 或 = 我的 department → 看得到
⑤ 我的職務 scope = self？          → 只剩第①關
```

- 「歸屬人」依表而定：`tours` = `COALESCE(controller_id, created_by)`、`orders` = `COALESCE(sales_id, created_by)`、其餘 = `created_by`。
- 「為空 → 視為總部/全店資料、放行」沿用現有 `scope_branch_match()` 的退化精神，延伸到 department。
- 寫（INSERT/UPDATE/DELETE）走同一判斷（定案 4）；功能權限（能不能寫）由 API 層 `requireCapability` 把關。

### 1.1 兩種「範圍」：看的範圍 vs 選人的範圍（William 2026-05-26 提問釐清）

- **A 看的範圍**（本卡主體、綁職務 data_scope）：一個人能看到/改到哪些 row。
- **B 選人的範圍**（建單時人員下拉該列誰）：建某分公司的單時，「團控/負責業務」選單該只列該分公司的員工。

兩者獨立但需對齊。**現況（探子查證）：B 沒做** — 建團/建訂單的人員下拉用 `useEmployeesWithCapability`，只按 capability 過濾、不按 branch，會跑出別分公司的人；`useEmployeesSlim` 的 slim select 連 branch_id 都沒撈。

**目標**：人員選單預設只列「這張單所屬分公司」的員工、附「顯示其他分公司」選項供跨分公司調人支援（例外）。

**要改的檔（歸 Phase 4）**：
- `src/data/entities/employees.ts` — slim select 補 `branch_id, accessible_branch_ids`
- `src/lib/permissions/useEmployeesWithCapability.ts` — 加 `currentBranchOnly?` / `branchId?` 參數 + 過濾
- `src/app/(main)/tours/_components/tour-form/TourBasicInfo.tsx`（controller 選單，`:48`）
- `src/app/(main)/orders/_components/add-order-form.tsx`（`:42`）+ `order-edit-dialog.tsx`（`:57`）（sales 選單）

## 2. Schema 改動（Phase 1）

### 2.1 新建 `departments` 表

```sql
CREATE TABLE public.departments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id uuid NOT NULL REFERENCES public.workspaces(id) ON DELETE CASCADE,
  code text NOT NULL,
  name text NOT NULL,
  is_default boolean NOT NULL DEFAULT false,   -- 「總部」那筆 = true
  is_active boolean NOT NULL DEFAULT true,
  display_order integer NOT NULL DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
```

## 3. 守門邏輯（RLS，Phase 2，🔴 最高風險）

### 3.1 新核心函數 `can_see_row()`

```sql
CREATE OR REPLACE FUNCTION public.can_see_row(
  p_branch uuid, p_department uuid, p_owner uuid
) RETURNS boolean
LANGUAGE plpgsql STABLE SECURITY DEFINER SET search_path TO 'public','pg_temp'
AS $$
DECLARE
  v_me uuid; v_ws uuid; v_branch uuid; v_dept uuid; v_scope text;
BEGIN
  SELECT e.id, e.workspace_id, e.branch_id, e.department_id,
         COALESCE(r.data_scope, 'self')
    INTO v_me, v_ws, v_branch, v_dept, v_scope
  FROM public.employees e
  LEFT JOIN public.workspace_roles r ON r.id = e.role_id
  WHERE e.user_id = auth.uid() LIMIT 1;

  IF v_me IS NULL THEN RETURN false; END IF;
  IF p_owner = v_me THEN RETURN true; END IF;            -- ① 保底
  IF v_scope = 'group' THEN RETURN true; END IF;          -- ②
  IF v_scope = 'branch' THEN
    RETURN p_branch IS NULL OR p_branch = v_branch;        -- ③
  END IF;
  IF v_scope = 'department' THEN
    RETURN p_department IS NULL OR p_department = v_dept;  -- ④
  END IF;
  RETURN false;                                           -- ⑤ self：只剩①
END; $$;
```
- 注意：workspace 隔離仍在各表 policy 第一段 `workspace_id = get_current_user_workspace()` 把關，`can_see_row` 不重複查 workspace（效能）。

### 3.2 主表 policy 樣板

```sql
-- 以 tours 為例（歸屬人 = controller 優先）
USING (
  workspace_id = get_current_user_workspace()
  AND can_see_row(branch_id, department_id, COALESCE(controller_id, created_by))
)
-- orders：COALESCE(sales_id, created_by)
-- 其餘 32 表主表：created_by
```
- WITH CHECK 同 USING（範圍同時管看與改）。

### 3.3 子表走 inherited

- `order_members` / `*_items` / `receipt_invoice_allocations` 用 `setup_inherited_rls(child, parent, parent_fk)`——可見性 = 父單可見性，不自己判斷 branch/department。

### 3.4 哪些表套什麼

| 類別 | 表 | RLS 規則 |
|---|---|---|
| 主業務表（有歸屬人） | tours / orders / customers / receipts / payment_requests / quotes / invoices / contracts / journal_vouchers / salary_settlements / disbursement_orders … | `can_see_row(branch, dept, owner)` |
| 子表 | *_items / order_members / receipt_invoice_allocations | inherited（跟父單） |
| 共用設定/主檔（71 張） | branches / brands / departments / bank_accounts / payment_methods / chart_of_accounts / workspace_* / countries … | `setup_workspace_scoped_rls`（純 workspace、不吃範圍） |

### 3.5 遷移舊的 `cross_branch.*` 機制

- 現有靠 `cross_branch.read` 的角色 → backfill `data_scope = 'group'`（§5.2）。
- code 層改：`tours` 列表（`useToursPaginated`）、`orders` 頁的「看全部 vs 看自己」切換，從「有沒有 cross_branch.read」改讀職務 `data_scope`（Phase 4）。
- `cross_branch.read/write`、`cross_department.read/write`（後者 code 0 reference）capability 於 Phase 5 清理。

## 4. Seed 職務組（Phase 3）

### 4.1 預設 6 職務 + 範圍

| 職務 | data_scope | 定位 |
|---|---|---|
| 系統管理員 | **group** | 全功能全範圍（唯一 group） |
| 會計 | **branch** | 財務/會計全權、查帳唯讀 |
| 團控 | **branch** | 團務/訂單操作、財務唯讀 |
| 業務 | **self** | 自己的團/訂單/客戶 |
| 分公司負責人（建分公司時補） | **branch** | 分公司內全業務 |
| 部門主管（建部門時補） | **department** | 部門內全業務 |

> **實際上線（2026-05，角落／御風兩間、皆有分公司）**：先各建「系統管理員(group)」＋「分公司負責人(branch)」兩職務、把現有人員掛進去，其餘職務（會計／團控／業務／部門主管）作範本、之後由系統管理員按需自建/調整。先求能跑、再細調。

### 4.2 各職務 capability 包（核心 module；加值 module 隨 feature 開通另給）

> 表示法：RW = read+write、R = read only、— = 無。對到真實 capability code（`<tab>.read` / `<tab>.write`）。

**系統管理員**：全部 module 全 RW（含 hr.roles、settings.company、workspaces 若該 workspace 開 tenants feature）。

**會計**（branch）
- `finance.*` 全 RW（payments / payments-confirm / requests / disbursement / reports / settings）
- `accounting.*` 全 RW（vouchers / accounts / period-closing / opening-balances / checks / reports）
- `tours.overview` R、`tours.orders` R、`orders.list` R（查帳）
- `database.customers` R、`database.suppliers` R
- `hr_salary_settlement` RW、`finance.requests-salary` RW
- `settings.company` R ⚠️ **待拆細**：只該碰「財務設定」（銀行帳戶/付款方式/科目），不碰系統設定（品牌/分公司/feature）
- `channels.*` RW、`calendar` R

**團控**（branch）
- `tours.*` 全 RW（overview/orders/members/itinerary/quote/contract/closing/template）
- `orders.list` R、`orders.create`/`orders.edit` 視業務（草案：訂單由業務開、團控 R）
- `database.customers` RW、`database.suppliers` R
- `finance.payments` R、`finance.requests` R（看團的收支）
- `channels.*` RW、`calendar` RW

**業務**（self）
- `tours.overview/orders/members/itinerary/quote` RW、`tours.contract` R
- `orders.*` RW（list/create/edit/payments/travelers）
- `database.customers` RW
- `finance.payments` R、`finance.requests` R（自己經手）
- `channels.*` RW、`calendar` RW

**分公司負責人**（branch）≈ 團控 + 會計查閱權，範圍 branch（看本分公司全部）
- `tours.*` RW、`orders.*` RW、`database.*` RW
- `finance.*` R（出納/報表查閱）、`accounting.reports` R
- `hr.employees` R、`channels.*` RW、`calendar` RW、`settings.company` R

**部門主管**（department）≈ 業務 + 部門內查閱
- `tours.*` RW、`orders.*` RW、`database.customers` RW
- `finance.payments` R、`finance.requests` R、`hr.employees` R
- `channels.*` RW、`calendar` RW

> ⚠️ 以上為草案，最終以 William 在視覺矩陣的調整為準（待最終確認，§8）。

### 4.3 動態補職務（建分公司/部門時）

- 觸發：分公司管理頁建第一間「非預設」分公司 → 前端提示「要不要建『分公司負責人』職務（權限已配好）？」→ 確認 → 呼叫 onboarding API（`POST /api/roles/seed-preset { preset: 'branch_manager' }`）→ INSERT workspace_roles + role_capabilities。
- 部門同理（建第一個非總部部門 → 補「部門主管」）。
- **走 API、不走 DB trigger**（紅線 E）。preset 定義集中在一個 server module（如 `src/lib/permissions/role-presets.ts`），seed migration 與動態補職務共用同一份 preset 真相。

## 5. 資料遷移（Phase 5）

### 5.1 backfill department_id
- 每 workspace 建「總部」departments 筆；所有現有 employees + 32 張業務表的 department_id 指向總部。

### 5.2 backfill workspace_roles.data_scope
- 有 `cross_branch.read` capability 的角色 → `group`
- 名稱含「系統管理員 / admin」(is_admin=true) → `group`
- 其餘 → 保守設 `branch`（避免一上線就把同事資料藏起來造成業務中斷；之後客戶自行收緊到 department/self）
- ⚠️ 這是「上線不破壞現狀」的折衷：現狀其實多是 branch 級可見，先對齊 branch、再讓客戶按需收緊。

### 5.3 清理
- DROP `role_capabilities.can_write_others`、`employees.accessible_department_ids`
- 廢 `cross_branch.*` / `cross_department.*` capability（確認 code 改完）
