## 為什麼要建測試環境

過去 production-only 開發方式有四個痛點：

| 痛點                       | 真實 case                                                                                                                                       |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **動 schema 怕死客戶資料** | 5/14 加 trigger `trg_workspaces_onboarding_seed` + 5/10 寫的 `createDimensions()` API 雙寫撞 unique → 自 5/14 起無人能成功建租戶（紅線 E 由來） |
| **改 RLS 全員登不進去**    | 4/20 動 workspaces RLS 加 FORCE → 全員登入失敗（紅線 A 由來）                                                                                   |
| **驗證流程要清真實資料**   | 跨租戶滲透測試只能在 prod 跑、危險                                                                                                              |
| **永豐金流上線前要對接**   | 永豐有 UAT 沙盒、production 不能拿真實卡測                                                                                                      |

## 三層環境架構

```
┌──────────────────────────────────────────────────────┐
│ Layer 1: 本機開發（每個工程師 / Alex / Logan / ...）  │
│ - supabase start（本機 docker、port 54321/54322）     │
│ - .env.development.local                              │
│ - Mac 開機跑、改 schema 試錯、reset 不影響任何人       │
└──────────────────────────────────────────────────────┘
                       ↓ commit / PR
┌──────────────────────────────────────────────────────┐
│ Layer 2: Supabase Branch（PR 預演、Pre-merge gate）   │
│ - mcp__supabase-aierp__create_branch 從 main fork    │
│ - 跑 migration 預演、CI 用、merge 完自動 cleanup       │
│ - Snapshot prod schema、可選 anonymize prod 資料      │
│ - 永豐 UAT 對接、發測試卡單                            │
└──────────────────────────────────────────────────────┘
                       ↓ merge to main
┌──────────────────────────────────────────────────────┐
│ Layer 3: Production（erp.venturo.tw）                 │
│ - mcp__supabase-aierp__apply_migration                │
│ - 真實客戶資料、真實金流                                │
│ - 監控 + 報警走 Sentry + Grafana（將來）              │
└──────────────────────────────────────────────────────┘
```

### 第四層（可選 / 暫不做）

| Layer                                                 | 用途                                | 何時需要                          |
| ----------------------------------------------------- | ----------------------------------- | --------------------------------- |
| **獨立 staging project**（譬如 `yizhan-erp-staging`） | QA / 客戶 demo / 永久存活的測試環境 | 客戶 demo 頻率高 / 第三方滲透測試 |

## Layer 1：本機 Supabase（首要）

### 為什麼

- **免費**：完全本機 docker、不計入 Supabase 月費
- **快**：改 schema 秒重置、不卡 cloud round trip
- **隔離**：跟其他工程師 / production 完全脫鉤

### 安裝 / 啟動

```bash
# 一次安裝
brew install supabase/tap/supabase  # 已裝可跳過

# 在專案根目錄
cd ~/Projects/yizhan-erp
supabase start
```

`supabase start` 會：

- 跑一個 PostgreSQL container（port 54322）
- 跑 Supabase Studio（http://localhost:54323）
- 跑 Auth / Storage / Realtime / Edge Functions

### env 切換

新建 `.env.development.local`（不進 git）：

```bash
NEXT_PUBLIC_SUPABASE_URL=http://localhost:54321
NEXT_PUBLIC_SUPABASE_ANON_KEY=<本機 anon key、supabase start 輸出時印>
SUPABASE_SERVICE_ROLE_KEY=<本機 service_role key>
```

`npm run dev` 自動讀 `.env.development.local` 優先於 `.env.local`。

### 初始 seed

本機 supabase 起來後是空的、需要 seed：

```bash
# 跑所有 migration
supabase db reset

# 補 seed data（建假 workspace / 假員工 / 假帳單）
npm run seed:local  # 需寫這個 script、目前還沒
```

**待做：寫 `scripts/seed-local.ts`**

- 建 1 個假 workspace（譬如 `LOCAL_TEST`）

## Layer 2：Supabase Branch（PR 預演）

### 為什麼

- **生產級 schema 預演**：跑 migration 時跟 prod 一模一樣（同個 Postgres / 同個版本）
- **CI 整合**：PR 動 schema → CI 自動 create branch → 跑 migration → 過了才允許 merge
- **資料隔離**：branch 有自己的 DB、不會撞 prod
- **永豐 UAT 對接點**：在 branch 對接 `eposuat.sinopac.com`、用測試卡跑 end-to-end

### MCP 工具

```typescript
mcp__supabase - aierp__create_branch({ name: 'pr-123-sinopac-integration' })
mcp__supabase - aierp__list_branches()
mcp__supabase - aierp__merge_branch({ branch_id: '...' })
mcp__supabase - aierp__delete_branch({ branch_id: '...' })
```

### CI workflow（要寫）

`.github/workflows/pr-supabase-branch.yml`：

```yaml
name: PR Supabase Branch Migration Check
on:
  pull_request:
    paths:
      - 'supabase/migrations/**'

jobs:
  branch-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Create Supabase branch
        run: |
          # 用 MCP 或 CLI 建 branch
          # apply 所有 new migration
          # 跑 npm run audit:rls
          # pass 才 comment「safe to merge」
```

### 限制

- Branch **有時間限制**（Supabase 預設 14 天後自動清）
- Free tier 同時只能存在 1 個 branch
- 每次建 branch 要 30-90 秒
- 適合「動 schema」型 PR、純 code 改動可跳過

## 風險 / 限制

1. **Supabase branch 14 天時效**：long-lived PR 要每 7 天 force-refresh branch
2. **Free tier branch 同時只 1 個**：多人同時動 schema 要排隊
3. **本機 docker 吃資源**：每個工程師 +1GB RAM / 2GB disk
4. **永豐 UAT 特店號取得時間**：依永豐業務速度、可能 1-4 週
5. **seed 不能含真實客戶資料**：避免 GDPR 風險、用假名 / generated data
