## 一句話

DB 端寫入只要 10-25 ms、慢的是「API 內串了 5-6 次 sequential 等待 + 1 次外部 Auth 驗證」、修兩條改造全網站 108 個 API 同時受惠、每個按鈕快 80-150 ms。

## 為什麼之前沒抓到（誠實檢討）

不是誰偷懶、是**整套 audit / 紅線制度根本沒有「效能」這條維度**。

### 既有 audit 在量什麼

| Audit                      | 量什麼                                          | 量效能嗎 |
| -------------------------- | ----------------------------------------------- | -------- |
| `audit:rls`                | 6 層架構偏離、workspaces FORCE、散刻 capability | ❌       |
| `audit:writes`             | DB trigger × API 雙寫撞號                       | ❌       |
| `audit:orphans`            | auth.users 孤兒                                 | ❌       |
| `audit:realtime`           | entity hook 訂閱但 publication 沒加             | ❌       |
| 5 維度健檢「讀取效能」維度 | 有沒有走 entity hook（**結構**對不對）          | ❌       |
| 夜間自動健檢               | 跑上面 9 個 audit、報結構偏離                   | ❌       |

**所有 audit 量的都是「正確性」**、沒有任何一個量「實際多快」。

### 紅線清單漏洞

```
紅線 0: 沒超級管理員（資安）
紅線 A: workspaces 不 FORCE（資安）
紅線 B: FK 指 employees（正確性）
紅線 C: admin client per-request（資安）
紅線 D: 不開作弊後門（會計正確性）
紅線 E: 同表寫入 SSOT（正確性）
紅線 F: client 走 entity hook（正確性）
紅線 G: SWR cache per-user key（資安）
紅線 H: 業務表 RLS 過 workspace（資安）
```

全是事故驅動：4/20 RLS 事故 → 紅線 A、5/14 撞號 → 紅線 E、5/17 跨帳號 cache → 紅線 G、5/21 expense_categories → 紅線 H。

**全是「資安 / 正確性」事故、效能沒事故、就沒紅線、就沒 audit、就沒人 review**。設計慣性一路累加、108 個 API 全照同套寫、沒人停下來問「這樣會不會太慢」。

直到 2026-05-23 William 抱怨「傳送中很慢」、才第一次有人翻 pg_stat_statements。

## 三個 leverage 最大的改造

### 🥇 改造 1：JWT 本地驗證取代 `supabase.auth.getUser()`

#### 現況（`src/lib/auth/server-auth.ts`）

```ts
const {
  data: { user },
  error,
} = await supabase.auth.getUser()
//                                          ↑ 打 Supabase GoTrue server、外部 HTTP 30-80ms
```

每個 API、每個 server action、每次 SSR、都打一次外部 GoTrue。一天累積 10 萬次。

#### 改法：用 `supabase.auth.getClaims()` 本地驗 JWT

Supabase 官方 SDK 有 `getClaims()` 方法、用 JWT public key 在本機數學驗證、純 CPU、< 1ms、不打外網。

```ts
const {
  data: { claims },
  error,
} = await supabase.auth.getClaims()
// claims.sub = user.id
// claims.email = user.email
// claims.user_metadata = user.user_metadata
```

#### 安全分析

「打 GoTrue 跟本地驗 JWT 差在哪」這個問題、實際上是「**revocation 即時性**」：

| 場景                                          | getUser（打 GoTrue）                           | getClaims（本地驗）                       |
| --------------------------------------------- | ---------------------------------------------- | ----------------------------------------- |
| JWT 過期（1 小時後）                          | 自動失效                                       | 自動失效                                  |
| 用戶 logout（client 清 cookie）               | 失效                                           | 失效                                      |
| Admin 在 Supabase Dashboard 砍 user           | **立即失效**                                   | **JWT 有效期內仍可用**（最多 1 小時殘留） |
| 員工被 disable（employees.is_active = false） | 不關 auth.getUser 的事、靠 capability check 擋 | 同                                        |
| 員工被開除（capability 全砍）                 | 不關 auth.getUser 的事、靠 capability check 擋 | 同                                        |

**關鍵發現**：只有「admin 在 Dashboard 砍 user」這一個場景、getUser 比 getClaims 強。其他場景兩者一樣。

而我們**底層還有兩道保險**：

- RLS（DB 層）— 每個 query 當下查 user → workspace、被砍立刻擋
- `requireCapability`（API 層）— 每個寫入當下查 capability、被砍立刻擋

**結論**：拿掉 getUser、不是少一層保險、是少一層「**重複保險**」。被砍的 user 最多在大廳晃 1 小時、進不了任何辦公室、也碰不了任何文件。

#### 預期效益

- 每個 API / SSR / server action 省 30-80 ms
- 全站影響：108 API + 56 server caller × 每天上萬次
- 累計每天 user-facing latency 省 **數十分鐘**

#### 風險 / 副作用

- ⚠️ JWT 殘留期 1 小時內、被砍 user 仍能讀（但寫不了、進不了 RLS 守的房間）
- ⚠️ 改 `getServerAuth` 影響 56 個 caller、要全套測試
- ⚠️ 要看 Supabase SDK 版本有沒有 `getClaims`、若沒有就自己用 `jose` 驗 JWT

#### 推薦做法

1. 開個 PR 改 `src/lib/auth/server-auth.ts` 一個檔
2. 跑 type-check + 全套 e2e（特別是登入 / 登出 / 切 workspace）
3. 在 dev 跑 30 次測穩定性
4. 上 production 監控 7 天、看有沒有「JWT 殘留」實際造成問題

### 🥈 改造 2：合併 capability check 成 1 個 SQL

#### 現況（`src/app/api/lib/check-capability.ts`）

```ts
// 兩次 sequential DB call
const { data: employee } = await client
  .from('employees')
  .select('role_id')
  .eq('id', employeeId)
  .single()
const { data } = await client
  .from('role_capabilities')
  .select('enabled')
  .eq('role_id', roleId)
  .eq('capability_code', code)
  .maybeSingle()
```

#### 改法 1（簡單）：寫成一個 JOIN query 或 DB function

```sql
CREATE FUNCTION has_capability(p_employee_id uuid, p_code text)
RETURNS boolean
LANGUAGE sql STABLE
AS $$
  SELECT EXISTS (
    SELECT 1 FROM employees e
    JOIN role_capabilities rc ON rc.role_id = e.role_id
    WHERE e.id = p_employee_id
      AND rc.capability_code = p_code
      AND rc.enabled = true
  );
$$;
```

API 端改成：

```ts
const { data: allowed } = await client.rpc('has_capability', {
  p_employee_id: employeeId,
  p_code: code,
})
return allowed === true
```

#### 改法 2（更狠）：server memory cache

```ts
// 用 LRU cache、key = `${employeeId}:${code}`、TTL 60 秒
const cache = new LRUCache<string, boolean>({ max: 10000, ttl: 60_000 })

export async function hasCapabilityByCode(employeeId: string, code: string) {
  const key = `${employeeId}:${code}`
  const cached = cache.get(key)
  if (cached !== undefined) return cached

  const allowed = await dbCheck(...)
  cache.set(key, allowed)
  return allowed
}
```

#### 預期效益

- 改法 1：每個 API 省 5-10 ms（1 次 round trip）
- 改法 2：99% API 0 ms（cache hit）、第一次仍跑 SQL

#### 風險

- 改法 1：低、只是合 SQL
- 改法 2：cache 殘留問題（admin 撤 capability 後、1 分鐘內舊判斷還生效）— 跟改造 1 同類風險、可接受

### 🥉 改造 3：拿掉「傳送中」UI 狀態

不是後端優化、是**前端視覺優化**。

#### 現況

`src/app/(main)/channels/_components/ChannelView.tsx:175-217`：

```
按送 → 立刻顯示「傳送中」泡泡 + spinner → 等 API 回 → 改回真實 bubble
```

API 真的要 200-300 ms、但用戶眼睛**看到一個閃動 spinner**、體感慢。

#### 改法

按送 → **直接顯示真實 bubble**（含時間、樣式跟正常 message 一模一樣）→ 背景 await API → 失敗才把 bubble 標紅 + 還原 draft。

LINE / Slack / Discord 都是這樣做。

#### 預期效益

- API 沒變快、但用戶眼睛看不到等待 = 體感**秒送**
- 1 個檔、低風險、今晚就能上
