## ✅ 決策更新（William 2026-05-23）

**合約 granularity = 彈性兩種模式：**
- 預設/常見：**每訂單一份**（一團一訂單一起簽、該訂單代表人簽）
- 公司行號：**整團一份**（簽一整份、不分訂單）
- schema 已支援、不用改：`tour_id`（必）+ `order_id`（選）。order_id 有值 = 每訂單合約；null = 整團合約。
- **梯次**：公司分 3 梯出團 = 系統上 3 個不同日期的「團」（不是一團 3 訂單）→ 梯次在 tour 層分、每個梯次團各自走上述兩模式之一。

**因此 UI + 總覽要支援雙模式：**
- 訂單列「合約」按鈕 → 建/管「每訂單合約」
- tour 層另給「整團合約」入口（公司一份）→ 建 order_id=null 的合約
- 總覽進度雙模式（更新提案 C）：
  - 有整團合約 → 顯示「整團合約：已簽 / 未簽 / 未建立」
  - 否則（每訂單模式）→「已簽 X / 共 Y 訂單」
  - 系統依「該 tour 有沒有 order_id=null 的合約」自動判定顯示哪種

## 三、提案 A — 流程優化（電子合約更好用）

**減少業務手key、自動帶資料：**
1. **選代表人**：從該訂單的團員清單（`member_ids`）下拉選一位當代表 → 自動帶該團員的 `signer_name / id_number / phone`（不用重打）
2. **集合時間自動帶**：
   - 讀行程表第一段航班起飛時間（itinerary flight）→ 自動填「起飛前 3 小時」為集合時間（可編輯）
   - 沒航班 = 留空 / 手動填
3. **航廈 + 集合聯絡人**：
   - 桃園機場航廈（T1/T2）下拉 + 集合聯絡人欄（emergency_contact_* 已有欄位可複用、或新增 assembly_contact）
   - 存進 `contract_data.assembly = { time, terminal, contact_name, contact_phone }`
4. **送出方式**：產生連結 → LINE / Email 寄給客戶簽（sent_via 已有）→ 客戶走現成 4 步簽署流程
