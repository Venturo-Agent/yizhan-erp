# 建表 SOP — 6 層 checklist（yizhan-erp）

## TL;DR

> blueprint 講「過去寫的人沒看完整 6 道門」。
> 這份 SOP = **每張新表上線前必過的 6 層 checklist + 範本**。漏一層 = bug 從那層漏出來。

新表三句話：

1. **先想清楚資料屬性**（業務 / 子表 / 員工關係 / 全租戶 / 系統 / 個人）→ 決定走哪個 RLS pattern
2. **6 層 checklist 逐條打勾**（L1 → L6）
3. **5 個 SSOT + 中央 module 索引更新**

## Phase 0：資料屬性分類（先想清楚）

| 資料屬性          | 範例                                 | RLS pattern                          | 是否需 workspace_id |
| ----------------- | ------------------------------------ | ------------------------------------ | ------------------- |

## Phase 1：6 層 checklist（逐條打勾）

> 跟 blueprint Layer 1-6 完全對齊。每張新表上線前必過。

### ☐ L1 — 租戶 Feature Gate（workspace_features）

**問**：這張表屬於哪個 module？這個 module 已經登錄在 `workspace_features` 了嗎？

- [ ] 確認 feature_code（`tours` / `finance` / `hr` / `library` / `channels` / `accounting` / `calendar` / `tenants` ...）
- [ ] 如果是**全新 feature**：seed migration 新增 feature row、預設給現存 workspace 開通
- [ ] sidebar `requiredPermission` 對齊
- [ ] 寫到 `src/lib/permissions/features.ts`
- [ ] `<ModuleGuard module="feature_code">` 路由守門

**範本 — seed migration 新增 feature**：

```sql
INSERT INTO workspace_features (workspace_id, feature_code, enabled)
SELECT id, '<feature_code>', true FROM workspaces
ON CONFLICT (workspace_id, feature_code) DO UPDATE SET enabled = true;
```

### ☐ L2 — 角色 Capability（role_capabilities）

**問**：這張表的 read / write / delete 各對應什麼 capability？

- [ ] Capability code 命名：`<module>.<tab>.<action>`（如 `tours.read`、`finance.payments.write`）
- [ ] 寫到 `src/lib/permissions/capabilities.ts`（常數）
- [ ] 寫到 `src/lib/permissions/module-tabs.ts`（**HR /hr/roles UI 列出可勾項目**、5/12 channels 踩過）
- [ ] API route 守門：`await requireCapability(req, 'module.tab.write')`
- [ ] seed migration 預設給適當 role 開通

**範本 — 五個 SSOT 對齊（CLAUDE.md 維度 8）**：

```bash
# commit 前自我 audit
grep -n "<新 capability>" src/lib/permissions/capabilities.ts        # 1
grep -n "code: '<新 capability>'" src/lib/permissions/module-tabs.ts # 2 ← 5/12 踩這坑
grep -n "code: '<新 capability>'" src/lib/permissions/features.ts    # 3
grep -n "workspace_features\|role_capabilities" supabase/migrations*/*<新 feature>*.sql # 4
ls src/app/(main)/<新 module>/                                       # 5
```

### ☐ L3 — 三維 Org Scope（brand × branch × department）

**問**：這張表的 row 屬於某個分公司 / 部門嗎？預設員工只看自己的、還是全 workspace 看？

- [ ] 適用三維：寫 `scope_visible('<module>', row_id)` function 或 inherit parent
- [ ] 不適用三維（如 customers 全 workspace 共享）：標註 N/A
- [ ] 跨界 capability：`cross_branch.read` / `cross_department.read` 對齊
- [ ] **不准散刻**：`sales_id = me` 或 `created_by = me` 寫進 USING（紅線 #2）

**範本 — scope_visible function 草稿**：

```sql
-- 在 scope-helpers migration 加 case
CREATE OR REPLACE FUNCTION scope_visible(p_module text, p_row_id uuid)
RETURNS boolean LANGUAGE plpgsql STABLE AS $$
BEGIN
  CASE p_module
    WHEN '<新 module>' THEN
      -- 譬如：自己 row、自己部門 row、有跨部門 capability
      RETURN EXISTS (
        SELECT 1 FROM <table> t
        WHERE t.id = p_row_id
          AND (t.created_by = current_employee_id()
               OR t.department_id IN (SELECT department_id FROM employee_departments WHERE employee_id = current_employee_id())
               OR has_capability_for_workspace(t.workspace_id, 'cross_department.read'))
      );
  END CASE;
END;
$$;
```

### ☐ L4 — 狀態守門（is_row_editable）

**問**：這張表有「鎖定」狀態嗎？已 paid / approved / locked 的 row 還能改嗎？

- [ ] 有狀態：寫 `is_row_editable('<module>', row_id)` function
- [ ] 無狀態：標註 N/A
- [ ] RLS UPDATE / DELETE policy 用 `scope_visible AND is_row_editable`
- [ ] 前端 hide 編輯按鈕（loading 防連點 + capability 守門 + 狀態 disabled）

**範本 — is_row_editable function 草稿**：

```sql
CREATE OR REPLACE FUNCTION is_row_editable(p_module text, p_row_id uuid)
RETURNS boolean LANGUAGE plpgsql STABLE AS $$
BEGIN
  CASE p_module
    WHEN '<新 module>' THEN
      RETURN EXISTS (
        SELECT 1 FROM <table>
        WHERE id = p_row_id
          AND status NOT IN ('locked', 'approved', 'paid')
      );
  END CASE;
END;
$$;
```

### ☐ L5 — DB RLS（資料庫紅線）

**問**：走哪個 setup procedure？

- [ ] 業務主表：`CALL setup_workspace_scoped_rls('<table>')`
- [ ] 業務子表：`CALL setup_inherited_rls('<table>', '<parent_table>', '<parent_id_col>')`
- [ ] 員工 join 表：`CALL setup_join_table_rls('<table>', '<employee_col>')`
- [ ] 全租戶共享：手寫 `SELECT TO authenticated USING (true)`、寫操作 admin only
- [ ] 系統表：admin only、不開 user policy
- [ ] 個人化：手寫 `USING (user_id = auth.uid())`
- [ ] **❌ 紅線檢核**：
  - [ ] **不准** FORCE RLS on workspaces（紅線 A）
  - [ ] **不准** ENABLE RLS + 0 policy（全擋）
  - [ ] **不准** policy 寫 `OR workspace_id IS NULL`（全人看到）
  - [ ] **不准** 散刻 4 條 CREATE POLICY（走 procedure）

**範本 — 業務主表 migration**：

```sql
-- supabase/migrations/<timestamp>_create_<table>.sql

-- 1. 表結構
CREATE TABLE public.<table> (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id uuid NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  code text,
  name text NOT NULL,
  status text DEFAULT 'active',
  -- ... 業務欄位
  created_by uuid REFERENCES public.employees(id) ON DELETE SET NULL,
  updated_by uuid REFERENCES public.employees(id) ON DELETE SET NULL,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  deleted_at timestamptz
);

-- 2. Index（效能規格）
CREATE INDEX <table>_workspace_created_idx ON public.<table> (workspace_id, created_at DESC);
CREATE UNIQUE INDEX <table>_workspace_code_unique ON public.<table> (workspace_id, code) WHERE deleted_at IS NULL;
CREATE INDEX <table>_deleted_at_idx ON public.<table> (deleted_at) WHERE deleted_at IS NOT NULL;

-- 3. RLS（走 procedure、不散刻）
CALL setup_workspace_scoped_rls('<table>');

-- 4. updated_at trigger
CREATE TRIGGER <table>_updated_at BEFORE UPDATE ON public.<table>
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- 5.（選配）capability seed
INSERT INTO role_capabilities (role_id, capability_code)
SELECT id, '<module>.<tab>.read' FROM workspace_roles WHERE code IN ('admin', 'manager')
ON CONFLICT DO NOTHING;
```

### ☐ L6 — 防呆 / SSOT（中央 module）

**問**：寫操作有沒有走中央 module？

- [ ] 編號生成：用 `@/lib/codes.ts` 對應 helper、**不准** 前端自己組 `D` + Date.now()
- [ ] 新增編號類型：先寫 RPC migration（`generate_<x>_code`）→ 加進 `codes.ts` → types.ts 補
- [ ] 錯誤處理：API catch 用 `dbErrorResponse(err)`、**不准** `error.message` 直 return
- [ ] 連點防護：dialog 必傳 `loading` prop（ESLint `venturo/form-dialog-loading-required` 自動偵測）
- [ ] audit context：寫操作 API 必 call `recordApiAuditContext(client, { actorId, reason })`
- [ ] FK 紅線 B：`created_by` 等審計欄位指 `employees(id)`、**不是** `auth.users(id)`
- [ ] 寫入時：`created_by: currentUser?.id || undefined`、**不是** `|| ''`（空字串會炸）

**範本 — API route**：

```ts
// src/app/api/<module>/route.ts
import { NextRequest } from 'next/server'
import { requireCapability } from '@/lib/auth/require-capability'
import { getSupabaseAdminClient } from '@/lib/supabase/admin'
import { recordApiAuditContext } from '@/lib/audit/record-context'
import { dbErrorResponse } from '@/lib/db-error-translate'
import { generate<Code> } from '@/lib/codes'

export async function POST(req: NextRequest) {
  const { user, workspaceId } = await requireCapability(req, '<module>.<tab>.write')
  const body = await req.json()
  const client = getSupabaseAdminClient()  // per-request、紅線 C

  await recordApiAuditContext(client, { actorId: user.employeeId, reason: 'create-<x>' })

  // 編號走 RPC（防競態、紅線 L6）
  const code = await generate<Code>(client, workspaceId)

  const { data, error } = await client.from('<table>').insert({
    workspace_id: workspaceId,
    code,
    ...body,
    created_by: user.employeeId ?? undefined,  // 紅線 B
  }).select().single()

  if (error) return dbErrorResponse(error)  // 翻中文業務語言
  return Response.json({ data })
}
```
