# 出納單品項級重構 — 完整 Spec

## TL;DR

當前模式：出納單 = 勾選請款單批次
新模式：出納單 = 勾選請款**品項**（payment_request_items）

原因：跨行手續費要算、按品項算最準（不同品項供應商不同銀行）。

## 新 Flow

```
新增出納 dialog（多步驟）：
  Step 1: 選公司出帳帳戶（如「合庫」）→ from_bank_account_id
  Step 2: 勾選 payment_request_items（顯示同行✓ / 跨行⚠️）
  Step 3: 提示框「同行 X 筆 0 元 / 跨行 Y 筆、填總手續費」
  Step 4: 確認 → 加入「預覽暫存」（frontend state、不寫 DB）
  Step 5: 切換帳戶（國泰）、勾剩下、循環 Step 2-4
  Step 6: 全部分完 → 「預覽出納單」
  Step 7: 預覽按帳戶分組 + 底部總表
  Step 8: 確認 → 一次儲存所有 disbursement_orders + items
  Step 9: 出帳 status='paid' → 系統自動建「手續費請款單」回 tour
```

## William 拍板的 5 個關鍵設計

### Q1: 請款單品項可拆 ✓

請款單 A 有 5 品項、可拆 3 筆出帳。

### Q2: 部分出帳「自動 fork 新請款單」

業務語意：每張請款單最終必須「同一 status」、不允許 partial。

實作：
- 請款單 A 含 5 品項
- 出帳 3 品項時：
  - 把那 3 品項從 A 移到新請款單 B（fork）
  - B status='billed'、A 剩 2 品項 status='pending'
- 原 A 的 total_amount 自動重算
- 新 B 的 code 衍生 e.g. A-FORK-1

**好處**：
- 報表清楚（每張請款單 = 同 status）
- 不需要 partial 狀態

**注意**：
- fork 出去的新請款單 retain 原 tour_id / supplier 等
- 原請款單還能繼續加品項（如果業務有此需求）

### Q3: 手續費請款單「出完帳後回填」

避免無限循環：
- 出納單 status='paid' 那一刻、系統自動建「手續費 payment_request」回 tour
- 這張手續費請款單 status='paid'（直接 billed、不再走出帳流程）
- 跟其他「正常請款」分流（不會被選進新出納單）

實作：
- `payment_requests.expense_type = 'bank_fee'` 標記
- expense_type='bank_fee' 不顯示在「待出帳」清單
- 直接寫進 tour 帳務、影響 tour 成本統計

### Q4: 銀行對應嚴格按 bank_code

不做「同集團」例外（業界一致、銀行端不會區分）。
跨行 = user 自己填手續費（同集團免 → user 填 0）。

### Q5: 「預存」= 純 frontend state

預覽前：所有出納單草稿存 React state、不寫 DB
預覽通過：一次性 batch INSERT
Refresh = 全掉、要重做

## Schema 設計

### 新表：disbursement_order_items

```sql
CREATE TABLE disbursement_order_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  disbursement_order_id UUID NOT NULL REFERENCES disbursement_orders(id) ON DELETE CASCADE,
  payment_request_item_id UUID NOT NULL REFERENCES payment_request_items(id),
  -- snapshot（避免 source 改了不一致）
  amount NUMERIC NOT NULL,
  supplier_bank_code TEXT,  -- snapshot 供應商銀行
  fee_amount NUMERIC DEFAULT 0,  -- 該品項分攤的手續費
  has_cross_bank_fee BOOLEAN DEFAULT FALSE,
  workspace_id UUID NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_doi_disbursement ON disbursement_order_items(disbursement_order_id);
CREATE INDEX idx_doi_item ON disbursement_order_items(payment_request_item_id);
```

### disbursement_orders 加欄位

```sql
ALTER TABLE disbursement_orders
  ADD COLUMN from_bank_account_id UUID REFERENCES bank_accounts(id),
  ADD COLUMN total_fee NUMERIC DEFAULT 0,
  ADD COLUMN batch_uuid UUID;  -- 同 batch 一起預覽

COMMENT ON COLUMN disbursement_orders.from_bank_account_id IS '公司出帳帳戶（決定是否跨行）';
COMMENT ON COLUMN disbursement_orders.batch_uuid IS '多帳戶分批的 batch id、同 batch 一起預覽';
```

```sql
    'tour_expense', 'salary', 'bank_fee', 'other'

```

### Fork 邏輯：SQL function

```sql
CREATE OR REPLACE FUNCTION fork_payment_request_for_partial_billing(
  p_request_id UUID,
  p_item_ids UUID[]
)
RETURNS UUID
LANGUAGE plpgsql
AS $$
DECLARE
  v_new_request_id UUID;
  v_original RECORD;
BEGIN
  SELECT * INTO v_original FROM payment_requests WHERE id = p_request_id;

  -- 建新請款單（copy original + code + 'FORK-N'）
  INSERT INTO payment_requests (
    workspace_id, tour_id, code, request_date, expense_type, ...
  )
  SELECT
    workspace_id, tour_id,
    code || '-FORK-' || (SELECT count(*) + 1 FROM payment_requests WHERE code LIKE v_original.code || '-FORK-%'),
    request_date, expense_type, ...
  FROM payment_requests WHERE id = p_request_id
  RETURNING id INTO v_new_request_id;

  -- 移動指定品項到新請款單
  UPDATE payment_request_items
  SET request_id = v_new_request_id
  WHERE id = ANY(p_item_ids);

  -- 重算兩張請款單的 total_amount
  UPDATE payment_requests SET amount = (
    SELECT COALESCE(SUM(subtotal), 0) FROM payment_request_items WHERE request_id = p_request_id
  ) WHERE id = p_request_id;

  UPDATE payment_requests SET amount = (
    SELECT COALESCE(SUM(subtotal), 0) FROM payment_request_items WHERE request_id = v_new_request_id
  ) WHERE id = v_new_request_id;

  RETURN v_new_request_id;
END;
$$;
```

## ⚠️ 風險 / 注意事項

1. **舊資料相容性**：當前 disbursement_orders 走「請款單級」、改新模式後舊 data 怎麼 migrate？
   - 建議：新增 `is_legacy` flag、舊 data 標 legacy、不走新邏輯
   - 或：所有舊 disbursement 自動 fork 一次（risky）

2. **報表向後相容**：舊報表查 disbursement → request、新報表查 disbursement → items → request
   - SQL view 抽象：建 `v_disbursement_full` view、隱藏內部結構

3. **手續費的 tour 成本歸屬**：
   - 手續費請款單回 tour、但「跨行手續費」未必算 tour 的成本（是公司運營成本）
   - 業務面要拍：算 tour 成本還是公司成本？

4. **多帳戶分批的 race condition**：
   - 同一筆 item 不能被兩張 disbursement_order 同時勾
   - 防呆：勾完一筆、UI 標 disabled、不可再勾
   - DB 層 unique constraint on (payment_request_item_id) in disbursement_order_items
