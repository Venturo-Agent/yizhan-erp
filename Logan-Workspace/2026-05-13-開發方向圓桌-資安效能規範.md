## 0. 大方向（William 拍板、不可動）

### 一句話定義

**venturo-aierp = SaaS 多租戶 ERP 平台 + AI 資訊整合層**。

### 三層權限模型（不可繞）

```
┌────────────────────────────────────────────────────────┐
│ Tenant Layer：租戶買功能（workspace_features）          │
│   ├─ 漫途旅行社 買了 tours + finance + hr               │
│   ├─ 角落旅遊 買了 tours + hr                           │
│   └─ Client X 買了 tours + cis                          │
├────────────────────────────────────────────────────────┤
│ HR Layer：員工取得 capability（role_capabilities）      │
│   ├─ 業務角色 — tours.read + orders.write               │
│   ├─ 會計角色 — finance.* + accounting.*                │
│   └─ 助理角色 — tours.read + customers.read             │
├────────────────────────────────────────────────────────┤
│ User Layer：員工進系統、看見符合自身能力的東西            │
│   └─ user_id → employee → role → capability             │
└────────────────────────────────────────────────────────┘
```

**No exception 規則**：
- 任何訪問控制只走 `workspace_features` + `role_capabilities` + RLS
- 不准 hardcoded `if (workspace.code === '漫途')` / `isAdmin` 短路
- 「跨 workspace 能力」（譬如 `tenants`）= 一個普通 feature、漫途的 workspace 開了就有

### 未來方向（AI 整合）

- `platform_integrations` feature 加新整合（aitoearn / xhs / capture bot 等）
- 員工 SSO 串 ERP ↔ 工具
- 三維（brand × branch × department）打通跨工具排程
- AI 不在 schema 中央化、是「workspace 內 feature」的擴充

## 1. 圓桌出席（Logan 召集、6 個靈魂）

> 這場是「規範對齊」會議、不是「新功能設計」。各靈魂 push 自身領域的紅線。

### 🧱 資料庫架構師（DB Architect）
- **管**：schema / RLS / index / migration / FK / 軟刪
- **push**：
  - 每張新表必過 6 層 checklist（建表 SOP）
  - RLS 走 `setup_*_rls` procedure、不准散刻 4 條 CREATE POLICY
  - migration 入版控、不准手改 Dashboard 沒留 trace
  - composite PK → 加 surrogate `id uuid PK` + UNIQUE constraint

### 🛡 資安工程師（Security Engineer）
- **管**：多租戶隔離 / capability / API 守門 / FORCE RLS / 審計
- **push**：
  - **紅線 A**：workspaces 永遠 NO FORCE（4/20 事故案底）
  - **紅線 B**：審計欄位 FK 指 employees(id)、不是 auth.users(id)
  - **紅線 C**：admin client per-request、不准 singleton（跨 request 資料污染風險）
  - API route 寫操作 100% `requireCapability` 守門（已 audit / 0 漏）
  - secret 管理走 `~/.config/venturo/secrets.env`、不入 git
  - 所有寫操作 `recordApiAuditContext(client, { actorId, reason })`

### ⚡ 效能工程師（Performance Engineer）
- **管**：cache / SWR / pagination / 列表 / bundle / 列表 query
- **push**：
  - **列表預設 20 筆、分頁固定 15 筆、不給筆數選擇器**（egress 殺手）
  - **SWR config**：`revalidateOnFocus: false` + `dedupingInterval: 5min`
  - **Server-side filter**：列表搜尋走 PostgREST query string、不全撈再前端 filter
  - **Layout context SSOT**：`useLayoutContext` 一次撈 user / employee / workspace / capabilities / features
  - **Hydration race**：等 `_hasHydrated` 才判斷 capability、否則空權限誤判 redirect 死循環
  - **Realtime**：走 entity hook 內建 `useRealtimeSync()`、不自寫 channel
  - **Bundle**：動態 import 大 lib（jsPDF / xlsx > 100KB）、`next/image` 用 webp、icon 個別 import

### 💰 DBA / DBOps（Database Operator）
- **管**：Supabase cost / egress / connection pool / pooler / 索引使用率
- **push**：
  - **Egress budget**：每月 Supabase 流量上限、不能無限撈
  - **Connection pool**：API route 用 service_role admin client per-request、避免 connection leak
  - **Pooler URL**：CI / Linux 環境用 IPv4 pooler（`aws-0-xxx.pooler.supabase.com:6543`）、Mac 本機同
  - **Index 使用率**：定期跑 `pg_stat_user_indexes` 看 idx_scan、清沒人用的 index（成本）
  - **Long-running query 監測**：`pg_stat_statements` extension、抓慢查詢
  - **EXPLAIN ANALYZE**：寫新 query 前先看 plan、確保走 index、不是 sequential scan
  - **Soft delete index**：所有 `deleted_at` 欄位用 `WHERE deleted_at IS NULL` partial index

### 🏢 SaaS 業務架構師（SaaS Business Architect）
- **管**：租戶模型 / billing / feature gate / 跨租戶能力 / onboarding
- **push**：
  - **`workspace_features` = 客戶買的功能**（不是「給開發者方便」的 flag）
  - **新 module 上線 seed migration 必含**：`workspace_features` 預設啟用 + `role_capabilities` 預設 grant
  - **跨租戶能力**走 `tenants` feature（漫途的 workspace 買了 tenants = 漫途能管所有 workspace）
  - **不留隱性 onboarding 步驟**（不要「user 自己去開 feature 才能用」）
  - **Workspace 平等**：沒有「platform_owner / agency」分類、概念層不存在
  - **Billing 觀點**（未來）：feature category（basic / premium / enterprise）對應收費等級

### 🤖 AI 整合架構師（AI Integration Architect）
- **管**：未來 capture bot / aitoearn / xhs / LINE OA 整合 / iframe / API 接入
- **push**：
  - **AI 不在 schema 中央化**、走 `platform_integrations` feature 擴充
  - **整合走 workspace_integrations 表**（per-workspace 配置、不是 hardcoded）
  - **AI 操作走員工身分**（不是 system 身分）、所有 AI 寫入有 `created_by` audit
  - **iframe / 第三方 API 接入**：CSP / sandbox / origin check
  - **AI 內容資料屬租戶**（不是平台共用）、RLS 跟一般業務表同樣隔離

## 3. 資安規範（從 0 寫、第一次工程師版）

> 紅線：違反 = client data 外洩 = 商業終結。

### 3.1 多租戶隔離

| 規範 | 落實機制 | 違反後果 |
|---|---|---|
| 所有業務表必有 `workspace_id` | schema + RLS USING (`workspace_id = get_current_user_workspace()`) | 跨租戶資料外洩 |
| `workspaces` 表永遠 NO FORCE RLS | 紅線 A、自動 audit | 全員登不進系統（4/20 事故） |
| 寫操作 admin client per-request | `getSupabaseAdminClient()` 函數 / 不准 singleton export | 前 request 的 context 殘留、跨租戶污染 |
| 跨租戶能力 = `tenants` feature | 不在 code 寫死、走 `workspace_features` | 平等原則違反、未來不可控 |

### 3.2 Capability 守門（雙層）

| 層級 | 規範 | 工具 |
|---|---|---|
| **API 層** | 寫操作必 `await requireCapability(req, CAPABILITIES.X)` | `src/lib/auth/require-capability.ts` |
| **DB 層** | RLS UPDATE / DELETE policy 含 `has_capability_for_workspace(ws_id, 'X')` | DB function、不准散刻 |
| **UI 層**（次要） | 按鈕 disabled by capability、Tab 隱藏 by capability | `useMyCapabilities()` hook |

**不准散刻 capability 字串**（5/13 caller migration 全清、紅線、ESLint 守）：
- ❌ `requireCapability(req, 'tours.read')`
- ✅ `requireCapability(req, CAPABILITIES.TOURS_READ)`

### 3.3 審計欄位（紅線 B）

```sql
-- 對的：FK 指 employees
created_by uuid REFERENCES public.employees(id) ON DELETE SET NULL

-- 錯的：FK 指 auth.users
created_by uuid REFERENCES auth.users(id) ON DELETE SET NULL
```

**前端寫入**：
```ts
// 對的
created_by: currentUser?.id ?? undefined

// 錯的（空字串 → uuid 型別 cast 炸）
created_by: currentUser?.id || ''
```

**例外**（合規、可指 auth.users）：`user_id` / `sender_id` / `friend_id`（這些 row 本身就是 user account、不是「員工操作紀錄」）

### 3.4 Secret 管理

- 所有 token / key / URL 集中：`~/.config/venturo/secrets.env`（chmod 600）
- 文件 / chat 不准寫 value、只寫變數名（`$SUPABASE_DB_URL`）
- 用的時候 `source secrets.env`、用完不留
- GitHub secrets 走 `gh secret set --body "$VAR"`、不貼 web UI

### 3.5 SQL Injection 防護

- 全 ORM / Supabase client、用 parameterized query
- DB function 內用 `format()` + `%I` (identifier) / `%L` (literal) escape、不字串拼接

### 3.6 Audit trail（寫操作必留）

```ts
// API route 內、寫操作前
await recordApiAuditContext(client, { actorId: user.employeeId, reason: 'create-order' })
```

涵蓋：30+ route（其餘漸進補）

### 3.7 紅線清單（不可繞）

| 紅線 | 對應 | 落實 |
|---|---|---|
| #0 沒有特權 | code 不准 hardcoded admin 短路 | ESLint + audit:rls grep |
| A workspaces NO FORCE | 紅線 A | audit:rls 自動偵測 |
| B 審計 FK → employees | 紅線 B | audit:rls 自動偵測 |
| C admin per-request | 紅線 C | code review |
| 不准 `as any` / `: any` | 規範 | type-check + ESLint |
| 不准 `--no-verify` | 規範 | 人為紀律 |

## 4. 效能規範（從 0 寫、第一次工程師版）

> 紅線：違反 = Supabase egress 費爆 = 商業上不可持續。

### 4.1 讀取（SELECT）

| 規範 | 說明 | 違反後果 |
|---|---|---|
| **列表預設 20 筆、分頁 15 筆** | `defaultPageSize` 在 entity hook 設、UI 不給「每頁筆數」選擇器 | 全表撈 = egress 殺手 |
| **SWR `revalidateOnFocus: false`** | 切回 tab 不 refetch | 重複 query 浪費 |
| **SWR `dedupingInterval: 5min`** | 5 分鐘內不重新查 | 同頁不同 component 各撈 |
| **Server-side filter** | 列表搜尋走 PostgREST `&name=ilike.%X%`、不全撈再 JS `.filter()` | 全撈再 filter = 全撈成本 |
| **Layout context 一次撈** | `/api/auth/layout-context` 抓 user + employee + workspace + capabilities + features | 每頁各 query 一次 = 5x cost |
| **Hydration 等 `_hasHydrated`** | 不等會空 capabilities → 誤判 unauthorized → redirect 死循環 | 用戶白屏 |
| **Realtime 走 `useRealtimeSync()`** | entity hook 內建、不自寫 channel | leak channel = 連線爆 |
| **訊息列表 scroll up infinite load** | 不一次撈全部 | 一次撈 1000 條訊息炸 |

**反 pattern**：
- ❌ `useSuppliers({ all: true })` 撈全表客戶（漫途 385、再多會炸）
- ❌ 前端 `.filter(s => s.name.includes(q))` 搜尋
- ❌ 每個 component 自己 `supabase.from('xxx').select('*')`
- ✅ `useSuppliersPaginated({ search: q, page: 1, pageSize: 15 })`

### 4.2 寫入（INSERT / UPDATE）

| 規範 | 落實 |
|---|---|
| 編號走 RPC + advisory lock | `@/lib/codes.ts`（11 種）、防競態 |
| 兩步驟操作 atomic | INSERT A + INSERT B 用 transaction / RPC（員工 email rollback 教訓） |
| `created_by: ... ?? undefined` | 紅線 B、不是 `|| ''` |
| 儲存按鈕 `disabled={loading}` | 防連點、ESLint `form-dialog-loading-required` 自動偵測 |
| 寫入失敗 toast + state 還原 | 不靜默失敗 |
| 寫入 audit context | `recordApiAuditContext` |
| RLS scope_visible + is_row_editable | UPDATE / DELETE 雙守門 |

### 4.3 Index 規格

| 表類別 | 必有 index | 用途 |
|---|---|---|
| 所有有 `workspace_id` 的表 | `(workspace_id, created_at DESC)` | RLS 過濾 + 列表排序 |
| FK 欄位 | `(<fk_column>)` | join 效能 |
| 業務編號 | `(workspace_id, code) UNIQUE` | 防撞 + 查詢 |
| Soft delete | `(deleted_at) WHERE deleted_at IS NOT NULL` partial | filter 加速 |
| 全文搜尋（之後） | GIN index on `to_tsvector(...)` | 大 dataset search |

### 4.4 Bundle 規格

| 規範 | 落實 |
|---|---|
| 大 library 動態 import | `await import('@/lib/pdf-xxx')` for jsPDF / xlsx |
| 重頁面 `dynamic(() => import('...'), { ssr: false })` | client-only chunk |
| `next/image` | 自動 webp / 多 size |
| icon 個別 import | `import { Save } from 'lucide-react'`、不 `import *` |
| Tailwind purge 自動 | 不亂寫 safelist |
| Sentry sample rate | 0.1 / 0.01、prod 不要 1.0 燒錢 |
| Server component default | 需要互動才加 `'use client'` |

### 4.5 Supabase Cost 規範

| 規範 | 為什麼 |
|---|---|
| **不撈全表**（`all: true` 慎用） | egress 主要殺手 |
| **不寫 N+1 query** | 列表 + 每 row fetch detail = N+1、走 join / RLS  |
| **Realtime 用 entity hook** | 不自寫 channel、避免 leak |
| **Cron job 用 service_role** | 跨 workspace 操作走 admin client、不冒充 user |
| **Background task 寫 audit_logs / cron_execution_logs** | 觀測 cost 從哪來 |
