# 全站狀態 SSOT 盤點報告

## 起因

William 在 `/tours/TW260321A`（Liz 高爾夫球團）看到請款單列表狀態：
- I01 (3/19) → **「已付款」**
- I02 (4/2)  → **「Billed」**（William 看成 Filled、其實是 b/f 字形相近）
- I04 (4/28) → **「Billed」**

3 張單狀態 label 不一致、要求做全盤 SSOT 盤點。

## SSOT 定義

正式 SSOT：`src/lib/design/status-tone-map.ts`
- `STATUS_LABEL_MAP[type][status] → 中文 label`
- `STATUS_TONE_MAP[type][status] → tone (pending/info/success/warning/danger/neutral)`
- 用法：`<StatusBadge type="payment_request" status={pr.status} />`
- fallback：找不到 mapping 回傳「原值」（raw 英文 enum）→ 這就是 William 看到「Billed」的原因

## 發現 B：DB CHECK constraint 缺漏（防禦層裸奔）

8 張表 status 欄**只有 NOT NULL、沒有 CHECK enum 限制**：

| 表 | 實際使用值 | 缺 CHECK 的風險 |
|---|---|---|
| **payment_requests** | pending / billed / paid | 任何字串都進得去 → A1 殘留就是這樣產生的 |
| orders | pending / confirmed | 同上 |
| disbursement_orders | paid | 同上 |
| quotes | draft / proposed | 同上 |
| todos | pending / in_progress / completed | 同上 |
| contracts | draft / signed / cancelled | 同上 |
| journal_vouchers | posted | 同上 |
| employees | active | 同上 |
| checks | (無資料) | 同上 |

**建議修法**：分批寫 migration 加 CHECK constraint（按業務 SSOT enum）。優先加 payment_requests / orders / disbursement_orders（會計鏈核心）。

## 發現 C：STATUS_LABEL_MAP 缺漏（UI label 漏 raw）

### C1. payment_request 沒寫 billed（直接 root cause）

```ts
// src/lib/design/status-tone-map.ts:174
payment_request: {
  pending: '未付款',
  confirmed: '待付款',
  paid: '已付款',
  // 缺 billed → fallback 顯示英文 "billed"
}
```

**建議**：加 `billed: '已付款'`（防 A1 沒清乾淨時 raw 漏出）。

### C2. receipt — map 寫了 cancelled、但 DB CHECK 不允許

```ts
// status-tone-map.ts:180
receipt: {
  pending: '待處理',
  confirmed: '已確認',
  cancelled: '已取消',  // ← DB CHECK 不允許這值
  refunded: '已退款',
}

// DB CHECK：
receipts.status = ANY (['draft', 'pending', 'pending_verify', 'confirmed', 'refunded', 'rejected'])
```

**建議**：map 砍 cancelled、加 draft / pending_verify / rejected（對齊 DB CHECK）。

### C3. contract 缺 cancelled / draft

```ts
// status-tone-map.ts:156
contract: { unsigned: 'pending', signed: 'success' }
// DB 實際用 cancelled / draft / signed
```

**建議**：map 加 `draft / cancelled`（對齊實際使用）。

### C4. 整型缺 SSOT 的 type

| 表 | 實際 status | 在 STATUS_LABEL_MAP？ |
|---|---|---|
| ai_agents | active / disabled | ❌ |
| invoice_batches | pending / partial / paid / cancelled | ❌ |
| itineraries | draft / published | ❌ |
| kb_sailings | available / waitlist / ... | ❌ |
| workspace_billing_records | pending / paid / overdue | ❌ |

**建議**：報告完後分批補 mapping（不阻塞核心、可漸進）。

## 發現 D：散刻 status label 的 anti-pattern（破壞 SSOT）

### D1. 殘留舊 enum「approved」（finance/treasury/page.tsx:131）

```ts
status: pr.status === 'paid' ? '已付款' : pr.status === 'approved' ? '已核准' : '待審核',
```

`approved` 在 5/15 砍 4 狀態 → 3 狀態時就消失了、但這行沒跟著清。
影響：任何 confirmed/billed/pending 的 PR、在這頁顯示「待審核」（錯）。

**建議**：改用 SSOT — `getStatusLabelFor('payment_request', pr.status)`、或直接接 StatusBadge。

### D2. inline 三元 receipt status（同檔 line 117）

```ts
status: r.status === 'confirmed' ? '已確認' : '待確認',
```

同樣的散刻。**建議**：改用 SSOT。

### D3. PAYMENT_REQUEST_STATUS_LABELS 寫了 billed: '已出帳'（_disbursement/_constants.ts:33）

```ts
export const PAYMENT_REQUEST_STATUS_LABELS = {
  pending: DISBURSEMENT_LABELS.待處理,
  confirmed: DISBURSEMENT_LABELS.已確認,
  billed: DISBURSEMENT_LABELS.已出帳,  // ← 跟 SSOT 完全不對齊
}
```

SSOT 講 `pending=未付款 / confirmed=待付款 / paid=已付款`，這份卻講「待處理 / 已確認 / 已出帳」、且包含已死的 `billed` key。

**建議**：整個檔砍掉 / 改為 re-export `STATUS_LABEL_MAP.payment_request`。

### D4. statusLabels 重複（finance/requests/_types.ts:48）

```ts
export const statusLabels: Record<PaymentRequestStatus, string> = {
  pending: '未付款', confirmed: '待付款', paid: '已付款',
}
```

跟 SSOT 完全一致、但維護兩份要兩份改。useRequestTable.tsx 拿來做 sort。

**建議**：sort 改 import `getStatusLabelFor`、砍 statusLabels。

### D5. PayablesTab.tsx STATUS_LABELS（自定義 confirmed/billed map）

```ts
const STATUS_LABELS: Record<string, string> = {
  confirmed: COMPONENT_LABELS.STATUS_CONFIRMED,
  billed: COMPONENT_LABELS.STATUS_BILLED,
}
```

仍假設有 billed 存在。**建議**：對齊 SSOT。

### D6. tour-costs.tsx 業務邏輯混中文（最糟）

```ts
// line 142
status: data.status === '已確認' ? 'confirmed' : 'pending',

// line 176, 192
status: '待確認' as const,
```

把「中文 label」當「業務狀態值」存。後續任何 i18n / 比對都會炸。

**建議**：改用英文 enum、UI 顯示再透過 SSOT 轉中文。優先級高（雷大）。

### D7. tour-receipts.tsx _STATUS_MAP 棄用未砍

```ts
const _STATUS_MAP: Record<string, { label: string; tone: StatusTone }> = { ... }
void _STATUS_MAP  // 標棄用、但檔案還在
```

**建議**：直接砍、減少 SSOT 雜訊。

### D8. todos / contracts / quote 模組各自的 *_STATUS_LABELS

`todos/_constants/labels.ts` 的 `TODO_STATUS_LABELS`、`orders/_contracts/constants/labels.ts` 的 `CONTRACT_STATUS_LABELS`、`orders/_quotes/_constants/labels.ts` 的零散字串。

**判斷**：這些是「業務模組 constants 中央化」、跟 SSOT 角色重疊但不衝突。可保留、但建議內部都從 SSOT 拉。
