### 🟥 Tier A 原始討論（保留作參考）

#### A1. `orders.code` 雙欄位 SSOT
現況：`code` 跟 `order_number` 雙欄位、寫法不一致。
選項：
- (a) 砍 `code`、`order_number` 為 SSOT → 要 migration 清舊資料
- (b) `code` = order full code（同 order_number）、`order_number` deprecate
- (c) 不動、留下「視 row 而定」的爛狀態

#### A2. `employees.email` vs `personal_info.email` SSOT
現況：兩處同步寫、API 規格說 employees.email 是 SSOT、但 form 兩邊都寫。
選項：
- (a) 砍 `personal_info.email`、全站只用 `employees.email`
- (b) 反過來
- (c) 維持雙寫

#### A3. 錯誤翻譯文案 tone
要寫 `translateDbError(error)` helper、把 Postgres 英文錯誤翻成中文業務語言。
範例你看哪個風格：

A 風格（業務友善）：
- 23505 → 「這個 email 已經有人在用了、請改別的」
- 23503 → 「找不到對應的廠商、請先建立廠商再回來」

B 風格（中性技術）：
- 23505 → 「欄位 email 重複、請檢查」
- 23503 → 「外鍵驗證失敗、請檢查相關資料」

C 風格（你寫的具體範例）

#### A4. 紅線 B 三表 FK 修法（created_by 指 auth.users 改 employees）
受影響：`email_system` / `tour_control_forms` / `image_library`
現有 NULL / 不正確 created_by 怎麼處理？
- (a) 砍掉現有 row（demo 階段、資料不重要）
- (b) backfill 用 system_employee 作 default
- (c) 改成 NULLABLE FK、保留 NULL row

#### A5. 模組搬 entity hook 順序
現況：員工 / 訂單 / 行程 / 財務 4 個模組混用舊 store + 新 entity hook。
我建議順序：員工先（bug 最多）→ 訂單 → 行程 → 財務。
你同意嗎？還是有別的優先？
