# 員工資料 + 供應商 — Bug 全掃描報告

## TL;DR — 直接回答 William 今天遇到的事

> 「員工資不能存檔、存檔有衝突、整個供應商感覺漏洞百出」

**為什麼會這樣？三個根本原因（兩個模組共犯）**：

1. **編號是前端算的、沒鎖**
   - 員工 `E001` 跟供應商 `S00001` 都是前端從 store 找最大值加一
   - 兩個分頁同時新增、或新租戶看不到舊租戶（被 RLS 蓋住）算出來一樣
   - 後端 unique constraint 擋下、噴 23505 → 顯示給你看的就是「儲存失敗」

2. **錯誤訊息被吃掉**
   - DB 噴英文 `Key (workspace_id, lower(email))=... already exists`
   - 前端 toast 統一翻成「儲存失敗，請稍後再試」
   - 你看不出來是 email 撞、編號撞、還是欄位錯
   - 重試 → 又撞 → 又「儲存失敗」→ 灰心

3. **按鈕沒鎖**
   - 點「儲存」沒 `disabled={loading}`
   - 慢回應時連點 2-3 下、第一筆寫進去、第二筆撞 unique → 衝突
   - 供應商完全沒做、員工有些地方沒做

**修好上面三個、今天測到的「存不了 / 有衝突」90% 會消失**。

不是你的問題。是 code 沒收乾淨。

## 員工資料模組 — 10 個 bug

### 🔥 炸級（5）

#### E-001 API 錯誤訊息吃掉真實衝突原因
- **症狀**：建重複 email、顯示「建立登入帳號失敗」、看不出原因
- **根因**：`/api/employees/create` 直接 return Supabase 英文錯誤
- **證據**：`src/app/api/employees/create/route.ts:99-100`
- **修法**：API 攔 23505、轉中文業務訊息「此 email 已被使用」
- **風險**：低、只改 return 訊息

#### E-002 改 email 沒 rollback、員工會永遠登不進來 ⚠️
- **症狀**：改員工 email → employees 表改成功 → auth.users 改失敗 → 員工下次登入用新 email、auth.users 還是舊的 → **永遠登不進**
- **根因**：兩步驟操作沒事務、沒 rollback
- **證據**：`src/app/api/employees/[id]/route.ts:76-110`
- **修法**：auth sync 失敗時、立即 rollback employees UPDATE 回舊值
- **風險**：要跑 `tests/e2e/login-api.spec.ts` 驗證（紅線 A）
- **特別注意**：這是 production 等級 bug、之前出過、還沒收乾淨

#### E-003 終止員工後還顯示在列表
- **症狀**：按「終止員工」設 status=terminated、但列表 store 沒 refetch、終止員工還在
- **根因**：page.tsx 用舊 `useUserStore`（沒 soft delete 概念）、不是新 entity hook
- **證據**：`src/app/(main)/hr/page.tsx:40-42` + `src/stores/user-store.ts`
- **修法**：
  - A（快）：page.tsx filter 加 `&& !emp.deleted_at`
  - B（正確）：整頁改用 `useEmployees` entity hook
- **風險**：A 改一行、B 改整頁

#### E-004 fetchAll 沒 await、refetch 競態
- **症狀**：存完員工關 dialog 但列表還沒重 load、看到舊資料
- **根因**：`onSubmit={() => { fetchAll() }}` 沒 await
- **證據**：`src/app/(main)/hr/page.tsx:267-269`、`284-286`
- **修法**：改 `async () => { await fetchAll(); ... }`
- **風險**：低

#### E-005 員工編號競態 ⚠️
- **症狀**：兩個 HR 同時新增、都算到 E100、第二筆撞 unique → 失敗
- **根因**：`EmployeeForm.tsx:330-336` 純前端從 store 找最大值加一、無鎖
- **證據**：`src/app/(main)/hr/_components/EmployeeForm.tsx:328-336`
- **修法**：改後端產編號（API SELECT max FOR UPDATE 鎖）、前端不算
- **風險**：要改 API + 移前端邏輯、影響範圍只在新增員工

### 😣 痛級（3）

#### E-006 Email 雙軌（personal_info.email + employees.email）
- 寫入時兩邊都寫、SSOT 破碎、舊資料殘留
- 證據：`EmployeeForm.tsx:193, 266, 349`
- 修法：全砍 personal_info.email、只用 employees.email、寫 migration 遷移舊資料

#### E-007 created_by 沒寫進去（審計缺失）
- 建員工時不記錄誰建的
- 證據：`src/app/api/employees/create/route.ts:87-92`
- 修法：insert payload 加 `created_by: guard.employeeId`

#### E-008 錯誤訊息英文、不是中文業務語言
- API 沒統一翻譯層、原 Postgres 英文錯誤直接漏出
- 修法：跟 E-001 一起修

### 🗑 髒級（2）

#### E-009 RLS 還在用 is_super_admin() 繞道
- 違反紅線 0（系統內沒有超級管理員）
- 證據：`supabase/migrations/20260112030000_reenable_employees_rls.sql:36`
- 修法：Phase 2（DB 重建時）一起砍、現在不動

#### E-010 Soft delete 欄位有、但 store 層沒用
- employees 表有 `deleted_at`、舊 store 沒 filter
- 跟 E-003 同根、一起修

## 供應商模組 — 7 個 bug

### 🔥 炸級（1）

#### S-001 供應商編號跨租戶撞 ⚠️
- **症狀**：新租戶建供應商生 `S00001`、舊租戶也有 `S00001`、unique constraint 擋下
- **根因**：`generateSupplierCode()` 用被 RLS 隔離的 SELECT、新租戶看不到舊租戶的編號
- **證據**：
  - `SuppliersPage.tsx:27-57`（前端算碼）
  - `supabase/migrations/20260421120000_tenant_scoped_unique_codes.sql:72-73`（DB 改成 `UNIQUE(workspace_id, code)` 了、但前端邏輯沒同步）
- **修法**：改用 `useSuppliers()` entity hook 的結果算編號（已 workspace 隔離）、或 SELECT 加 `.eq('workspace_id', currentWorkspaceId)`
- **風險**：低、entity hook 本來就會 retry

### 😣 痛級（3）

#### S-002 沒 loading state、可連點
- 點儲存沒 disabled、連點會重複建
- 證據：`SuppliersPage.tsx:178-237`、`SuppliersDialog` 沒收 loading prop
- 修法：加 `useState(false)` loading state、傳給 dialog

#### S-003 錯誤訊息通用、不顯示原因
- catch 統一 toast「儲存失敗，請稍後再試」
- 證據：`SuppliersPage.tsx:233-236` + `labels.ts:53`
- 修法：判斷 error.code、23505 → 「編號已存在」、23503 → 「外鍵錯誤」

#### S-006 RLS 沒檢查 capability
- policy 只比對 `workspace_id`、不檢查員工有沒有 `library.suppliers.read` capability
- 員工被砍權限後還是能讀寫
- 違反 CLAUDE.md「RLS 用 has_capability_for_workspace()」
- 證據：`supabase/migrations/20260510130000_phase2_remove_privilege_concept.sql:74-83`
- 修法：policy 改成 `workspace_id = ... AND has_capability_for_workspace('library.suppliers.read')`
- 風險：改完跑 login-api 測試（紅線 A）

### 🗑 髒級（3）

#### S-004 created_by 寫 auth.users(id)、不是 employees(id)
- 違反紅線 B
- 證據：`supabase/migrations/20251229160000_create_suppliers.sql:26` 沒 FK
- 修法：補 FK migration + 改 createEntityHook 傳員工編號

#### S-005 suppliers 表 RLS DISABLED
- 表上 RLS 被 `DISABLE ROW LEVEL SECURITY`、policy 寫了但沒生效
- 雙保險變單保險（只靠前端 entity hook 過濾）
- 證據：`supabase/migrations/20251229160000_create_suppliers.sql:33`
- 修法：新 migration `ENABLE ROW LEVEL SECURITY`、但要先修 S-006 確保 policy 對

#### S-007 form validation 太寬
- `submitDisabled={!formData.name}` 只擋名稱空、其他必填欄位沒檢查
- 證據：`SuppliersDialog.tsx:113`

## 共通模式（兩個模組都犯）

這些是「組織性」問題、不是單點 bug、修一個其他地方也要補：

1. **編號前端算、不上鎖** — E-005 + S-001 同源、應該全部改後端產碼（或走 entity hook 的 retry 機制）
2. **錯誤訊息沒翻譯層** — E-001 + E-008 + S-003 同源、應該寫個 `translateDbError(error)` helper、API 統一用
3. **儲存按鈕沒 disabled={loading}** — S-002 + 員工部分頁面、應該掃全站
4. **created_by 寫錯類型** — E-007 + S-004 同源（紅線 B）、entity hook 統一改
5. **舊 store vs 新 entity hook 混用** — E-003 + E-010 + E-006 都跟「員工模組沒搬到 entity hook」有關

## 意外發現

## Logan 的判斷

William、講白一點：

**今天測到的「漏洞百出」、不是真的整個系統爛。是兩個模組共犯三個重複的小錯**（編號、訊息、按鈕鎖）、加上員工模組沒搬到 entity hook 留的尾巴。

修法的優先順序我建議：

- **這週內把第一輪（5 個）清完**、你之後測 CRUD 就會順
- **第二輪（紅線 + 資安）下週做**、E-002 email rollback 一定要修、那是 prod-grade
- **第三輪當技術債、卡到誰再說**

不要一次全修。你只有一個身體、Carson 也不寫 code。一輪一輪推、每輪修完你親自測一次。

開始修之前我會再來問你：「先修第一輪、A 或 B 方案、哪一個？」你只要拍板、我就動。
