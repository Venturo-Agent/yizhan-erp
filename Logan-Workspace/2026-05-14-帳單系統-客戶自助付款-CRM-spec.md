## 一、為什麼要做（業務背景）

### 痛點
- 現況：收款單依附在「**訂單**」下、跟「**個別客戶**」沒 schema 強制關聯（雖然 `receipts.customer_id` 欄位已預留、但實際新建收款常常沒寫）
- 結果：未來做 CRM 看「**這個客戶歷年付了我多少錢**」沒辦法
- 而且：4 個團員一團、會計 / 業務每次手動核對「誰匯了多少」很累

### 解法（自動化鏈）
1. **訂單列表 → 開帳單**（給每個客人一張、各自 unique link）
2. **客戶用 link 看帳單 + 自助匯款**（先做匯款、未來擴充信用卡 / Line Pay）
3. **客戶填後五碼 + 金額 → 系統自動建「待確認」收款**
4. **會計手動對銀行對帳單 → 確認**（紅線：客戶不能直接 confirm）
5. **CRM 客戶 detail 自然撈到該客戶歷次交易記錄**

### 業務價值
- 會計從「**每張收款手動建**」→「**只要對帳**」(50% 工時下降)
- 客戶從「**業務員追著要錢 / 來回對後五碼**」→「**自助自動**」
- CRM 自動累積「客戶交易歷史」、可做未來行銷 / 老客戶分析

## 三、資料模型

### 新表：`invoices`

```sql
CREATE TABLE public.invoices (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id uuid NOT NULL REFERENCES public.workspaces(id),

  -- 關聯
  order_id uuid NOT NULL REFERENCES public.orders(id),
  customer_id uuid NOT NULL REFERENCES public.customers(id),
  member_id uuid REFERENCES public.order_members(id), -- 對應團員（可選）

  -- 金額
  total_amount numeric(20, 2) NOT NULL CHECK (total_amount > 0),
  paid_amount numeric(20, 2) NOT NULL DEFAULT 0 CHECK (paid_amount >= 0),

  -- 狀態 + token
  status text NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'partial', 'paid', 'overdue', 'cancelled')),
  public_token uuid NOT NULL DEFAULT gen_random_uuid() UNIQUE,
  token_expires_at timestamptz,

  -- 業務
  due_date date,
  notes text,

  -- 審計
  created_by uuid REFERENCES public.employees(id) ON DELETE SET NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_invoices_workspace ON public.invoices(workspace_id);
CREATE INDEX idx_invoices_customer ON public.invoices(customer_id);
CREATE INDEX idx_invoices_order ON public.invoices(order_id);
CREATE INDEX idx_invoices_token ON public.invoices(public_token);
CREATE INDEX idx_invoices_status ON public.invoices(status);
```

### `receipts` 加欄位

```sql
ALTER TABLE public.receipts
  ADD COLUMN invoice_id uuid REFERENCES public.invoices(id),
  ADD COLUMN bank_account_last5 text CHECK (bank_account_last5 ~ '^\d{5}$');

-- status 已有、加新值
-- 'draft' / 'confirmed' / 'refunded' (existing)
-- + 'pending_verify' (新、客戶填的待會計確認)
-- + 'rejected' (新、會計退回)

ALTER TABLE public.receipts
  DROP CONSTRAINT IF EXISTS receipts_status_check;
ALTER TABLE public.receipts
  ADD CONSTRAINT receipts_status_check
  CHECK (status IN ('draft', 'pending_verify', 'confirmed', 'refunded', 'rejected'));

CREATE INDEX idx_receipts_invoice ON public.receipts(invoice_id);
```

### Trigger：自動算 invoice.paid_amount

```sql
-- 當 receipts.status 改變或 amount 改變、自動 recalculate invoice.paid_amount
-- 只算 status='confirmed' 的、不算待確認 / 已退
CREATE OR REPLACE FUNCTION public.recalc_invoice_paid_amount()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
  v_invoice_id uuid;
  v_new_paid numeric(20, 2);
  v_invoice_total numeric(20, 2);
  v_new_status text;
BEGIN
  v_invoice_id := COALESCE(NEW.invoice_id, OLD.invoice_id);
  IF v_invoice_id IS NULL THEN RETURN NEW; END IF;

  SELECT COALESCE(SUM(actual_amount), 0) INTO v_new_paid
  FROM public.receipts
  WHERE invoice_id = v_invoice_id AND status = 'confirmed';

  SELECT total_amount INTO v_invoice_total
  FROM public.invoices WHERE id = v_invoice_id;

  v_new_status := CASE
    WHEN v_new_paid >= v_invoice_total THEN 'paid'
    WHEN v_new_paid > 0 THEN 'partial'
    ELSE 'pending'
  END;

  UPDATE public.invoices
  SET paid_amount = v_new_paid, status = v_new_status, updated_at = now()
  WHERE id = v_invoice_id;

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_recalc_invoice_paid
  AFTER INSERT OR UPDATE OR DELETE ON public.receipts
  FOR EACH ROW EXECUTE FUNCTION public.recalc_invoice_paid_amount();
```

### Status flow（receipts）

```
[客戶填表]
   ↓
pending_verify（待確認、🕒、不算進 invoice.paid_amount）
   ↓ 會計按「確認對帳」
confirmed（已確認、進帳）→ trigger recalc invoice.paid_amount
   ↓ 會計按「退回」
rejected（異常、附原因、客戶看到要重填）

[後台直接建]
   ↓
draft（會計手動建）
   ↓
confirmed
   ↓ 會計按「退款」
refunded
```

## 四、RLS（最關鍵、防外洩）

### invoices

```sql
ALTER TABLE public.invoices ENABLE ROW LEVEL SECURITY;
-- ⚠️ 不准 FORCE（紅線 A）

-- authenticated 走標準 workspace_scoped
SELECT setup_workspace_scoped_rls('invoices');

-- anon 角色：只能 SELECT 帶對 token 的單筆
CREATE POLICY "invoices_anon_by_token" ON public.invoices
  FOR SELECT TO anon
  USING (
    -- 必須帶對 token（API 路由會 set request.jwt.claims.invoice_token）
    public_token::text = current_setting('request.headers.x-invoice-token', true)
    AND (token_expires_at IS NULL OR token_expires_at > now())
    AND status != 'cancelled'
  );

-- anon 不開 INSERT / UPDATE / DELETE
-- 提交付款走 API route + service_role
```

### receipts（既有表、加 anon policy）

```sql
-- anon 只能 SELECT 該 invoice 的 receipts
CREATE POLICY "receipts_anon_by_invoice_token" ON public.receipts
  FOR SELECT TO anon
  USING (
    invoice_id IN (
      SELECT id FROM public.invoices
      WHERE public_token::text = current_setting('request.headers.x-invoice-token', true)
        AND (token_expires_at IS NULL OR token_expires_at > now())
    )
  );

-- anon 不直接 INSERT、走 API route
```

## 五、API Routes

### 公開（anon）

```
GET  /api/public/invoices/[token]
  → 拉該 invoice 詳情 + 既有 receipts 列表
  → header 帶 x-invoice-token、後端 set RLS context
  → 回傳：{ invoice, receipts, customer (basic info) }

POST /api/public/invoices/[token]/pay
  body: { amount, bank_account_last5, payment_date, notes }
  → Zod schema 驗證
  → 後端走 admin client（per-request 新建）
  → 建一張 receipt、status='pending_verify'
  → 回傳：建立的 receipt
  → rate limit: 同 IP / token 每分鐘 5 次
```

### 後台（authenticated）

```
POST /api/invoices  (建帳單)
  body: { order_id, splits: [{ customer_id, member_id, amount }] }
  → 守門：finance.invoices.write capability
  → 一次建 N 張 invoice（拆單）
  → 各自 public_token + due_date

POST /api/receipts/[id]/verify  (會計確認)
  → 守門：finance.receipts.verify capability
  → 必須是 status='pending_verify'
  → 改 status='confirmed' → trigger 自動 recalc invoice.paid_amount

POST /api/receipts/[id]/reject  (會計退回)
  body: { reason }
  → 守門：同上
  → 改 status='rejected' + 寫退回原因
```

## 七、資安 Checklist

### Token 安全
- ✅ public_token = UUID v4（128 bit、unguessable）
- ✅ 不准 incrementing ID
- ✅ 設過期（預設訂單結團後 30 天 / 或 due_date + 30 天）
- ✅ 取消訂單 → invoice.status='cancelled' → token 失效

### RLS
- ✅ invoices anon SELECT 必須帶對 token + 未過期 + 非 cancelled
- ✅ receipts anon SELECT 限於該 invoice
- ✅ anon 不開任何 INSERT / UPDATE / DELETE
- ✅ 不准 FORCE（紅線 A）

### API
- ✅ 走 admin client、per-request 新建（紅線 C）
- ✅ Zod schema 驗證 input
- ✅ rate limit（同 IP / token 1min 5 次）
- ✅ audit log（IP / user-agent / token / 操作）

### 輸入驗證
- ✅ 後五碼 regex `^\d{5}$`
- ✅ 金額 > 0 且 ≤ invoice.remaining
- ✅ 匯款日 ≤ 今日

### Abuse
- ✅ customer_id 從 token 撈、客戶不能改
- ✅ 客戶不能改 / 刪別人的 receipt
- ✅ confirmed 的客戶看不到「確認」按鈕
- ✅ 紅線 D：confirmed receipt 客戶絕對不能改 / 退、會計也只能走沖正

### 跨 workspace 滲透
- ✅ 跑既有 `tests/e2e/cross-tenant.spec.ts` 含新功能 case
- ✅ 拿 A workspace token → 查 B workspace invoice 應 reject
