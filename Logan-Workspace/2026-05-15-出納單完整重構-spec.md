# 出納單完整重構 + 通知抽象層

> William 2026-05-15 凌晨拍板 4 大方向 + 1 個新特殊功能 + 通知抽象層。
> 撤回 5/14 部分判斷（status filter 放寬、紅線 D 月結期）。
> 出納單不是新功能、是「結帳的最後一步」、上線必備。

## 為什麼

5/14 出納單品項級重構（羅根做的）只解了「品項拆分」、沒解：

1. 一張請款單多供應商時、要能單獨剔除某品項
2. 主管 review 出帳時想剔除某張 PR、不該「整批退單重來」
3. 公司手續費 mode 不明（現在 wizard 讓 user 選 equal/proportional、應該公司設定 default）
4. 手續費怎麼分回每筆 PR、要不要自動產手續費 PR、差額怎麼處理 — 全沒處理
5. 系統通知散刻（出納單剔除、團確認、HR 異動 ... 每個 feature 各刻一份）

### Phase 1：通知抽象層（最先、其他都靠它）

**檔**：`src/lib/notifications/system-notify.ts`

**signature**：
```ts
export interface SystemNotifyInput {
  workspace_id: string
  body: string
  recipient_employee_id?: string | null  // null = workspace 公開、有值 = 個人通知
  payload?: Record<string, unknown>      // 結構化資料（連結、ID、metadata）
  topic?: string                          // 'disbursement.item_removed' 等分類
}

export async function notifySystem(
  input: SystemNotifyInput
): Promise<{ message_id: string } | { error: string }>
```

**實作**：
1. admin client per-request（紅線 C）撈該 workspace 的 system_notice channel
2. 不存在 → 自動建一個（initial seed migration 應已建、保險起見 fallback）
3. insert channel_messages：
   - `channel_id` = system_notice channel
   - `message_type` = 'system'
   - `sender_employee_id` = NULL（系統觸發、不掛任何員工）
   - `body` = input.body
   - `recipient_employee_id` = input.recipient_employee_id ?? null
   - `payload` = input.payload ?? null
4. Supabase realtime 自動推到前端（channels 系統既有機制）

**caller 例**：
```ts
await notifySystem({
  workspace_id,
  body: `出納單 #${order_number} 剔除 1 筆請款（${supplier_name}）、已移至 ${next_date} 重新請款`,
  topic: 'disbursement.item_removed',
  payload: { disbursement_order_id, excluded_request_id, next_disbursement_date },
})
```

**工程量**：1-2 hr

### Phase 4：剔除自動 cascade 下週 + 發系統通知

#### 剔除流程

當 main step 把某張 PR / 某個品項從 wizard 拿掉（不勾）、wizard 流程**只影響本次出納單**、PR 本身**留在 pending**、下次 wizard 進來還會看到。

這是「**沒勾**」、不是「**剔除**」。

新需求是「**主動剔除**」：主管 review 後決定**這批不付**、PR 要明確標記「下一週再說」、本週 wizard 不再列出。

#### Schema

```sql
ALTER TABLE public.payment_requests
  ADD COLUMN IF NOT EXISTS next_disbursement_date date,
  ADD COLUMN IF NOT EXISTS excluded_from_disbursement_at timestamptz;

COMMENT ON COLUMN public.payment_requests.next_disbursement_date IS
  '剔除後的下一次預定出帳日。本次 wizard 列表會排除 next_disbursement_date > today 的 PR';
COMMENT ON COLUMN public.payment_requests.excluded_from_disbursement_at IS
  '上次被剔除的時間戳（audit 用）';
```

#### Wizard filter 升級

```ts
const filtered = items.filter(it => {
  // 既有條件
  if (linked) return false
  if (lockedInDoi) return false
  if (reqStatus !== 'pending') return false
  // 新條件：next_disbursement_date 在未來、本週不列
  const nextDate = it.payment_requests?.next_disbursement_date

  return true

```

#### 剔除 API

`POST /api/disbursement/exclude-item` body：
```json
{
  "payment_request_id": "uuid",
  "reason": "廠商有問題（optional、free text）",
  "next_disbursement_date": "2026-05-22"
}
```

server 邏輯：
1. require_capability `finance.disbursement.write`
2. update payment_requests set
   - `next_disbursement_date` = input.next_disbursement_date
   - `excluded_from_disbursement_at` = now()
3. recordApiAuditContext（reason 進 audit）
4. **呼叫 notifySystem**：
   ```ts

     workspace_id,

     payload: { payment_request_id, next_disbursement_date, reason },
   })
   ```
5. return 200 + updated PR

#### 下週日期演算法

`src/lib/disbursement/next-disbursement-date.ts`：

```ts
/**
 * 算下一個出帳日。
 *
 * - 公司有設 default_billing_day_of_week（譬如 4 = 週四）→ 下個週四
 * - 公司沒設 → fallback 加 7 天
 */
export function calculateNextDisbursementDate(
  fromDate: Date,
  weekday: number | null,
): Date {
  if (weekday === null) {
    const next = new Date(fromDate)
    next.setDate(fromDate.getDate() + 7)
    return next
  }
  const today = fromDate.getDay()
  const daysUntil = ((weekday - today + 7) % 7) || 7
  const next = new Date(fromDate)
  next.setDate(fromDate.getDate() + daysUntil)
  return next
}
```

### Phase 5：分攤算法

`src/lib/disbursement/fee-distribution.ts`：

```ts
export type FeeMode = 'average' | 'unified'

export interface DistributionInput {
  mode: FeeMode
  actual_fee: number                      // 銀行實扣
  unified_amount_per_item?: number        // unified mode 才用
  items: { id: string; subtotal: number }[]
}

export interface DistributionOutput {
  per_item_fees: { id: string; fee: number }[]  // 每筆 PR / item 分到的手續費
  overflow: number                         // unified mode 才會有差額（公司收入）
  total_collected: number                  // 公司實際收的總和
}

export function distributeFees(input: DistributionInput): DistributionOutput
```

#### Average mode

```
actual_fee = 15、items = 10 筆
per_item_fees = [1, 1, 1, 1, 1, 1, 1, 1, 1, 6] ← 最後一筆吃尾數
overflow = 0
total_collected = 15
```

實作：
```ts
const perItem = Math.floor(actual_fee / items.length)
const remainder = actual_fee - perItem * items.length
const fees = items.map((it, idx) => ({
  id: it.id,
  fee: idx === items.length - 1 ? perItem + remainder : perItem,
}))
return { per_item_fees: fees, overflow: 0, total_collected: actual_fee }
```

#### Unified mode

```
unified_amount_per_item = 30、items = 10 筆、actual_fee = 15
per_item_fees = [30, 30, 30, 30, 30, 30, 30, 30, 30, 30]
total_collected = 300
overflow = 300 - 15 = 285（公司收入、入到 overflow_account）
```

實作：
```ts
const fees = items.map(it => ({ id: it.id, fee: unified_amount_per_item }))
const total = unified_amount_per_item * items.length
return { per_item_fees: fees, overflow: total - actual_fee, total_collected: total }
```

#### Edge case：unified 公司虧

如果 `unified_amount_per_item * items.length < actual_fee`、公司虧錢。
**Wizard fill-fee step 偵測這 case、跳 warning**（不擋、user 可繼續、但要明確警告）。
overflow 變負數時不開「收款單」、改開「公司支出單」（v2 處理、Phase 6 還沒到）。

**工程量**：1 天（含 unit test）

### Phase 6：手續費自動產生請款單 / 差額收款單

#### Average mode 出帳

batch-create transaction 加一筆「手續費 PR」：
- supplier = 銀行（譬如該 batch.from_bank_account 對應的銀行）
- amount = actual_fee（譬如 15）
- description = `跨行手續費 / 出納單 #DO-2026-05-14 / 10 筆`
- 自動 link 到 disbursement_order

或者更簡單：手續費**不開 PR**、直接寫進 `disbursement_orders.fee_amount`、出納單列印報表時顯示。

> 決策：先用「不開 PR、寫進 disbursement_orders.fee_amount」、避免污染請款單列表

#### Unified mode 出帳

batch-create transaction 加：
1. 每筆 PR 的 fee_amount = unified_amount_per_item（譬如 30）
2. 出納單 fee_amount = unified_amount_per_item × items.length（公司向客戶 / 業務員收的總額）
3. **自動產生一張「收款單」**：
   - receipt_type = '手續費差額入賬'
   - amount = overflow（譬如 285）
   - account_id = company.transfer_fee_overflow_account_id
   - description = `出納單 #DO-2026-05-14 手續費差額入賬（公司收 300 / 銀行實扣 15）`
   - related_disbursement_order_id = 該 DO 的 id
   - status = 'confirmed'

#### 跟現有 audit 對齊

- batch-create route 已 record audit context、加 reason `'unified fee overflow'`
- 自動產生的收款單跟 disbursement_order 用同一個 audit_batch_id 串起來、可追溯

**工程量**：1-2 天
