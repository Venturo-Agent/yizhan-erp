# API 守門普及率盤點（C）

> [!info] 為什麼盤這份
> 老大實測時撞到 `401 / 403 / 200`、要能立刻判斷「是預期還是 bug」。這份是 58 個 API route 的守門矩陣 — 每個 route 用了哪些守門、哪些是合理 auth-only、哪些是特殊驗證、哪些可能漏。

---

## 一、概覽統計

掃描範圍：`src/app/api/**/route.ts`、共 **58 個 endpoint**。

| 守門機制 | 普及率 | 數量 | 說明 |
|---|---:|---:|---|
| `requireCapability(...)` | 72.4% | 42 / 58 | HR 職務權限、依 capability key 守 |
| `recordApiAuditContext(...)` | 34.5% | 20 / 58 | 寫 audit log |
| `withAudit(...)` | **0%** | 0 / 58 | 新版 audit wrapper、**全專案無人用** |
| `getServerAuth()`（auth-only） | 32.8% | 19 / 58 | 只驗登入態、不查 capability |
| 其他 secret 驗證 | 3.4% | 2 / 58 | `CRON_SECRET` / `verifyLineSignature` |

> [!warning] withAudit 普及率 0%
> 老大上一輪盤點以為是 0%、現在實測完還是 0%。沒人用、可能設計做了沒用、或是新規範還沒推。**不是 regress、是還沒落地**。

### 2-3 auth-only（getServerAuth、不走 capability、9 個）

撞 `401` = 沒登入、合理。撞 `200` = 任何登入員工都能用、**確認業務上是否該擋**。

| Route | 註解（部分有寫在 code header） |
|---|---|
| `airports` | reference 表、auth-only 合理 |
| `amadeus-totp` | header 註：員工自己用 Amadeus 票務 TOTP、登入即可 |
| `amadeus-totp/current` | 同上 |
| `amadeus-totp/setup` | 同上 |
| `auth/change-password` | header 註：員工改自己密碼、登入即可（不需 capability） |
| `auth/reset-employee-password` | header 註：HR 重設員工密碼、用 admin 流程驗證、本身已守 |
| `fetch-image` | image proxy、auth 防外部濫用 |
| `log-error` | 前端錯誤回報、登入即寫 |
| `storage/upload` | 一般員工上傳檔案、auth 即可 |

## 三、實測時的 401/403 判斷指南

### 撞 401 → 沒登入
- 重新登入再試
- 例外：cron / line webhook 撞 401 = secret / signature 對不上、**正常防外部呼叫**

### 撞 403 → 有登入、capability / feature 不給過

判斷流程：
1. **路由是 `/workspaces/*` 或 `/cis/*`？** → `PLATFORM_CAPABILITY_ROUTES`、要 `platform.is_admin` capability
2. **路由對應的 module 在 workspace_features 沒開？** → 例如 accounting / office 是 premium、新建 workspace 預設關
3. **路由對應的 module 在 role 沒給 tab 權限？** → `useMyCapabilities().canReadAnyInModule(moduleCode)` 全空
4. **API 直接 403？** → 該 endpoint 用了 `requireCapability(key)`、role 沒給該 key

### 撞 200 但畫面空白 → 不是守門問題、是資料/UI 層面

## 四、已知合理 / 不合理的「沒 capability」

### 合理（業務需求）
- `auth/change-password`：自己改自己、不需 capability
- `auth/reset-employee-password`：內部用 admin 流程驗證
- `amadeus-totp/*`：員工自己用、登入即可
- `airports`：公開 reference 資料
- `fetch-image`：image proxy、不涉業務
- `log-error`：前端錯誤回報

### 待確認（建議補 capability）
- ⚠️ `settings/env`：寫環境變數設定、極敏感、目前只 `requireCapability('settings.env.write')` 沒加 audit
- ⚠️ `users/[userId]/role`：改員工 role、缺 audit log
- ⚠️ `workspaces/[id]` PUT/DELETE：改 workspace 設定、auth-only 略寬（內部有 `auth.data.workspaceId === id` scope check、實際 OK 但 audit 該補）
- ⚠️ `auth/sync-employee`：登入時建/更新 employee record、缺 audit
- ⚠️ `tenants/create`：onboarding 註冊新租戶、auth-only + 內部 admin check、**實測時老大會走這個 endpoint、撞錯先看 admin role 有沒有掛**
