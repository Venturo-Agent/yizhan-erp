# Part 2：分店守則（`yizhan-erp/CLAUDE.md`）

## 優先順位

**原文**：
> 資安 #1 → 效能 #2 → SSOT #3
> - 資安第一：洞 = 客戶資料外洩、商業終結
> - 效能第二：SaaS 化讀取量 = Supabase 成本
> - SSOT 第三：不是不重要、但跟前兩者衝突時讓位

**譬喻**：
餐廳優先順序：**食安 > 翻桌率 > 菜單統一**。
食安出事 = 關門大吉（資安 = 客戶資料外洩 = 商業終結）。
翻桌率慢 = 賺不到錢（效能慢 = Supabase 帳單暴漲）。
菜單兩家分店不一致 = 困擾、但不會關門（SSOT 破 = 麻煩、但不致命）。

✅ **保留**：這個優先順位邏輯永遠對。

---

## 五大方向

### 方向 1：路由連結（HR / 租戶 / 顆粒度）

**原文**：
> 動到頁面 / 路由前、確認三個 layer 都對齊：
> - 路由本身（Next.js app/ 目錄）
> - HR 顆粒度（role_capabilities、module.tab.action 三段式 code）
> - 租戶開通（workspace_features、租戶有沒有買這個功能）

**譬喻**：
新增一個「會議室」要三件事都對齊：
1. **門牌**（路由）— 大家走得到
2. **每個員工有沒有鑰匙**（HR 權限）— 不是誰都能進
3. **這家分店有沒有買會議室服務**（租戶開通）— 沒買的分店看不到

少一條就會「點下去進不去」「沒權限的人看到了」「沒買的分店也看到」。

✅ **保留**：基本架構、保留。

---

### 方向 2：修改程式時的前後端影響（GitNexus）

**原文**：
> 動 symbol 前必跑 impact。不准 grep 替代、call graph 比文字搜尋準。

**譬喻**：
動一根水管前、先查「這根水管接到哪幾個房間」 — 不能只看牆面（grep）、要看完整管路圖（call graph）。
不然動了一根、結果別的房間沒水、被罵。

⚠️ **要改**：GitNexus 我這次 session 沒用、因為老大說「不要用 MCP 連舊資料庫」、加上索引可能 stale。要評估 GitNexus 是不是還適用 yizhan-erp、還是已經過時。

### 方向 3：介面 UI 架構

**原文**：
> 之後會整體重寫、本節暫不規範。先沿用既有：列表用 EnhancedTable / ListPageLayout、Dialog 必設 level={1|2|3}、莫蘭迪色系 morandi-*。

**譬喻**：
裝潢還沒最終定案、先按目前的標準（列表用同款桌子、彈窗有三個層級、配色用莫蘭迪色）做、不要自己亂創。

✅ **保留**：直到 UI 大改前都適用。

---

### 方向 4：RLS 資安

**原文**：
> - RLS policy 用 has_capability_for_workspace()、不准散刻 capability check
> - Admin client 必 per-request、不准 singleton
> - API route 守門：require-capability + RLS 雙保險
> - 動 RLS migration 前跑 tests/e2e/login-api.spec.ts

**譬喻**：
銀行金庫的「**個人隱私柵欄**」（RLS）：
- 用同一個身分檢查機（has_capability_for_workspace）、不要每個櫃台自己刻一個
- 銀行行長拿的萬能鑰匙（admin client）每次借出後就鎖回保險箱（per-request、不共用）
- API 進入點要兩道鎖（capability + RLS）
- 動隱私柵欄前先測登入（不然動完所有人登不進來）

✅ **保留**：技術正確性、必留。

---

### 方向 5：資料讀取連線（含防連點）

**原文**：
> - Client cache：SWR、revalidateOnFocus: false、dedupingInterval: 5min
> - 列表預設 20 筆、分頁固定 15 筆、不給「每頁筆數」選擇器
> - 列表搜尋 server-side
> - 防連點：所有「儲存 / 刪除 / 確認」按鈕必須 disabled={loading}

**譬喻**：
- 客戶按按鈕後 5 分鐘內再按、不重新查（節省 Supabase 查詢費）
- 列表一頁固定 15 筆（每頁讀越多錢越貴、不給客戶調）
- 客戶按「儲存」按鈕、按下去就鎖住（防雙擊重複寫入）

✅ **保留**：成本控制 + 防呆、必留。

## 紅線（最嚴重等級）

### 紅線 0：絕對不准刪 William 的檔案 / 資料

**原文**：
> 檔案層：不准刪 src/ 下任何既有檔、除非 William 明說「砍掉 X」
> 不准刪 vault（brain/wiki/）下任何檔案
> 不准 rm -rf、git clean -f、git checkout .
> 重構時用「新增 + 取代引用」、舊檔留著由 William 決定砍不砍
>
> 資料層：
> 不准 DROP TABLE 任何有資料的表（特別是：payment_requests / disbursement_orders / receipts / employees / customers / tours / orders / quotes / 任何 HR 相關表）
> 不准 DELETE FROM 既有資料、不准 TRUNCATE
> 不准寫不可逆的 destructive migration（DROP COLUMN with data、ALTER COLUMN type 會 silent truncate）

**譬喻**：
這條是「**不准動古董花瓶**」 — 因為舊 venturo-erp 有 16 員工 / 385 客戶 / 1425 請款單**真實營運資料**、刪了沒救。

🔴 **要改**：yizhan-erp **是新專案、新資料庫、沒真實營運資料**。這條紅線的「保護古董花瓶」理由不適用了。
- 「不准刪 src/ 既有檔」 → 可以放寬：可以砍舊 code（像我今天砍 features/suppliers/）
- 「不准 DROP TABLE」 → 可以放寬：schema 還在演化、可改可重寫
- 「不准 DELETE FROM」 → 可以放寬：新資料庫
- 但仍要保留：「**做之前要說清楚為什麼**」、不能無腦砍

### 紅線 2：審計欄位 FK 一律指 employees(id)

**原文**：
> front-end currentUser?.id 拿的是 employees.id、不是 auth.users.id。FK 指 auth.users insert 必炸。
> created_by / updated_by / performed_by → REFERENCES public.employees(id) ON DELETE SET NULL

**譬喻**：
紀錄「誰建這筆」的時候、要寫**員工編號**（E001）、不是寫 Supabase 帳號編號（uuid）。
前端傳的是員工編號、如果系統設計成「要 Supabase 帳號編號」、寫入就會失敗。

✅ **保留**：技術正確性紅線。

⚠️ 但盤查發現有 12 處違反（指 auth.users 不指 employees）— 老大說 Supabase 還沒建表、之後重寫時用對的版本。

## Build / Commit 規則

**原文**：
> npm run type-check    # commit 前必過
> npm run lint          # 不能新增 console.log
> ./scripts/check-standards.sh --strict   # 憲法守門
> NEVER --no-verify / --no-gpg-sign 跳過 hook
> NEVER 新增 as any / : any
> NEVER push 有 type error 的程式碼
> 不要 amend 已 push 的 commit、永遠新增 commit

**譬喻**：
出貨前三道檢查（編譯 / 規範 / 憲法）、缺一道就退貨。
不准用偷雞方法跳過檢查。
不准把「箱裡有破洞的貨」推上架。
不准回頭改已經出貨的箱子（amend pushed commit）、有問題就出新一箱蓋上去。

✅ **保留**：協作紀律、永遠對。
