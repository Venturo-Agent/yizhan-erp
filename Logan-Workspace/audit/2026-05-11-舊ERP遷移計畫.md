## 3. 三個 remap 規則

### 3-1 workspace_id remap

```
舊 CORNER workspace_id → 新 ERP 漫途 workspace_id (b2222222-2222-2222-2222-222222222222)
舊 YUFEN  workspace_id → 新 ERP 漫途 workspace_id (b2222222-2222-2222-2222-222222222222)
```

也就是兩個舊 workspace 的資料**合併**到新 ERP 漫途 workspace 下。

> ⚠️ 注意：customers / tours 等表本來有 workspace 隔離。CORNER 跟 YUFEN 客戶可能有重複（同一個客戶在兩家旅行社都報名過）— 要 dedupe（推薦：keep 最新一筆、merge 訂單到留下的那筆）。

### 3-2 employee user_id remap

| 舊 ERP user_id | 新 ERP user_id | 員工 |
|---|---|---|
| 舊 CORNER 簡瑋廷 / Carson / 黃亞萍 / 林熙捷 user_id | 全 remap → William 的 new user_id | 角落員工不搬、created_by 給 William |
| 舊 YUFEN 范詩屏 user_id（ada@gmail.com）| 新 ERP 范詩屏 user_id（ada@yufen.com）| 已存在、對應 |
| 舊 YUFEN 陳建文 / 王卉穎 / 范竹 / 陳佩雯 / 陳相君 user_id | 新建 user_id（{emp_no}@yufen.com）| 5 個新建 |

舊資料的 `created_by` / `updated_by` / `controller_id` / `leader_id` 等 uuid 欄位、按上表 remap。

### 3-3 PK 不撞策略

舊 ERP 的 PK uuid 直接用、不重新 generate（避免 FK 對不上）。
新 ERP 既有 demo 資料先砍光（**Step 0** 做）、避免 PK 撞。

## 5. schema diff 預期問題

需做之前先跑（task #15、本卡未列）：

```sql
-- 對每張要搬的表、對比新舊兩邊 column 差異
SELECT 'old' AS src, column_name, data_type FROM information_schema.columns 
  WHERE table_schema='public' AND table_name='customers'
UNION ALL
SELECT 'new', column_name, data_type FROM ...（新 ERP 對應）
ORDER BY column_name, src;
```

預期會撞的差異：
- `customers.passport_image_url` 舊存 base64、新 schema 可能改成只接 URL
- `tours.tour_service_type` 新加 outsource / visa / esim（舊沒、不是問題）
- `employees.email` 新強制 NOT NULL（舊御風 5 個沒 email、本計畫已給）
- `orders` 新加 `controller_id`（舊沒、要查 default）
- `customers / suppliers` 新可能有 audit 欄位（created_by_workspace_id 之類）— 從本次 shared_data 學到
- jsonb 欄位 schema 內的 key 改變（很可能漂、要 case by case）
