## 重寫時必合併的技術點（不是業務紅線、是技術正確性）

### 技術紅線 A：workspaces 表絕不能 FORCE RLS

**為什麼**：
- FORCE RLS 會讓 service_role（系統行長鑰匙）也被擋
- 登入 API 用 admin client 查 workspace code 拿到空
- **結果：所有人登入失敗**
- 2026-04-20 真的出過事故、4/20 那次是把 workspaces 表設成 FORCE → 全員無法登入

**規則**：
- `workspaces` 表 RLS 可以 ENABLE、但**永遠 NO FORCE**
- 任何動 RLS policy 的 migration、apply 前必先跑 `tests/e2e/login-api.spec.ts`
- 歷史 fix migration：`20260420d_fix_workspaces_force_rls.sql`

**新版 CLAUDE.md 建議寫法**（白話）：
> ### 技術紅線 A — workspaces 表的 RLS 不能設 FORCE
> RLS 是「個人隱私柵欄」、FORCE 是「強化版」（連系統管理員都被擋）。但是登入流程要用系統管理員身分查 workspace 資料、被擋 = 全員登入失敗。
> - workspaces 可以開 RLS（一般版）、不能開 FORCE
> - 動 RLS policy 的 migration、必先跑 `npm run test:e2e -- login-api.spec.ts`

### 技術紅線 B：審計欄位 FK 必指 employees(id)、不是 auth.users(id)

**為什麼**：
- 前端 `currentUser?.id` 拿到的是 `employees.id`（員工編號、像 E001）
- 不是 `auth.users.id`（Supabase Auth 帳號的 uuid）
- 如果欄位 FK 設成 `REFERENCES auth.users(id)`、寫入 `currentUser.id` 會找不到對應的 `auth.users` row → **insert 失敗**

**規則**：
- 審計欄位（紀錄誰操作的）：`created_by` / `updated_by` / `performed_by` / `uploaded_by` / `locked_by` / `last_unlocked_by` / `deleted_by` → `REFERENCES public.employees(id) ON DELETE SET NULL`
- 例外（合規、可指 auth.users）：`user_id` / `sender_id` / `friend_id`（這些 row 本身就是 Supabase 用戶、不是員工操作紀錄）
- 前端寫入：`created_by: currentUser?.id || undefined`（不是 `|| ''`、空字串會炸）

**新版 CLAUDE.md 建議寫法**（白話）：
> ### 技術紅線 B — 紀錄「誰建這筆」要寫員工編號、不是 Supabase 帳號編號
> 員工編號（E001）跟 Supabase 帳號編號（uuid）是兩個不同的 ID。前端拿到的是員工編號、所以資料庫的「誰建的」欄位要設成「指向員工表」、不能指向「Supabase 帳號表」。

**現況**：
- 已盤查：12 處 migration 違反此原則（emails / tour_control_forms / image_library / folders / files / tour_documents / traveler_conversations / file_audit_logs）
- 老闆說：Supabase 還沒寫表格、之後重寫時用對的版本

### 技術紅線 C：Admin client 必 per-request、不准 singleton（共用實例）

**為什麼**：
- Admin client = 系統行長鑰匙（service_role、繞過 RLS）
- 如果做成 singleton（全部 request 共用同一把）、跨 request 會殘留前一個 request 的 context
- **結果：跨工作區資料洩漏**

**規則**：
- `src/lib/supabase/admin.ts` 必須每個 request 重新建立
- 不准 export 一個全局 admin variable
- 函數名應該是 `getSupabaseAdminClient()`、每次 call 都新建

**新版 CLAUDE.md 建議寫法**（白話）：
> ### 技術紅線 C — 系統管理員鑰匙每次借用後鎖回保險箱
> 每個請求進來、新建一把鑰匙、用完丟掉。不准放在保險箱外面共用、會混到別人的資料。
