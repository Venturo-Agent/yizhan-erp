# 公共資源管理（Shared Data）規格

> 全平台共用「**代號 / 內容資料**」reference 層、開了 `shared_data_*` feature 的 workspace 可維護、其他 workspace 唯讀使用。
>
> 跟「租戶管理」（workspaces feature）一樣是通用 feature、不是漫途專屬概念。漫途剛好現在是唯一開了這 feature 的 workspace、未來其他客戶開了就一樣有。

## 1. 範圍：6 樣資源、分兩類

| 類別 | 包含 | 邏輯 | UI 範圍 |
|---|---|---|---|
| **代號類**（codes）| 銀行 / 國家 / 機場 | 全世界通用代號、客觀事實 | 全 workspace 看得到、找不到能新增 |
| **內容類**（content）| 景點 / 飯店 / 餐廳 | 累積有偏好的內容 | 本期只給開了 `shared_data_content` feature 的 workspace 用（目前只有漫途）、其他 workspace 暫不出現在行程編輯器、未來串機器人時再開 |

### 2-4 砍 workspace_id（內容類 3 張）

cctk 的 B2 commit 已經處理了 `ref_airports` 上的 `trigger_auto_set_workspace_id`、本次延續同邏輯：

```sql
-- 砍 trigger（如果有）
DROP TRIGGER IF EXISTS trigger_auto_set_workspace_id ON public.attractions;
DROP TRIGGER IF EXISTS trigger_auto_set_workspace_id ON public.hotels;
DROP TRIGGER IF EXISTS trigger_auto_set_workspace_id ON public.restaurants;

-- 砍 workspace_id 欄位 + 對應 index

ALTER TABLE public.hotels      DROP COLUMN IF EXISTS workspace_id;
ALTER TABLE public.restaurants DROP COLUMN IF EXISTS workspace_id;
```

### 2-5 RLS 改全讀

6 張表：

```sql
-- SELECT 全 authenticated 通
CREATE POLICY shared_data_select ON public.{table}
  FOR SELECT TO authenticated USING (true);

-- INSERT/UPDATE/DELETE 看 capability + workspace 開了對應 feature
-- 用 has_capability_for_workspace(...) helper（venturo-aierp 已有）
CREATE POLICY shared_data_write ON public.{table}
  FOR INSERT TO authenticated WITH CHECK (
    has_capability_for_workspace(auth.uid(), 'shared_data.{type}.write')
  );
```

## 3. API 守門（代號類 3 個）

### 3-1 API endpoints

| Endpoint | Method | 守門 capability | 動作 |
|---|---|---|---|
| `/api/banks` | POST | `shared_data.banks.write` | upsert ref_banks + 寫 audit |
| `/api/countries` | POST | `shared_data.countries.write` | upsert ref_countries + 寫 audit |
| `/api/airports` | POST | `shared_data.airports.write` | upsert ref_airports + 寫 audit |

> 既有 `/api/airports` 守門是 `hasAdminCapability`（系統主管 / 漫途獨享）、本次改成 `requireCapability('shared_data.airports.write')`。

### 3-2 不做「自動產 todo」（William 拍板更正、2026-05-11）

原規格寫「找不到 → 自動 INSERT + 產 todo 給漫途 review」是錯的設計、違反「workspace_features + role_capabilities」兩層守門規範。

**正確設計**：12 條 capability **只給漫途 admin role**、其他 workspace 沒有 `*.write` capability。
- 漫途自己人新增 → 自己 review、不需要 todo 通知自己
- 客戶 workspace 行政人員找不到 → 走「漫途幫他開 feature」既有流程、不需要 escape hatch

**未來想開放給某客戶 workspace**：
1. 漫途幫他 INSERT `workspace_features (workspace_id=客戶id, feature_code='shared_data_codes', enabled=true)`
2. 客戶 workspace 自己 admin 把 `shared_data.*.write` capability 分配給他想給的 role
3. 完全走既有規範、不需要任何特殊邏輯
