## 一、開發方向（高層 / 為什麼）

漫途 yizhan-erp 是「平台供應商 + 自家旅行社 dogfood」雙身份的 SaaS、定位上跟 Salesforce 那種純平台不同。所以：

1. **產品優先給漫途自己跟角落用、能上架賣才考慮其他**
2. **不做 over-engineering** — 沒有 Carson 跟 William 自己用不到的功能、未來才加
3. **品質凌駕進度** — William 在意的「重複實作、各自為政、登入三四個版本」這種事不能再發生

## 二、開發方針（中層 / 怎麼做）

### 1. 一條路、不開分支

每件事**只有一個寫法**、不允許「abc 三套都能用」：
- 列表頁拿資料 → createEntityHook 或 SWR hook、不准自己包 fetch + setState
- API 守門 → requireCapability、不准自己寫 if (isAdmin)
- Dialog → 用統一 Dialog component、不准抄一份新的
- Table → 用 EnhancedTable、不准 hard-code <table>

如果現有路不適用、**先改現有的**、不要開新路。

### 2. 沒有特權概念

整個系統不存在「平台方 vs 租戶方」、所有 workspace 平等：
- 漫途也是一個 workspace、不寫死
- 跨 workspace 能力（如租戶管理）靠 workspace_features 控制、不靠 user role 特權
- code 不准出現 if (isAdmin) / requireCapability('platform.*') / workspace.type === 'platform_owner'

### 3. 規範跟自動化檢查綁在一起

寫一條規則就要有 ESLint rule 或 pattern detector 在 build 時擋下違規。沒辦法自動驗證的規則就不寫進規範、改進 CLAUDE.md 當哲學原則。

死文件半年就過時、自動化才是真的守門。

### 4. 重複實作不能再發生

每次新功能完成後跑「sweep」、grep 整個 codebase 找「同樣的事各自寫」的 pattern。發現重複就抽。

## 三、開發規範（細節 / 具體 do / don't）

### 訪問控制（資安）

- 所有路由都要走 `workspace_features`（workspace 有沒有買）+ `role_capabilities`（user 有沒有權）兩道閘門
- 跨 workspace 操作必走額外 capability（例如 workspaces.read）+ 檢查 caller workspace 開了對應 feature
- DB 層 RLS：(Phase 2) 全部改吃 `has_capability_for_workspace(...)` + features 檢查、不允許 OR is_super_admin() bypass

### 資料存取

- 列表 / detail 一律走 createEntityHook 或 useXxx SWR hook
- mutation 散在 page 內 OK、不算重複
- ref_* 表（金融機構、機場、國家等）是「共用資料層」、所有 workspace 都能讀、UI 不開放寫、要改用 migration

### 元件 / UI

- 所有 dialog 用 src/components/ui/dialog
- 所有 table 用 EnhancedTable
- 列表頁用 ListPageLayout、內容頁用 ContentPageLayout
- 中文字串不准 hardcode、走 _constants/labels.ts
- forbidden classes（自定義 css）不准用、有 ESLint rule 擋

### 程式碼

- type-check + lint 必過、不准 commit
- 一個 page.tsx 超過 1000 行就要拆 sub-component
- 三段重複還忍、四段就抽
- dead code 加 _ 前綴只是過渡、整段 unused 就直接砍
- commit message 寫 why、不寫 what（diff 自己會說 what）

### 文件

- code repo 的 docs 只留 SECURITY.md（資安鐵律 + 共用資料層）
- 規範、開發進度、決策紀錄寫進黒羽 vault（CornerVenturo-Vault/01-Active-Projects/ERP/）
- CLAUDE.md 是 Logan 工作 SSOT、放鐵律 + 索引指 vault
