# Phase 7：單張多銀行整併 + 砍付款方式選擇

> William 2026-05-16 拍板：同一期出帳整併成**一張** disbursement_order、內部按銀行分類。撤回 5/15 「N 帳戶 = N 張單」設計。順手砍掉 wizard 選付款方式（PR 階段已選）。整併方案結構性根除剛剛踩到的撞號 bug。

## 起源（5/16 踩到的）

新增出納單時選了兩個出帳帳戶、系統產出兩張 `DO260521-001` 撞號（同 code、同 batch_uuid、不同 bank_account_id）。本來預期 second batch 拿 `DO260521-002`、實際拿 `DO260521-001`。已 SQL 撤銷（見「已撤銷測試資料」段）。

撞號 root cause：
- `generate_disbursement_no` RPC 用 `pg_advisory_xact_lock`、transaction-level、commit 後釋放
- batch-create loop 內 sequential `await` 兩次、雖然第一筆 INSERT 已 commit、但 PostgREST connection pool 仍可能用 stale snapshot、SELECT max(code) 撈不到剛 commit 的那筆
- `disbursement_orders.code` **沒 UNIQUE constraint**（只 PK on id）、DB 不擋
- 結果：兩次 RPC 都回 `DO260521-001`

修法決策：
- **不修 race**、改重構流程：整併一張 → 每次新增只 call 一次 RPC → 結構性根除
- UNIQUE constraint 列 follow-up（保護未來、見 Phase 7.5）

## 為什麼這樣改

| 改之前 | 改之後 |
|---|---|
| 一張單 = 一個帳戶（disbursement_orders.bank_account_id 是 FK） | 一張單 = 一期出帳（內含 N 個帳戶 group） |
| 多選帳戶 = 拆 N 張單、N 個編號 | 多選帳戶 = 1 張單、內部按銀行 group |
| 付款方式在 wizard header 選 | 付款方式由每筆 PR 帶過來、wizard 不選 |
| 撞號 race 條件存在 | 結構性消除 |

業務含義：
- 「這一期出帳」對會計是一個概念、不該被技術層的「銀行帳戶」拆碎
- 一張單 = 一個流程 = 一個 PDF = 一個審批
- 付款方式（電匯 / ATM / 現金等）每筆 PR 已選定、wizard 重選會 override、UX 多餘

## generate_disbursement_no race 根因（技術附錄）

RPC 實作（`supabase/migrations/20260424100000_generate_request_and_disbursement_no_advisory_lock.sql`）：

```sql
CREATE FUNCTION generate_disbursement_no(p_workspace_id uuid, p_disbursement_date date)
RETURNS text AS $$
DECLARE ...
BEGIN
  v_prefix := 'DO' || to_char(p_disbursement_date, 'YYMMDD');
  v_lock_key := abs(hashtextextended(p_workspace_id || ':disbursement:' || v_prefix, 0));
  PERFORM pg_advisory_xact_lock(v_lock_key);  -- ← xact-level、commit 後 release

  SELECT max(code) INTO v_max_code FROM disbursement_orders
    WHERE workspace_id = p_workspace_id AND code LIKE v_prefix || '-%';
  ...
END;
$$;
```

問題：
- `pg_advisory_xact_lock` 在 commit 時釋放、下次 call（新 transaction）會重新拿 lock
- 兩次 call 之間若 INSERT 已 commit、新 transaction 的 SELECT 理論上 read-committed 看得到
- 但 PostgREST 用 connection pool、不同 connection 不一定立即看到對方 commit（取決於 isolation level / pool 狀態）
- 加上 `disbursement_orders.code` 沒 UNIQUE constraint、DB 不擋撞

整併方案下：
- 一張 DO → 只 call 一次 generate_disbursement_no → race 沒機會發生
- Phase 7.5 加 UNIQUE 後、即使未來再有類似流程也擋住
