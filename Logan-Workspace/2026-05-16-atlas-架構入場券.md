## 4. 六層架構（核心規則）

> 每個 API 請求必須過 6 道門。過去的 bug 幾乎都來自「有人只蓋了 2-3 道門」。

```
你 → 客戶大門 → 樓層大門 → 部門大門 → 房間鎖 → 物品櫃 → 收據
      L1          L2          L3          L4        L5        L6
```

### L0：系統基本假設

- **所有 workspace 平等**：沒有 platform_owner / agency 之分
- 跨 workspace 能力 = 「我這 workspace 買了 `tenants` feature」
- **沒有** `if (isAdmin) { 跳過所有檢查 }` 這種硬編碼特權

### L1：租戶 Feature Gate

**守門**：這家公司有沒有買這個功能？

```sql
workspace_features (workspace_id, feature_code, enabled)
```

- UI：`<ModuleGuard module="feature_code">` 路由守門
- sidebar `requiredPermission` 對齊
- 禁止：`if (workspace.code === '漫途')` 硬編碼 → 走 `workspace_features`

### L2：角色 Capability

**守門**：這個員工有沒有這個能力？

```ts
// 命名規則：module.tab.action
'hr.employees.read' / 'finance.payments.write' / 'cross_branch.read'
```

- API 守門：`requireCapability(req, 'module.tab.write')`
- DB function：`has_capability_for_workspace(workspace_id, capability)`
- UI：按鈕 disabled by capability，Tab 隱藏 by capability
- 5 個 SSOT（漏一個就壞）：
  1. 路由 `src/app/(main)/<module>/`
  2. `src/lib/permissions/capabilities.ts`
  3. `src/lib/permissions/module-tabs.ts`（HR UI 列出可勾項目，**5/12 踩過坑**）
  4. `src/lib/permissions/features.ts`
  5. seed migration（workspace_features + role_capabilities）

### L3：三維 Org Scope

**守門**：員工屬哪個品牌 / 分公司 / 部門？

```
workspace 下有：brands → branches → departments → employees
```

- DB function：`scope_visible('module', row_id)` 統一守門
- 禁止：`sales_id = me` 或 `created_by = me` 散刻進 RLS USING
- 跨界能力：有 `cross_branch.read` / `cross_department.read` 才能跨

### L4：狀態守門

**守門**：這筆資料現在能不能改？

```sql
is_row_editable('module', row_id)  -- 走這個，不要散刻
```

- `orders.payment_status = 'paid'` → 不能改
- `payment_requests.status = 'approved'` → 不能改
- `journal_vouchers.is_locked = true` → 不能改
- RLS UPDATE / DELETE：`scope_visible AND is_row_editable` 雙守門
- 禁止：已封存狀態能解鎖（技術紅線 D，GAAP 原則）

### L5：DB RLS

**守門**：資料庫層最後一道，就算應用層失職也守住

3 個標準 procedure（走這個，不散刻 4 條 CREATE POLICY）：

```sql
CALL setup_workspace_scoped_rls('<table>');          -- 業務主表
CALL setup_inherited_rls('<table>', '<parent>', '<parent_id>');  -- 子表
CALL setup_join_table_rls('<table>', '<employee_col>');  -- 員工 join 表
```

- 禁止：`workspaces` 表 FORCE RLS（技術紅線 A，打斷登入）
- 禁止：ENABLE RLS + 0 policy（全擋）
- 禁止：policy 寫 `OR workspace_id IS NULL`（洞）

### L6：防呆 / SSOT

**守門**：防連點、防編號競態、統一錯誤語言

| 場景 | 走這個 | 禁止這個 |
|------|--------|----------|
| 編號生成 | `@/lib/codes.ts`（11 種） | 前端自己算 D + Date.now() |
| 錯誤翻譯 | `dbErrorResponse(error)` | `return { error: err.message }` |
| 連點防護 | `<FormDialog loading={loading}>` | 按鈕沒 disabled={loading} |
| audit trail | `recordApiAuditContext(client, {...})` | 無 audit |
| created_by FK | `employees(id)` | `auth.users(id)` |
| created_by 值 | `user.id \|\| undefined` | `user.id \|\| ''`（空字串炸 FK） |

## 5. 新表 6 層 SOP（完整版在 `2026-05-13-建表-SOP.md`）

**Phase 0**：資料屬性分類

| 類型 | 範例 | RLS pattern |
|------|------|-------------|
| 業務主表 | customers / orders / tours | `setup_workspace_scoped_rls` |
| 業務子表 | order_members / journal_lines | `setup_inherited_rls` |
| 員工關係 join | employee_branches | `setup_join_table_rls` |
| 全租戶共享 ref | ref_airlines / kb_cruise_lines | `SELECT TO authenticated USING (true)` |
| 系統表 | api_usage / audit_logs | admin only |
| 個人化 | user_preferences | `user_id = auth.uid()` |

**標準業務主表 migration 範本**：

```sql
CREATE TABLE public.<table> (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id uuid NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  code text,
  name text NOT NULL,
  status text DEFAULT 'active',
  created_by uuid REFERENCES public.employees(id) ON DELETE SET NULL,
  updated_by uuid REFERENCES public.employees(id) ON DELETE SET NULL,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  deleted_at timestamptz
);

-- Index（效能規格）
CREATE INDEX <table>_workspace_created_idx ON public.<table> (workspace_id, created_at DESC);
CREATE UNIQUE INDEX <table>_workspace_code_unique ON public.<table> (workspace_id, code) WHERE deleted_at IS NULL;

-- RLS（走 procedure，不散刻）
CALL setup_workspace_scoped_rls('<table>');

-- updated_at trigger
CREATE TRIGGER <table>_updated_at BEFORE UPDATE ON public.<table>
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

## 7. 技術紅線（違反會炸）

### 紅線 0：沒有超級管理員

- ❌ `if (isAdmin)` / `if (user.is_admin)` 跳過權限
- ❌ `requireCapability('platform.is_admin')` 萬能 capability
- ❌ `workspaces.type === 'agency'` 守門（`workspaces.type` 欄位在 code 層已砍）
- ✅ 全走 `workspace_features` + `role_capabilities` + RLS 三道

### 紅線 A：workspaces 絕不 FORCE RLS

- ENABLE 可以，FORCE 絕對不行
- 打斷登入（admin client 無法查 workspace）

### 紅線 B：審計欄位 FK → employees(id)

- `created_by / updated_by / deleted_by / locked_by` → `REFERENCES employees(id) ON DELETE SET NULL`
- 例外：`user_id / sender_id` 才指 `auth.users(id)`
- 前端：`created_by: user.id || undefined`，**不是 `|| ''`**

### 紅線 C：admin client 必 per-request

- `getSupabaseAdminClient()` 每次 call 重新建立
- 不 export 全局 admin variable（跨 request 污染）

### 紅線 D：封存不能解鎖（GAAP 原則）

- 月結後不能解鎖
- 結團後不能重開
- 已確認傳票只能開沖正傳票，不能改數字
- ❌ `forceUnlock` / `adminOverride` / `bypassPeriodLock` / `reopenClosedXxx`

### 其他禁止清單

| 禁止 | 正確 |
|------|------|
| `as any` 蓋 type error | 找根因修 |
| `: any` 標 type | 補正確 type |
| `--no-verify` 跳 hook | 查清楚 hook 為什麼擋 |
| `\|\| ''` 空字串 FK | `\|\| undefined` |
| 散刻 4 條 CREATE POLICY | 走 `setup_*_rls` procedure |
| 前端撈全表再 JS filter | server-side filter + PostgREST |
| inline `supabase.rpc('generate_*')` | 走 `@/lib/codes.ts` |
| `error.message` 直接 return | 走 `dbErrorResponse(error)` |
| hardcoded 中文 JSX | 抽到 `_constants/labels.ts` |
| 自己寫 Realtime channel | 走 `createEntityHook` 內建 `useRealtimeSync()` |
