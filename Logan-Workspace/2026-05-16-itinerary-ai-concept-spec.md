## 一、核心問題定義

**現有 AI 排行程的痛點**：
- ChatGPT / Gemini 從網路資料學習，不知道哪個景點已關閉、哪條路線不合理
- 產出的景點品質不可控（「很鳥的景點」也會推薦）
- 沒有時間軸概念，排出的行程在地理上常常繞路
- 輸出格式無法直接給客人使用

**Venturo 的差異化解法**：
> AI 只從 Venturo 自建資料庫推薦，不自己亂編。資料庫品質 = 行程品質。

這是護城河：別人有 Claude / Gemini，但沒有這份由業者累積的景點品質資料。

## 三、技術架構

### 3.1 距離矩陣預計算（Google Maps 一次性成本）

**問題**：每次排行程都打 Google Maps API → 費用高、延遲大  
**解法**：每個目的地建置「點對點時間矩陣」，一次計算、永久快取

```
目的地（如清邁）有 60 個景點
→ 60 × 60 = 3,600 個點對
→ Google Distance Matrix API（最多 25×25 = 625 對/次）
→ 分 6 次批次 call，一次性成本約 $18 USD
→ 存進 DB，之後排行程完全不打 Google API
→ 新增景點時只補算「新景點 × 所有既有景點」（N 次 call）
→ 季度更新一次（道路改變）
```

**DB 結構**：
```sql
CREATE TABLE travel_time_matrix (
  origin_id    uuid REFERENCES attractions(id),
  dest_id      uuid REFERENCES attractions(id),
  mode         text,  -- 'driving' | 'walking' | 'transit'
  minutes      int,
  distance_km  decimal(8,2),
  synced_at    timestamptz DEFAULT now(),
  PRIMARY KEY (origin_id, dest_id, mode)
);
```

**觸發時機**：
- 新目的地景點建置完成 → 管理員手動觸發「建立距離矩陣」
- 新增單一景點 → 自動補算（僅計算新景點與所有既有景點的來回）
- 季度 cron → 重算更新

### 3.2 AI 行程生成流程

```
業者輸入：
  目的地、天數、人數、主題（家庭/蜜月/冒險/文化）、預算

↓ Step 1：從 DB 撈符合條件的景點
  - 按主題過濾 category / tags
  - 按預算過濾 ticket_price
  - 排除 is_active = false 的景點

↓ Step 2：從 travel_time_matrix 撈點間時間（無 Google API call）

↓ Step 3：排程最佳化（初版用貪婪演算法）
  - 地理聚集（同一區域的景點排在同一天）
  - 時間分配（早上體力好 → 走路多的景點、下午 → 室內輕鬆）
  - 開放時間校驗（不排進休館日、不排超過閉館時間）
  - 用餐時間預留（午餐 12:00、晚餐 18:30 各預留 1h）

↓ Step 4：Claude API 加敘述文字
  - 輸入：結構化行程 JSON（景點序列 + 時間 + 交通）
  - 輸出：每段的說明文字（「搭車約 20 分鐘後抵達...」）
  - 每次約 500-800 tokens（Haiku 模型，成本極低）

↓ Step 5：輸出格式渲染
```

### 3.3 輸出格式

| 格式 | 說明 | 實作方式 |
|------|------|---------|
| 時間軸視圖（ERP 內） | 橫向/直向，每個點有照片、時間、交通 | React component |
| 可分享網頁 | 給客人一個連結，手機開即看行程 | Next.js 動態頁 `/itinerary/[shareCode]` |
| PDF | 帶公司 logo，可列印帶去旅遊 | `pdf-lib` / `react-pdf` |
| Word | 業者繼續編輯修改 | `docx` library |

## 四、排程演算法分析

### Phase 1：貪婪最近鄰（Greedy Nearest Neighbor）
- 實作簡單、TypeScript 直接寫
- 品質「夠用」：避免明顯繞路
- 不適合大型行程（8天以上會開始走樣）

### Phase 2：2-opt 改善
- 在貪婪結果上做迭代優化
- 品質大幅提升、計算量仍可接受

### Phase 3：Google OR-Tools（Python 微服務）
- 業界標準旅行商問題（TSP）求解器
- 支援開放時間約束、天數分組、用餐時間預留
- 需要獨立 Python 服務（Vultr 上起一個 container）
- 適合 10 天以上、50 個以上景點的複雜行程

**初版用 Phase 1，有實際業者使用後再升級。**

當景點數量超過 500 個，用向量搜尋取代純 SQL filter：

```
業者輸入：「適合帶小朋友的清邁行程，喜歡文化但不喜歡人多」

→ 用 pgvector 做語意搜尋（description + notes 向量化）
→ 找出語意相符的景點（不只是 category = '文化' 的精確匹配）
→ 輸入排程引擎

Supabase 支援 pgvector extension，已在路線圖上。
```

## 六、UI/UX 規劃

### 入口（ERP 主流程）
```
訂單 / 報價 → 「產生行程」按鈕
  → 行程規劃面板（目的地、天數、主題、人數）
  → AI 生成草稿（30-60 秒）
  → 可拖曳調整景點順序
  → 確認後存為行程、可匯出
```

### 輸出入口
```
行程確認頁 → 「分享給客人」→ 生成可分享連結
            → 「下載 PDF」→ 帶 logo 版行程 PDF
            → 「匯出 Word」→ 可繼續編輯的 .docx
```
