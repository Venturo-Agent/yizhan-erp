## 一、核心概念

一棧 ERP 的客戶（旅行社）加購「官網報名」功能後，可以在自己的官網或 landing page 嵌入報名表單。表單呼叫一棧 ERP 的公開 API，讓旅行社的客人直接線上報名旅遊團。

**B2B2C 關係：**
```
一棧 ERP（一棧）
    └── 旅行社 workspace（一棧的客戶）
            └── 旅遊客人（旅行社的客人）
```

**整體流程：**
```
旅行社官網（已嵌入報名 widget）
    ↓ 客人填表 + 送出
POST /api/public/tours/{tourId}/register
    ↓ 建立 pending_review 訂單
AI Hub 業務通知 + 待處理清單出現新單
    ↓ 業務確認
正式訂單 → 收款 / 帳單流程
```

## 二、Feature Gate（加購功能）

功能代碼：`website_booking`

| 功能 | 未加購 | 已加購 |
|------|-------|-------|
| 取得公開 API Token | ❌ | ✅ |
| 旅遊團公開上架 | ❌ | ✅ |
| 接收網路報名訂單 | ❌ | ✅ |

**Workspace 設定：**
```sql
-- workspace_features 加一筆
feature_code = 'website_booking'

-- workspace_website_settings（新表）
workspace_id        UUID PK
public_api_token    TEXT  -- publishable key，可嵌入前端，限定只能建 pending 訂單
webhook_secret      TEXT  -- 可選，讓旅行社官網驗證事件真實性
registration_fields JSONB -- 哪些欄位必填（name/phone/email/line_id/pax_count/notes）
auto_confirm        BOOL  -- false（預設）= 需業務手動確認；true = 立即 confirmed
created_at          TIMESTAMPTZ
updated_at          TIMESTAMPTZ
```

**Public API Token 安全性：**
- 類比 Stripe publishable key — 可以放在前端 HTML
- 只允許：列出 published 旅遊團、建立 pending_review 訂單
- 不允許：讀取訂單資料、管理旅遊團、存取財務

## 三、訂單狀態機

### 3.1 新增欄位

```sql
-- orders 表新增
source        TEXT NOT NULL DEFAULT 'manual'
  -- 'manual'    業務在 ERP 內直接建立
  -- 'website'   官網報名表單
  -- 'line_ai'   LINE Bot / AI 需求單轉過來的

status        TEXT NOT NULL DEFAULT 'confirmed'
  -- 調整：手動建立預設直接 confirmed
  -- 網路來源預設 pending_review
```

### 3.2 狀態流

```
                    ┌─────────────────────────────────┐
                    │                                 │
                    ▼                                 │
manual 建立 ──→ confirmed ──→ in_progress ──→ departed ──→ completed
                    ▲              │
                    │              └──→ cancelled
website/line_ai ──→ pending_review
                    │
                    ├──→ confirmed（業務確認）
                    └──→ rejected（業務拒絕）
```

### 3.3 狀態定義

| 狀態 | 觸發 | 說明 |
|------|------|------|
| `pending_review` | 網路/AI 報名進來 | 等業務確認，不計入正式訂單統計 |
| `confirmed` | 業務確認 or 手動建立 | 正式訂單，計入出團人數 |
| `rejected` | 業務拒絕 | 額滿、不符資格、無法成團 |
| `cancelled` | 客人取消 or 業務取消 | 已 confirmed 後取消，有退款流程 |
| `in_progress` | 出發日到 | 出發前準備期 |
| `departed` | 實際出發 | 人已出發 |
| `completed` | 回程確認 | 結束，進財務結清 |

## 四、公開 API 設計

### 4.1 端點

```
基底：https://erp.venturo.tw/api/public/v1

驗證方式：
  Header: X-Public-Token: {workspace_public_api_token}
  或 Query: ?token={workspace_public_api_token}
```

### 4.2 列出可報名的旅遊團

```
GET /api/public/v1/tours

Headers:
  X-Public-Token: {token}

Response:
{
  "tours": [
    {
      "id": "uuid",
      "title": "日本關西五日",
      "departure_date": "2026-07-01",
      "return_date": "2026-07-05",
      "price_per_person": 45000,
      "currency": "TWD",
      "available_seats": 12,
      "total_seats": 20,
      "destination": "日本",
      "thumbnail_url": "...",
      "registration_deadline": "2026-06-15"
    }
  ]
}
```

**安全：** 只回 published 且出發日未到的旅遊團。不回 draft / template / 內部成本資訊。

### 4.3 取得單一旅遊團詳情

```
GET /api/public/v1/tours/{tourId}

Response: 同上，單筆，加上行程說明欄位
```

### 4.4 送出報名（核心）

```
POST /api/public/v1/tours/{tourId}/register

Headers:
  X-Public-Token: {token}
  Content-Type: application/json

Body:
{
  "contact": {
    "name": "王小明",
    "phone": "0912345678",
    "email": "ming@example.com",
    "line_user_id": "Uxxxx"        // 選填，LINE 報名時有
  },
  "pax_count": 2,                  // 報名人數
  "travelers": [                   // 選填，每位旅客資料
    { "name": "王小明", "id_number": "...", "passport": "..." }
  ],
  "notes": "素食×1",
  "referral_source": "instagram"   // 選填，從哪得知這個旅遊團
}

Response 成功：
{
  "success": true,
  "registration_id": "order-uuid",
  "status": "pending_review",
  "message": "報名成功！業務將於 1 個工作天內與您確認。",
  "next_steps": "請留意 LINE 或 Email 通知"
}

Response 失敗：
{
  "success": false,
  "error": "seats_not_available",
  "message": "此團名額已滿"
}
```

**錯誤代碼：**
| code | 說明 |
|------|------|
| `tour_not_found` | 旅遊團不存在 / 未開放 |
| `seats_not_available` | 名額不足 |
| `registration_closed` | 截止日已過 |
| `invalid_token` | 公開 token 無效 |
| `feature_not_enabled` | workspace 未開通官網報名 |
| `rate_limit_exceeded` | 同 IP 太頻繁 |

### 4.5 安全機制

| 機制 | 設計 |
|------|------|
| Rate limit | 每 IP 每小時最多 10 次報名 |
| Token 驗證 | Public token 對應 workspace，只能操作該 workspace 的資料 |
| 座位鎖 | 送出時先鎖 pax_count 個座位（15 分鐘），確認後正式扣 |
| CORS | Workspace 可設定允許的 origin（防止非授權網站呼叫） |
| 重複偵測 | 同 phone + tourId 在 24 小時內不能重複送出 |

## 五、AI Hub — 待處理介面

### 5.1 新 Tab：待處理

```
AI Hub
├── 對話列表
├── 待處理 🔴 (3)      ← 新增，badge 顯示未處理數量
└── 設定
```

### 5.2 待處理清單

每筆顯示：
- 來源 icon（🌐 官網 / 💬 LINE AI）
- 旅遊團名稱
- 報名人數、聯絡資訊
- 報名時間（幾分鐘前）
- 快速動作：**確認** / **拒絕** / **聯絡客人**

### 5.3 業務操作流

**確認：**
1. 點「確認」
2. 可填「備註給客人」
3. 訂單 status → confirmed
4. 系統自動通知客人（LINE 訊息 / Email）
5. 從待處理消失，進入正常訂單管理流程

**拒絕：**
1. 點「拒絕」
2. 選拒絕原因（額滿 / 日期不符 / 其他）
3. 訂單 status → rejected
4. 系統通知客人被拒絕原因 + 推薦其他可報名的團

**聯絡客人：**
- 如果有 line_user_id → 直接開 LINE 對話
- 如果有 email → 寄信
- 如果只有 phone → 顯示電話讓業務自己打

### 5.4 通知機制

新報名進來時通知業務的管道：

| 管道 | 條件 |
|------|------|
| Channels（ERP 內部）| 一定有，發到「AI Hub」頻道 |
| LINE（業務的 LINE）| 選填，workspace 設定業務 LINE ID |
| Email | 選填，workspace 設定通知信箱 |

## 六、資料庫變更清單

```sql
-- 1. orders 表加欄位
ALTER TABLE orders
  ADD COLUMN source TEXT NOT NULL DEFAULT 'manual',
  ADD COLUMN status TEXT NOT NULL DEFAULT 'confirmed';

-- source CHECK
ALTER TABLE orders
  ADD CONSTRAINT orders_source_check
  CHECK (source IN ('manual', 'website', 'line_ai'));

-- status CHECK
ALTER TABLE orders
  ADD CONSTRAINT orders_status_check
  CHECK (status IN (
    'pending_review', 'confirmed', 'rejected',
    'cancelled', 'in_progress', 'departed', 'completed'
  ));

-- 2. workspace_website_settings 新表
CREATE TABLE workspace_website_settings (
  workspace_id        UUID PRIMARY KEY REFERENCES workspaces(id) ON DELETE CASCADE,
  public_api_token    TEXT NOT NULL UNIQUE,
  webhook_secret      TEXT,
  registration_fields JSONB NOT NULL DEFAULT '{"name":true,"phone":true,"email":false,"pax_count":true}',
  auto_confirm        BOOLEAN NOT NULL DEFAULT false,
  allowed_origins     TEXT[],    -- CORS whitelist
  created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 3. workspace_features seed（需要業務手動開通）
-- feature_code = 'website_booking'

-- 4. orders 現有資料 backfill
UPDATE orders SET source = 'manual', status = 'confirmed'
WHERE source IS NULL;
```

## 九、概念延伸：這是 SaaS 的護城河

這個功能的戰略意義不只是「讓客人可以線上報名」：

1. **旅行社黏著度** — 官網報名串進 ERP，資料不用手抄，旅行社不想換掉系統
2. **AI 客服的知識庫** — 所有 published 旅遊團的資料都是 RAG 來源，AI 能正確回答客人問「還有沒有位置」
3. **CRM 自動建檔** — 每筆報名自動建客人資料，不用業務手動輸入
4. **未來收費點** — `website_booking` 是獨立 feature code，可以單獨加購，也可以綁月費方案
5. **數據** — 旅行社能看到每個旅遊團的轉換率（看了多少人、報了多少、成了多少），這是傳統旅行社沒有的數據
